1. Debugging mindset

Always work from the outermost visible layer inward. Never start by reading application code.

Rule: Every failure in this system has an observable symptom at one of five layers. Start at layer 1. Do not skip to layer 5 without confirming layers 1–4 are clean.

The five layers, in order:

  • 1 Is the pod running? kubectl get pods -n <namespace> Expected: STATUS = Running. If not Running: the pod never started, is crashing, or failed to pull its image.
  • 2 Is it healthy? kubectl describe pod <pod-name> -n <namespace> Check the Events section at the bottom. Events name the exact failure: image pull error, OOM kill, Kyverno policy block, liveness probe failure.
  • 3 Are the logs clean? kubectl logs <pod-name> -n <namespace> For a crashed pod: add --previous to see logs from the last failed instance. Log output names the exact line and error that caused the exit.
  • 4 Is the config correct? kubectl describe configmap <name> -n <namespace> kubectl describe secret <name> -n <namespace> Confirm the expected keys are present. For secrets: check keys exist, not values (values are base64-encoded). A missing key causes a missing env var at runtime.
  • 5 Is the image correct? docker inspect <image>:<tag> --format '{{json .Config.Labels}}' | jq Labels record COMPLIANCE, RUNTIME, INDUSTRY, PKG_MGR build args. A wrong label means a wrong image was promoted or the build ran with wrong args.
Warning: Do not read application source code until all five layers are confirmed clean. In 90% of cases the failure is a missing secret, wrong image tag, or missing pull secret — none of which are visible in application code.

The debugging decision tree

Use this table to navigate to the right section immediately.

What you observe Where to go Most likely cause
Pod STATUS = CrashLoopBackOff → Section 2 (startup) or Section 5 (k8s) Application exits non-zero on start. Check logs --previous.
Pod STATUS = ImagePullBackOff → Section 5.1 Image missing from GHCR, or pull secret absent from namespace.
Pod STATUS = Pending with no events → Section 5.3 Kyverno policy blocking pod scheduling.
Pod STATUS = Running but /health/ready = 503 → Section 2.2 Dependency (database, external service) not reachable.
docker build exits non-zero → Section 3 Wrong build-arg combination, missing compliance variant folder, missing lockfile.
GitHub Actions step fails → Section 4 Missing secret (GH_PAT, cosign), bad image reference, missing lockfile.
ArgoCD shows OutOfSync → Section 6.1 Manual kubectl apply diverged from Git state, or sync interval not elapsed.
ArgoCD shows Degraded → Section 6.2 Pods under the ArgoCD app are unhealthy. Follow pod failure paths.
HPA shows unable to get metrics → Section 5.4 metrics-server missing from cluster, or OBSERVABILITY=none removes /metrics.

2. Service startup failures

All four failures below cause the pod to exit immediately or fail its readiness probe within 10 seconds.

How to confirm it is a startup failure: kubectl get pods -n <ns> shows STATUS = CrashLoopBackOff or RESTARTS ≥ 2.
Run kubectl logs <pod> -n <ns> --previous to see the crash output.
STARTUP 2.1 — JWT_SECRET missing or too short
Symptom
Pod crashes immediately on startup. RESTARTS increments with no delay between attempts.
Log messages
JWT_SECRET required
# — or —
JWT_SECRET must be at least 32 chars
Cause A
The Secret object does not contain a JWT_SECRET key. The env var is absent at runtime. The application startup check sees an empty string and exits.
Cause B
The Secret object contains JWT_SECRET but the decoded value is shorter than 32 characters. The application startup check validates length and exits.
Diagnose
# Step 1 — confirm the secret exists and has the key
kubectl get secret <name> -n <ns> -o yaml

# Step 2 — decode and check length (≥32 chars required)
kubectl get secret <name> -n <ns> -o jsonpath='{.data.JWT_SECRET}' | base64 -d | wc -c

# Step 3 — confirm the pod's env var references the secret
kubectl describe pod <pod> -n <ns> | grep -A3 JWT_SECRET
Fix A
Secret key is missing. Create or patch the secret: 
# Generate a 48-character random secret
JWT_VAL=$(openssl rand -base64 36)

# Patch the existing secret
kubectl patch secret <name> -n <ns> \
  --type='json' \
  -p="[{\"op\":\"add\",\"path\":\"/data/JWT_SECRET\",\"value\":\"$(echo -n $JWT_VAL | base64 -w0)\"}]"
Fix B
Value is too short. Rotate to a value ≥ 32 characters: 
JWT_VAL=$(openssl rand -base64 36)
kubectl edit secret <name> -n <ns>
# Replace the JWT_SECRET value with: $(echo -n $JWT_VAL | base64 -w0)
Verify
# Restart the pod to pick up the new secret value
kubectl rollout restart deployment/<name> -n <ns>

# Wait for rollout
kubectl rollout status deployment/<name> -n <ns>

# Confirm pod is Running with 0 restarts
kubectl get pods -n <ns> | grep <name>
STARTUP 2.2 — Database connection refused
Symptom
Service starts but /health/ready returns HTTP 503. Pod remains in Running state but readiness probe fails continuously.
Log messages
Connection refused
# — or —
ECONNREFUSED 127.0.0.1:5432
# — or —
could not connect to server: Connection refused
        Is the server running on host "postgresql" and accepting
        TCP/IP connections on port 5432?
Cause A
DATABASE_URL in the secret points to the wrong hostname or port. The hostname does not resolve inside the cluster.
Cause B
The PostgreSQL pod is not running or not ready. The service DNS resolves but the pod behind it is down.
Diagnose
# Step 1 — check if the postgres pod is running
kubectl get pods -n <ns> | grep postgres

# Step 2 — check the DATABASE_URL value in the secret
kubectl get secret <name> -n <ns> -o jsonpath='{.data.DATABASE_URL}' | base64 -d

# Step 3 — test connectivity from inside the cluster
kubectl run pg-test --image=postgres:15-alpine --restart=Never \
  --env="PGPASSWORD=<pass>" -n <ns> --rm -it -- \
  psql -h postgresql -U <user> -d <db> -c "SELECT 1"

# Step 4 — port-forward and test from local machine
kubectl port-forward svc/postgresql 5432:5432 -n <ns> &
psql postgresql://<user>:<pass>@localhost:5432/<db> -c "SELECT 1"
Fix A
Wrong hostname in DATABASE_URL. Correct it in the secret. The in-cluster DNS name for a Service named postgresql in namespace myapp is: 
postgresql.myapp.svc.cluster.local
Short form (same namespace): postgresql. Update the secret: 
kubectl edit secret <name> -n <ns>
# Update DATABASE_URL to: postgresql://user:pass@postgresql:5432/db
# Re-encode: echo -n "postgresql://user:pass@postgresql:5432/db" | base64 -w0
Fix B
PostgreSQL pod is down. Check why and restore it: 
# Check postgres pod events
kubectl describe pod <postgres-pod> -n <ns>

# Check postgres pod logs
kubectl logs <postgres-pod> -n <ns> --tail=50

# If PVC is missing or full
kubectl get pvc -n <ns>
kubectl describe pvc <pvc-name> -n <ns>
Verify
# Confirm postgres pod is Running
kubectl get pods -n <ns> | grep postgres

# Confirm the service pod's readiness probe passes
kubectl get pods -n <ns> | grep <service-name>
# Expected: READY = 1/1, STATUS = Running

# Hit the readiness endpoint
kubectl port-forward svc/<service> 8080:8080 -n <ns> &
curl -s http://localhost:8080/health/ready
# Expected: HTTP 200
STARTUP 2.3 — FIPS compliance check fails
Symptom
Pod crashes on the first log line. No application logic runs — the FIPS check runs before any handler is registered.
Log messages
FIPS: OpenSSL FIPS module not active
# — or —
FIPS: non-FIPS runtime detected. Refusing to start.
# — or —
FIPS compliance check failed: OPENSSL_FIPS=0
Cause
The image was built with COMPLIANCE=fips but RUNTIME=alpine or RUNTIME=slim. Alpine and slim base images do not ship the FIPS-validated OpenSSL module. The startup check detects OpenSSL FIPS mode is inactive and refuses to start.
Diagnose
# Inspect the image labels to see which args were used
docker inspect <image>:<tag> --format '{{json .Config.Labels}}' | jq

# Expected for a correct FIPS image:
# {
#   "compliance": "fips",
#   "runtime": "fips",
#   ...
# }

# If runtime label is "alpine" or "slim": the image was built wrong
Fix
Rebuild the image with the correct combination. FIPS compliance requires both args set to fips: 
docker build \
  --build-arg COMPLIANCE=fips \
  --build-arg RUNTIME=fips \
  -t <service>:fips \
  services/<service>/
Verify
# Confirm runtime label on the rebuilt image
docker inspect <service>:fips \
  --format '{{index .Config.Labels "runtime"}}'
# Expected: fips

# Run the container locally and confirm it starts
docker run --rm -e JWT_SECRET=$(openssl rand -base64 36) \
  <service>:fips
# Should print startup logs without FIPS error
Related
See 12 Compliance variations for the full matrix of COMPLIANCE × RUNTIME allowed combinations. FIPS is the only compliance level that restricts the runtime choice.
STARTUP 2.4 — PCI compliance weak secret rejection
Symptom
Pod crashes at startup. The application reached the PCI middleware init phase before exiting. RESTARTS increments but the pod never reaches the HTTP handler phase.
Log messages
PCI: JWT_SECRET weak/plaintext secret (<16 chars)
# — or —
PCI: weak secret detected in env var DB_PASSWORD (length: 8, required: 16)
# — or —
PCI: refusing to start — 2 weak secrets detected
Cause
PCI compliance middleware runs a startup scan. Every environment variable whose name contains SECRET, KEY, TOKEN, or PASS is checked. Any such variable with a value shorter than 16 characters causes an immediate exit. PCI DSS Requirement 8.3.6 mandates minimum password length of 12; this middleware enforces 16.
Diagnose
# List all env vars injected into the pod that contain sensitive name patterns
kubectl get secret <name> -n <ns> -o json \
  | jq '.data | to_entries[] | select(.key | test("SECRET|KEY|TOKEN|PASS"; "i")) | {key: .key, len: (.value | @base64d | length)}'

# Any key with len < 16 must be rotated
Fix
Rotate every weak secret to a value ≥ 16 characters. 
# Generate compliant values
NEW_SECRET=$(openssl rand -base64 24)       # 32 chars
NEW_DB_PASS=$(openssl rand -base64 16)      # 22 chars

# Apply to the Kubernetes secret
kubectl edit secret <name> -n <ns>
# Replace each short value with the base64-encoded long value:
# echo -n "$NEW_SECRET" | base64 -w0
Verify
# Restart the deployment
kubectl rollout restart deployment/<name> -n <ns>
kubectl rollout status deployment/<name> -n <ns>

# Confirm pod starts without PCI error
kubectl logs -l app=<name> -n <ns> --tail=20
# Should NOT contain "PCI:" prefix lines
Related
PCI middleware behavior is defined in 10 Linux compliance. The startup check also runs when the pod restarts — rotating a secret requires a rollout restart to take effect.

3. Build failures

All four failures below occur during docker build before any image is produced.

How to get the full build error: Run docker build --progress=plain ... to disable the progress display and show every build step output in full.
BUILD 3.1 — COPY src/compliance/${COMPLIANCE}/ not found
Symptom
docker build exits non-zero. The failure is in an early COPY layer, before any npm/go/python step runs.
Error output
COPY failed: file not found in build context or excluded by .dockerignore:
  directory 'src/compliance/hipaa' does not exist
Cause
The --build-arg COMPLIANCE=<value> was passed but that variant folder does not exist. The Dockerfile has COPY src/compliance/${COMPLIANCE}/ — BuildKit substitutes the ARG value literally. If src/compliance/hipaa/ is absent but COMPLIANCE=hipaa was passed, the COPY fails.
Diagnose
# List available compliance variant folders for the service
ls services/<service-name>/src/compliance/

# Example output:
# none/  standard/  pci/  fips/
# (hipaa does not exist for this service)
Fix
Use one of the available variant names. If the desired compliance variant is genuinely missing: it must be created first. 
# Use an available variant
docker build --build-arg COMPLIANCE=standard services/<name>/

# To add a new variant (e.g. hipaa), create the folder and required files:
mkdir -p services/<name>/src/compliance/hipaa
# Then copy and adapt from: services/<name>/src/compliance/standard/
Verify
# Build must complete without error
docker build --build-arg COMPLIANCE=standard \
  --progress=plain \
  services/<name>/ \
  -t <name>:debug

# Confirm image exists
docker images | grep <name>
BUILD 3.2 — COMPLIANCE=fips without RUNTIME=fips
Symptom
docker build exits non-zero in the very first build stage. The error is explicit and does not come from a COPY or RUN command failure.
Error output
ERROR: COMPLIANCE=fips requires RUNTIME=fips.
       Alpine and slim runtimes do not include the FIPS-validated OpenSSL module.
       Re-run with: --build-arg RUNTIME=fips
Cause
The Dockerfile contains a build guard in the first stage: 
ARG COMPLIANCE
ARG RUNTIME
RUN if [ "$COMPLIANCE" = "fips" ] && [ "$RUNTIME" != "fips" ]; then \
      echo "ERROR: COMPLIANCE=fips requires RUNTIME=fips." && exit 1; \
    fi
This guard exists to prevent a runtime FIPS startup failure that would be harder to diagnose. It forces the issue to build time instead.
Fix
Add --build-arg RUNTIME=fips to the docker build command: 
docker build \
  --build-arg COMPLIANCE=fips \
  --build-arg RUNTIME=fips \
  services/<name>/
Verify
docker build \
  --build-arg COMPLIANCE=fips \
  --build-arg RUNTIME=fips \
  --progress=plain \
  services/<name>/ \
  -t <name>:fips

# Confirm labels are correct
docker inspect <name>:fips \
  --format '{{index .Config.Labels "compliance"}} {{index .Config.Labels "runtime"}}'
# Expected: fips fips
BUILD 3.3 — npm ci fails — missing lockfile
Symptom
docker build fails in the dependency installation layer. Earlier layers (FROM, COPY compliance, ARG) succeed.
Error output
npm error code EUSAGE
npm error
npm error `npm ci` can only install packages when your package.json
npm error and package-lock.json or npm-shrinkwrap.json are in sync.
npm error
npm error Missing: pnpm-lock.yaml

# — or for pnpm variant —
ERROR Run failed with exit code 1
 ERR_PNPM_NO_LOCKFILE  Cannot install with "frozen-lockfile" because pnpm-lock.yaml is absent
Cause
The service was built with PKG_MGR=pnpm but pnpm-lock.yaml does not exist in the service directory. The Dockerfile calls pnpm install --frozen-lockfile which requires a lockfile to already exist. This happens when a new developer runs npm install locally instead of pnpm install, producing a package-lock.json but not the required pnpm-lock.yaml.
Diagnose
# Check which lockfiles exist
ls services/<name>/

# Expected for PKG_MGR=pnpm:
# pnpm-lock.yaml  ← must exist
# package.json    ← must exist

# If package-lock.json exists but pnpm-lock.yaml does not:
# the service was installed with npm, not pnpm
Fix
Generate the missing lockfile locally using the correct package manager: 
# Install pnpm if not present
npm install -g pnpm

# Generate pnpm-lock.yaml
cd services/<name>
pnpm install

# Remove the wrong lockfile if it exists
rm -f package-lock.json

# Commit the lockfile
git add pnpm-lock.yaml
git commit -m "Add pnpm-lock.yaml for <service-name>"
Verify
# Rebuild — the pnpm install step must succeed
docker build \
  --build-arg PKG_MGR=pnpm \
  --progress=plain \
  services/<name>/ \
  -t <name>:lockfile-test
BUILD 3.4 — go build fails — package internal/compliance/active not found
Symptom
docker build fails in the go build step. The COPY step succeeds but the compiler cannot find the compliance package.
Error output
cannot find package "internal/compliance/active" in any of:
        /usr/local/go/src/internal/compliance/active (from $GOROOT)
        /app/internal/compliance/active (from $GOPATH)
Cause
The Dockerfile has a COPY step that places the compliance variant files at internal/compliance/active/. If the go build step runs before this COPY step, the Go module sees an empty or absent directory. This happens when the Dockerfile stages are in the wrong order — specifically when the compliance COPY is placed after the RUN go build command.
Diagnose
# Print the Dockerfile with line numbers to check step order
cat -n services/<name>/Dockerfile | grep -E "COPY|go build|RUN go"

# Correct order must be:
#   N   COPY src/compliance/${COMPLIANCE}/ ./internal/compliance/active/
#   M   RUN go build ...
# where N < M
Fix
Move the compliance COPY step to run before the go build command. The correct Dockerfile section order is: 
# Stage: builder
FROM golang:1.22-alpine AS builder

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download

# COMPLIANCE COPY must come before go build
ARG COMPLIANCE=standard
COPY src/compliance/${COMPLIANCE}/ ./internal/compliance/active/

# COPY remaining source after compliance variant is in place
COPY . .

# Now go build can see internal/compliance/active
RUN go build -o /app/server ./cmd/server/
Verify
docker build \
  --build-arg COMPLIANCE=standard \
  --progress=plain \
  services/<name>/ \
  -t <name>:go-test

# Build must reach the final stage without go package error

4. CI/CD failures

Three failures that occur in GitHub Actions workflows after a push or PR merge.

How to get the full CI log: GitHub UI → Actions tab → click the failed run → click the failed job → expand the failed step. Alternatively: gh run view <run-id> --log-failed
CI/CD 4.1 — cosign sign fails — invalid go template
Symptom
GitHub Actions workflow step named "Sign image" fails. All prior steps (build, push) succeed. The workflow fails after the image is already in GHCR.
Error output
Error: invalid go template: the digest is empty
# — or —
Error: cosign sign: parse "": empty url
# — or —
error signing ghcr.io/yarova-ca/pipeline-studio/<service>@:
  invalid digest format
Cause
The cosign sign step references ${{ steps.build.outputs.digest }}. This output is empty when the build step does not produce it. The docker/build-push-action only emits a digest output when push: true is set and the image is pushed to the registry. If the build step used push: false or load: true, no digest is produced and the cosign step receives an empty string.
Diagnose
# In the GitHub Actions run log, expand the "Build and push" step.
# Look for the line: "digest: sha256:..."
# If this line is absent, the build step did not push.

# Check the workflow file
cat .github/workflows/<workflow>.yml | grep -A10 "build-push-action"
# Required: "push: true" and "outputs: type=image,push=true"
Fix
Ensure the build step is configured to push and emit the digest: 
- name: Build and push
  id: build
  uses: docker/build-push-action@v5
  with:
    context: services/<name>
    push: true                          # ← must be true
    tags: ghcr.io/yarova-ca/pipeline-studio/<name>:${{ github.sha }}
    outputs: type=image,push=true       # ← must emit digest
    build-args: |
      COMPLIANCE=${{ inputs.compliance }}
      RUNTIME=${{ inputs.runtime }}
The cosign step then references: 
- name: Sign image
  run: |
    cosign sign --yes \
      ghcr.io/yarova-ca/pipeline-studio/<name>@${{ steps.build.outputs.digest }}
Verify
# Re-run the workflow from the GitHub UI or via CLI
gh workflow run <workflow> -R yarova-ca/pipeline-studio

# After it completes, verify the signature
cosign verify \
  --certificate-identity-regexp=".*" \
  --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
  ghcr.io/yarova-ca/pipeline-studio/<name>:<sha>
CI/CD 4.2 — generate-lockfiles cannot push to repository
Symptom
The "generate-lockfiles" workflow step succeeds in generating the lockfile but fails when trying to push the commit back to the repository.
Error output
remote: Permission to yarova-ca/pipeline-studio.git denied to github-actions[bot].
fatal: unable to access 'https://github.com/yarova-ca/pipeline-studio.git/': The requested URL returned error: 403
Cause A
The GH_PAT repository secret is missing. The workflow uses a PAT (Personal Access Token) to push commits, because the default GITHUB_TOKEN cannot trigger subsequent workflow runs.
Cause B
The GH_PAT secret exists but has expired or was revoked. GitHub PATs have a maximum expiry of 1 year (fine-grained) or never expire (classic — not recommended).
Cause C
The PAT was rotated but the secret was not updated in the repository settings. The workflow still has the old, now-invalid token value.
Diagnose
# Check if the secret exists in the repo
gh secret list -R yarova-ca/pipeline-studio | grep GH_PAT

# If GH_PAT is absent: it must be created
# If GH_PAT is present: the token may be expired — check at github.com/settings/tokens
Fix
Step 1: Generate a new PAT at github.com/settings/tokens. Required permissions for a fine-grained token: 
Repository: yarova-ca/pipeline-studio
Permissions required:
  Contents: Read and write     ← to push commits
  Pull requests: Read and write ← to open PRs if needed
  Workflows: Read and write    ← to trigger workflows
Step 2: Set the new PAT as a repository secret: 
gh secret set GH_PAT \
  -R yarova-ca/pipeline-studio \
  --body "<new-pat-value>"
Verify
# Re-run the failed workflow
gh run rerun <run-id> -R yarova-ca/pipeline-studio

# Watch the run
gh run watch -R yarova-ca/pipeline-studio
CI/CD 4.3 — Dependabot cannot open pull requests
Symptom
No Dependabot pull requests appear despite new dependency versions being available. The Dependabot alerts tab shows outdated dependencies but no PRs are opened.
Error output
# In repository Insights > Dependency graph > Dependabot:
Dependabot cannot open pull requests for this repository.
Reason: Dependabot does not have permission to create pull requests.

# — or in Dependabot alerts —
Unable to update: pull requests are disabled for this repository
Cause A
Dependabot alerts is enabled but Dependabot security updates or version updates is not enabled. Alerts detect vulnerabilities but do not automatically open PRs. Auto-PRs require the Dependabot updates feature to be separately enabled.
Cause B
.github/dependabot.yml has a syntax error or references a package ecosystem that does not exist in the repo (e.g. specifying package-ecosystem: npm for a service that uses pnpm without a package-lock.json).
Cause C
Branch protection rules require a specific number of reviews before merge, and the Dependabot bot account does not satisfy the review requirement. Dependabot can still open PRs but they cannot be auto-merged.
Diagnose
# Check if dependabot.yml is valid
cat .github/dependabot.yml

# Check Dependabot settings in GitHub UI:
# Settings > Code security > Dependabot security updates = enabled?
# Settings > Code security > Dependabot version updates = enabled?

# Validate the dependabot.yml syntax:
# github.com/yarova-ca/pipeline-studio/settings/security_analysis
Fix A
Enable Dependabot version updates in GitHub Settings: 
Repository Settings
  > Code security and analysis
  > Dependabot alerts:          Enable
  > Dependabot security updates: Enable
  > Dependabot version updates: Enable
Fix B
Correct the dependabot.yml syntax. Example of a valid configuration for a pnpm monorepo: 
version: 2
updates:
  - package-ecosystem: "npm"       # pnpm uses npm ecosystem
    directory: "/services/api-gateway"
    schedule:
      interval: "weekly"
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
  - package-ecosystem: "docker"
    directory: "/services/api-gateway"
    schedule:
      interval: "weekly"
Verify
# After fixing, check Dependabot insights within 24 hours
# github.com/yarova-ca/pipeline-studio/security/dependabot

# Force a Dependabot check by touching dependabot.yml:
git commit --allow-empty -m "Trigger Dependabot re-check"
git push

5. Kubernetes failures

Four failures that occur after the image is built and pushed but the pod does not reach Running state.

Start here for any pod not in Running state: kubectl get pods -A | grep -v Running | grep -v Completed
This shows every pod across every namespace that is not healthy.
KUBERNETES 5.1 — Pod in ImagePullBackOff
Symptom
kubectl get pods -n <ns> shows STATUS = ImagePullBackOff or ErrImagePull. The pod transitions: ErrImagePull (first attempt) → ImagePullBackOff (exponential backoff active).
Events output
kubectl describe pod <pod> -n <ns>
# Events section:
Warning  Failed     2m    kubelet  Failed to pull image
  "ghcr.io/yarova-ca/pipeline-studio/api-gateway:abc1234":
  rpc error: code = Unknown
  desc = failed to pull and unpack image:
  failed to resolve reference "ghcr.io/yarova-ca/pipeline-studio/api-gateway:abc1234":
  unexpected status code 401 Unauthorized
Cause A
The image does not exist at that tag in GHCR. The CI pipeline may have failed before pushing, or the tag name is wrong.
Cause B
The GHCR pull secret (ghcr-pull-secret) is not present in the namespace. Kubernetes cannot authenticate to GHCR to pull the image.
Diagnose A
# Verify the image exists at the exact tag
docker manifest inspect \
  ghcr.io/yarova-ca/pipeline-studio/<service>:<sha>

# If this returns "manifest unknown": the image does not exist.
# Check the CI run for the commit:
gh run list -R yarova-ca/pipeline-studio --branch=main | head -5
Diagnose B
# Check if the pull secret exists in the namespace
kubectl get secret ghcr-pull-secret -n <ns>

# Check if the default service account references it
kubectl get serviceaccount default -n <ns> -o json \
  | jq '.imagePullSecrets'

# Expected output: [{"name": "ghcr-pull-secret"}]
# If null or empty: the pull secret is not attached
Fix A
Re-trigger the CI pipeline to push the correct image: 
gh workflow run build-and-push.yml \
  -R yarova-ca/pipeline-studio \
  -f service=<name> \
  -f sha=$(git rev-parse HEAD)
Fix B
Create the pull secret and attach it to the service account: 
# Step 1 — create the pull secret (requires a GHCR PAT or token)
kubectl create secret docker-registry ghcr-pull-secret \
  --docker-server=ghcr.io \
  --docker-username=<github-username> \
  --docker-password=<ghcr-pat> \
  -n <ns>

# Step 2 — patch the default service account to use the secret
kubectl patch serviceaccount default -n <ns> \
  -p '{"imagePullSecrets": [{"name": "ghcr-pull-secret"}]}'

# Step 3 — delete the failing pod to trigger a re-pull
kubectl delete pod <pod> -n <ns>
Verify
# Watch the new pod come up
kubectl get pods -n <ns> -w

# Expected within 60s: STATUS transitions from ContainerCreating to Running
KUBERNETES 5.2 — Pod in CrashLoopBackOff
Symptom
kubectl get pods -n <ns> shows STATUS = CrashLoopBackOff. RESTARTS column shows 5 or more. The pod image pulled successfully — the container process is exiting non-zero.
Cause
The container process (the application) exits with a non-zero exit code after starting. This is different from ImagePullBackOff — the image is present and running, but the app crashes. The most common causes in pipeline-studio are covered in Section 2 (startup failures).
Diagnose
# Step 1 — read logs from the PREVIOUS (crashed) instance
kubectl logs <pod> -n <ns> --previous

# Step 2 — if the pod restarts too fast to catch logs, increase restart delay
kubectl get pod <pod> -n <ns> -o jsonpath='{.status.containerStatuses[0].lastState.terminated}'

# Step 3 — check the exit code
# Exit code 1: application error (check logs)
# Exit code 137: OOM kill (check resource limits)
# Exit code 143: SIGTERM (check liveness probe timeout)

# Step 4 — check resource usage
kubectl top pod <pod> -n <ns>
Fix — application error (exit code 1)
Read the crash log output and follow the matching Section 2 failure path. The most common log signatures: 
"JWT_SECRET required"        → Section 2.1
"ECONNREFUSED"               → Section 2.2
"FIPS: non-FIPS runtime"     → Section 2.3
"PCI: weak secret"           → Section 2.4
Fix — OOM kill (exit code 137)
The container exceeded its memory limit and was killed by the kernel. Increase the memory limit in the Helm chart: 
kubectl edit deployment <name> -n <ns>
# Find: resources.limits.memory
# Change from "128Mi" to "256Mi" (or larger)

# Alternatively, patch directly:
kubectl patch deployment <name> -n <ns> \
  --type='json' \
  -p='[{"op":"replace","path":"/spec/template/spec/containers/0/resources/limits/memory","value":"256Mi"}]'
Verify
# Watch until the pod reaches Running with 0 new restarts
kubectl get pods -n <ns> -w | grep <name>

# Confirm no --previous logs exist (new pod is clean)
kubectl logs <new-pod> -n <ns> --tail=20
KUBERNETES 5.3 — Kyverno blocks pod scheduling
Symptom
Pod is in STATUS = Pending with 0 restarts. kubectl describe pod Events section shows a Kyverno admission webhook error. The pod has never started — the API server blocked the Pod creation request.
Events output
kubectl describe pod <pod> -n <ns>
# Events section:
Warning  FailedCreate  10s  replicaset-controller  Error creating: admission webhook
  "kyverno-resource-validating-webhook-cfg" denied the request:
  resource denied by policy require-sbom-label/check-sbom-label:
  validation rule check-sbom-label failed at path /metadata/labels/sbom/:
  Label 'sbom' is required on all Pods
Cause
A Kyverno ClusterPolicy is enforced in the cluster. The admission webhook runs before the pod is created. If the pod spec does not satisfy the policy, the creation is blocked. Common blocking policies in pipeline-studio: 
require-sbom-label      — Pod must have label: sbom=true or sbom=attested
restrict-latest-tag     — image tag must not be "latest"
require-readonly-root   — container must set readOnlyRootFilesystem: true
require-resource-limits — container must set resources.limits.memory and .cpu
Diagnose
# Step 1 — get the exact policy and rule name from the event
kubectl get events -n <ns> --sort-by=.lastTimestamp | grep kyverno | tail -5

# Step 2 — read the full policy to see what it requires
kubectl get clusterpolicy require-sbom-label -o yaml

# Step 3 — check what labels the failing pod has
kubectl describe pod <pod> -n <ns> | grep -A10 "Labels:"

# Step 4 — list all active ClusterPolicies
kubectl get clusterpolicy -A
Fix
Add the required label(s) to the Helm chart deployment template. The label must be on the Pod template spec, not the Deployment metadata: 
# In: helm/<service>/templates/deployment.yaml
# Under: spec.template.metadata.labels

spec:
  template:
    metadata:
      labels:
        app: <service-name>
        sbom: "attested"          # ← add this if require-sbom-label policy is active
        # image tag is set in values.yaml — never use "latest"
After adding the label, re-apply: 
helm upgrade <release> helm/<service> -n <ns>
Verify
# Check PolicyReport for remaining violations in the namespace
kubectl get policyreport -n <ns> -o json | jq '.items[].results[] | select(.result == "fail")'

# Confirm pod is now running
kubectl get pods -n <ns> | grep <service-name>
KUBERNETES 5.4 — HPA cannot scale — metrics not available
Symptom
The HorizontalPodAutoscaler exists but does not scale the deployment. kubectl get hpa -n <ns> shows TARGETS = <unknown>/50%.
Error output
kubectl describe hpa <name> -n <ns>
# Events section:
Warning  FailedGetScale  10s  horizontal-pod-autoscaler
  unable to get metrics for resource cpu: unable to fetch metrics from resource metrics API:
  the server could not find the requested resource
  (get pods.metrics.k8s.io)
Cause A
metrics-server is not installed in the cluster. The Kubernetes HPA requires metrics-server to collect CPU and memory metrics from pods. Without it, all HPA targets show <unknown>.
Cause B
The service was built with OBSERVABILITY=none. This build arg removes the Prometheus metrics middleware. The service does not expose a /metrics endpoint. Custom metrics HPA (Prometheus Adapter) cannot scrape it.
Diagnose
# Check if metrics-server is installed
kubectl get pods -n kube-system | grep metrics-server

# Check if metrics-server is returning data
kubectl top pods -n <ns>
# If this errors with "Metrics API not available": metrics-server is missing

# Check if /metrics endpoint exists on the service
kubectl port-forward svc/<service> 8080:8080 -n <ns> &
curl -s http://localhost:8080/metrics | head -5
# If 404 or empty: OBSERVABILITY=none was used
Fix A
Install metrics-server into the cluster: 
helm repo add metrics-server https://kubernetes-sigs.github.io/metrics-server/
helm repo update

helm install metrics-server metrics-server/metrics-server \
  -n kube-system \
  --set args={"--kubelet-insecure-tls"}  # required for local clusters

# Verify metrics-server is running
kubectl get pods -n kube-system | grep metrics-server

# Wait ~60s for metrics to populate, then check HPA
kubectl get hpa -n <ns>
Fix B
Rebuild the service image with observability enabled: 
docker build \
  --build-arg OBSERVABILITY=prometheus \
  services/<name>/

# Or set OBSERVABILITY=full for OpenTelemetry + Prometheus
Then re-deploy so the new image is running before the HPA checks for metrics.
Verify
# Confirm HPA can now read metrics
kubectl get hpa -n <ns>
# Expected: TARGETS shows actual values, e.g. "22%/50%"

# Confirm top pods works
kubectl top pods -n <ns>

6. ArgoCD failures

Two failures visible in the ArgoCD UI or via kubectl get applications -n argocd.

ArgoCD application health overview: kubectl get applications -n argocd -o wide
Columns to check: SYNC STATUS (Synced / OutOfSync) and HEALTH STATUS (Healthy / Degraded / Progressing / Missing).
ARGOCD 6.1 — Application stuck in OutOfSync
Symptom
ArgoCD UI shows SYNC STATUS = OutOfSync for an application. kubectl get application <name> -n argocd shows STATUS = OutOfSync. The app may still be running correctly — OutOfSync means the cluster state differs from Git state.
Cause A
Someone ran kubectl apply or kubectl edit directly on a cluster resource that ArgoCD manages. The cluster state now differs from the Helm chart in Git. ArgoCD detects this as drift and marks the app OutOfSync.
Cause B
A new commit changed values.yaml or a Helm template in Git. ArgoCD polls Git every 3 minutes by default. If the sync interval has not elapsed, ArgoCD has not yet detected the new commit. The app will show OutOfSync until the next poll cycle.
Diagnose
# Check what is out of sync
argocd app diff <app-name>

# — or via kubectl —
kubectl get application <name> -n argocd -o json \
  | jq '.status.sync.comparedTo, .status.sync.status'

# Check when ArgoCD last synced
kubectl get application <name> -n argocd -o json \
  | jq '.status.reconciledAt'
Fix A — manual drift
Force ArgoCD to reconcile the cluster state back to Git: 
argocd app sync <app-name> --force

# This will overwrite any manual kubectl changes
# with what is in the Helm chart in Git

# Alternatively from kubectl (no argocd CLI required):
kubectl annotate application <name> -n argocd \
  argocd.argoproj.io/refresh=hard
Fix B — polling delay
Wait 3 minutes for the next poll cycle. Or trigger an immediate sync from the UI (Sync button) or CLI: 
argocd app sync <app-name>
Verify
# Confirm the application is Synced and Healthy
kubectl get application <name> -n argocd
# Expected: SYNC STATUS = Synced, HEALTH STATUS = Healthy

# Confirm no drift remains
argocd app diff <app-name>
# Expected: no output (no differences)
ARGOCD 6.2 — Application in Degraded state
Symptom
ArgoCD UI shows HEALTH STATUS = Degraded. The application may be Synced (Git matches cluster) but the deployment health check is failing. Degraded ≠ OutOfSync: Degraded means the running pods are not healthy.
Cause
ArgoCD's health check evaluates pod status. When one or more pods under the managed deployment are in CrashLoopBackOff, ImagePullBackOff, OOMKilled, or Pending, the application health degrades. Fixing the ArgoCD status requires fixing the underlying pod failure.
Diagnose
# Step 1 — identify which namespace the ArgoCD app deploys to
kubectl get application <name> -n argocd -o jsonpath='{.spec.destination.namespace}'

# Step 2 — check pod health in that namespace
kubectl get pods -n <target-ns>

# Step 3 — identify the STATUS of each pod
# CrashLoopBackOff → Section 5.2
# ImagePullBackOff → Section 5.1
# Pending          → Section 5.3 (Kyverno)
# unknown/Error    → check describe and logs
Fix
ArgoCD Degraded is always caused by pod-level failures. Fix the pod (following Section 5 paths as appropriate). ArgoCD health status updates automatically once pods are healthy — no ArgoCD-specific action is required beyond fixing the pods.
Verify
# Once pods are Running, ArgoCD health should recover within 30 seconds
kubectl get application <name> -n argocd -w

# Watch for HEALTH STATUS to change from Degraded to Healthy
# Expected within 60s of pods reaching Running state
Related
If the ArgoCD app deployed the wrong image (e.g. after a bad values.yaml commit), update the values.yaml in Git with the correct image tag and push. ArgoCD will detect the change on the next poll cycle (≤3 minutes) and sync automatically.

7. Quick diagnosis one-liners

Copy-paste commands for the most common "something is wrong, where do I start?" situations.

Namespace placeholder: Replace <ns> with your actual namespace. For cross-namespace sweeps, use -A (all namespaces) instead.

7.1 Cluster-wide health sweep

Run these in order when something is broken and you don't know where to start.

# 1. All pods not in Running or Completed state — any namespace
kubectl get pods -A | grep -v Running | grep -v Completed

# 2. Recent events across all namespaces — sorted by time, last 30
kubectl get events -A --sort-by=.lastTimestamp | tail -30

# 3. All ArgoCD application statuses
kubectl get applications -n argocd -o wide

# 4. Kyverno policy violations across all namespaces
kubectl get policyreport -A

# 5. HPA status across all namespaces
kubectl get hpa -A

# 6. All nodes — check for NotReady
kubectl get nodes

7.2 Per-namespace deep dive

Use after the sweep identifies a problem namespace.

# Pod list with restart counts and ages
kubectl get pods -n <ns> -o wide

# Events sorted by time — last 20 in the namespace
kubectl get events -n <ns> --sort-by=.lastTimestamp | tail -20

# All secrets and configmaps in the namespace
kubectl get secret,configmap -n <ns>

# Resource usage per pod
kubectl top pods -n <ns>

# Describe a specific pod (Events section is at the bottom)
kubectl describe pod <pod-name> -n <ns>

# Logs from a running pod (last 50 lines)
kubectl logs <pod-name> -n <ns> --tail=50

# Logs from a crashed pod (previous instance)
kubectl logs <pod-name> -n <ns> --previous --tail=50

7.3 Image verification

# Check what image a deployment is currently using
kubectl get deployment <name> -n <ns> \
  -o jsonpath='{.spec.template.spec.containers[0].image}'

# Verify the image exists in GHCR (requires docker login to ghcr.io)
docker manifest inspect ghcr.io/yarova-ca/pipeline-studio/<service>:<tag>

# Inspect labels on a local image
docker inspect <image>:<tag> --format '{{json .Config.Labels}}' | jq

# List all images in GHCR for a service (requires gh auth)
gh api \
  /orgs/yarova-ca/packages/container/pipeline-studio%2F<service>/versions \
  --jq '.[].metadata.container.tags'

7.4 Kyverno policy violations

# List all PolicyReports with failures
kubectl get policyreport -A -o json \
  | jq '.items[] | select(.summary.fail > 0) | {namespace: .metadata.namespace, name: .metadata.name, failures: .summary.fail}'

# Get the full failure details for a specific namespace
kubectl get policyreport -n <ns> -o json \
  | jq '.items[].results[] | select(.result == "fail") | {policy: .policy, rule: .rule, resource: .resources[0].name, message: .message}'

# List all active ClusterPolicies and their enforcement modes
kubectl get clusterpolicy -o custom-columns=\
NAME:.metadata.name,\
MODE:.spec.validationFailureAction,\
READY:.status.conditions[0].status

7.5 ArgoCD application inspection

# List all ArgoCD apps with sync and health status
kubectl get applications -n argocd \
  -o custom-columns=\
NAME:.metadata.name,\
SYNC:.status.sync.status,\
HEALTH:.status.health.status,\
NAMESPACE:.spec.destination.namespace

# Get full application status including resources
kubectl get application <name> -n argocd -o json | jq '.status.resources[] | select(.health.status != "Healthy")'

# Force immediate sync (bypasses the 3-minute poll interval)
kubectl annotate application <name> -n argocd \
  argocd.argoproj.io/refresh=hard --overwrite

# Get the diff between cluster state and Git state
argocd app diff <name>

7.6 Secret and configmap inspection

# List all keys in a secret (does not show values)
kubectl get secret <name> -n <ns> -o json | jq '.data | keys'

# Check a specific secret key length (for minimum-length validation)
kubectl get secret <name> -n <ns> \
  -o jsonpath='{.data.JWT_SECRET}' | base64 -d | wc -c

# Check all keys that match sensitive name patterns and their lengths
kubectl get secret <name> -n <ns> -o json \
  | jq '.data | to_entries[] | select(.key | test("SECRET|KEY|TOKEN|PASS"; "i")) | {key: .key, len: (.value | @base64d | length)}'

# Decode a specific secret key value (avoid in prod — logs may capture output)
kubectl get secret <name> -n <ns> \
  -o jsonpath='{.data.<key>}' | base64 -d

# Show all keys in a configmap
kubectl get configmap <name> -n <ns> -o json | jq '.data | keys'

7.7 Image pull secret check

# Check if ghcr-pull-secret exists in a namespace
kubectl get secret ghcr-pull-secret -n <ns>

# Check which image pull secrets are attached to the default service account
kubectl get serviceaccount default -n <ns> -o json | jq '.imagePullSecrets'

# Check all pods that are currently NOT in running state with their container status
kubectl get pods -A \
  -o jsonpath='{range .items[*]}{.metadata.namespace}{" "}{.metadata.name}{" "}{.status.containerStatuses[*].state}{"
"}{end}' | grep -v running

# Show images being used across all pods in a namespace
kubectl get pods -n <ns> \
  -o jsonpath='{range .items[*]}{.metadata.name}{" "}{.spec.containers[*].image}{"
"}{end}'

gh run list -R yarova-ca/pipeline-studio --limit 10

9. Environment-specific notes

Differences between local, staging, and production debugging contexts.

Context How to connect Extra constraints Safe to use fix commands?
Local (kind/minikube) kubectl is already pointed at the local cluster. No auth needed. metrics-server requires --kubelet-insecure-tls. Pull secrets may not be needed for locally-loaded images. Yes — full access. Destroying a local cluster is not a production incident.
Staging Set context: kubectl config use-context staging Kyverno policies are enforced. Secrets come from Vault. ArgoCD sync is automated. Yes for read commands. Fix commands require review — staging is used for demos.
Production Set context: kubectl config use-context production All policies enforced. FIPS may be required. All changes must go through ArgoCD — direct kubectl apply is blocked by Kyverno. Read-only only. All fixes via Git commit → ArgoCD sync. Never kubectl edit in prod.
Production rule: In production, all changes must go through Git and ArgoCD.
Direct kubectl apply, kubectl edit, or kubectl patch in the production cluster will:
1. Create drift that ArgoCD will detect and revert on the next sync.
2. Possibly be blocked by a Kyverno policy that prevents unauthorized mutations.
Fix production by committing to the Helm chart in Git and triggering an ArgoCD sync.

10. Escalation checklist

Before escalating a failure to another person, confirm you have run through this checklist.

Why this exists: 80% of escalations are resolved by the person looking at the same data the escalator already had access to. Complete this list before adding another person to the incident.
  • 1 Identified which pod is failing. kubectl get pods -A | grep -v Running | grep -v Completed
  • 2 Read the pod events section. kubectl describe pod <name> -n <ns> | tail -30 The Events section at the bottom of describe output names the exact failure in plain English.
  • 3 Read the previous pod logs (for CrashLoopBackOff). kubectl logs <name> -n <ns> --previous --tail=50
  • 4 Checked all secret keys are present with correct length. kubectl get secret <name> -n <ns> -o json | jq '.data | keys'
  • 5 Verified the image exists in GHCR at the correct tag. docker manifest inspect ghcr.io/yarova-ca/pipeline-studio/<service>:<sha>
  • 6 Checked the image labels to confirm COMPLIANCE and RUNTIME are correct. docker inspect <image>:<tag> --format '{{json .Config.Labels}}' | jq
  • 7 Checked Kyverno policy violations in the namespace. kubectl get policyreport -n <ns> -o json | jq '.items[].results[] | select(.result == "fail")'
  • 8 Checked the CI pipeline run for the commit being deployed. gh run list -R yarova-ca/pipeline-studio --limit 5
  • 9 Matched the symptom to the failure modes master table (Section 8).
  • 10 Read the full matching section (Section 2–6) for this failure. Attempted the fix steps documented there.

When escalating, share:

  • A The pod name, namespace, and STATUS from kubectl get pods.
  • B The Events section from kubectl describe pod.
  • C The last 30 lines from kubectl logs --previous.
  • D Which section of this guide was followed and what the result was.
