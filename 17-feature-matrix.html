What is this page?

The service-starters repo has 105 ready-to-run service templates — one for each framework and language. Out of the box, each template is a minimal "hello world" service with health endpoints.

This page explains 7 ways to customize any service — swapping the base image, adding compliance rules, wiring up authentication, connecting a database — all with a single command. No editing Dockerfiles. No writing code.

Here is the full command. Each --build-arg is one customization:

docker build --target runtime \
  --build-arg RUNTIME=chainguard \
  --build-arg COMPLIANCE=hipaa \
  --build-arg AUTH=jwt \
  --build-arg OBSERVABILITY=otel \
  --build-arg ORM=gorm \
  -t my-service services/16-gin
  • RUNTIME=chainguard → which base Linux image to use (affects security and CVEs)
  • COMPLIANCE=hipaa → which regulatory rules to activate (HIPAA, PCI, FIPS, PIPEDA, etc.)
  • AUTH=jwt → which authentication method to enforce (JWT, OAuth2, API key, mTLS, etc.)
  • OBSERVABILITY=otel → which monitoring instrumentation to add (OpenTelemetry, Prometheus, Datadog)
  • ORM=gorm → which database library to include (GORM for Go, Prisma for Node, SQLAlchemy for Python, etc.)

What is a build axis?

An axis = one dimension you can change about how a service is built and what it includes when it runs.

📁
Start: a service template from the repo services/16-gin — a minimal Go API server. Health endpoints only. No auth, no database, no compliance rules.
↓ pass to
🔨

--build-arg AUTH=jwt \
--build-arg OBSERVABILITY=otel \
--build-arg ORM=gorm
↓ produces
✅
A running container that has: HIPAA rules active — patient data masked in logs, audit trail on
JWT auth active — requests without a valid token → 401 Unauthorized
OpenTelemetry active — traces and metrics sent to your monitoring tool
GORM wired up — PostgreSQL connection and sample schema ready
Important: which axes work today vs which are coming soon
RUNTIME — Implemented — code exists in every applicable service. You can use it right now.
All other 6 axes — Designed — fully documented and planned, but not yet built into service source code. See Implementation status for details.

The 7 axes — what each one does

Each axis is a --build-arg KEY=value flag. Click any axis to expand its values table and example command.

1 Runtime Variant RUNTIME Implemented

Swaps the base Linux image your service container runs on. Every Docker container starts from a base image — this axis lets you choose which one, trading off image size, security posture, and compliance eligibility. This is the only axis implemented today. You can use it right now on all 28 Tier 1 services.

Why you'd use this
Use fips when your industry legally requires FIPS-validated cryptography (US federal government, defense contractors, healthcare).
Use slim or distroless to reduce the attack surface — no shell, no package manager in the running image.
Use alpine for the smallest standard image with broad ecosystem support.
Use scratch for Go and Rust static binaries — the absolute minimum: zero OS, only your binary.
CVE (Common Vulnerability and Exposure) = a publicly documented security flaw in software.
FIPS-140-2 = US government standard for cryptographic modules. Required for federal IT systems.
Distroless = image with no shell and no package manager. Minimal attack surface. Default for Java and Kotlin.
Scratch = completely empty image — only what you COPY into it. Default for Go and Rust static binaries.
SBOM (Software Bill of Materials) = a signed list of every dependency in the image, generated automatically by some base images.
How it works inside every Dockerfile (3 steps)

Step 1 — Build stage creates a minimal user file.

RUN printf 'root:x:0:0:root:/root:/bin/sh\n\
nobody:x:65534:65534:nobody:/nonexistent:/bin/false\n\
app:x:1001:1001::/home/app:/sbin/nologin\n' > /tmp/app-passwd \
 && printf 'root:x:0:\nnobody:x:65534:\napp:x:1001:\n' > /tmp/app-group

Why: different base images use incompatible commands to create users — adduser on Alpine, useradd on Debian, and no command at all on scratch or distroless (which have no shell). Creating the user file once in the build stage and copying it into any runtime image solves all four cases with one mechanism.

Step 2 — All base image options defined as named stages.

# Node.js example — each language has its own set
FROM node:22-alpine AS base-alpine
FROM node:22-slim   AS base-slim
FROM registry.access.redhat.com/ubi9/nodejs-22-minimal AS base-fips

Docker BuildKit (the modern Docker build engine) only pulls the stage that is selected. The other two are skipped entirely — no extra pull time.

Step 3 — ARG selects which base image becomes the runtime stage.

ARG RUNTIME=alpine
FROM base-${RUNTIME} AS runtime
WORKDIR /app
COPY --from=build /tmp/app-passwd /etc/passwd
COPY --from=build /tmp/app-group  /etc/group
COPY --from=build --chown=1001:1001 /app/dist ./dist
USER 1001

The /etc/passwd copy is what makes USER 1001 work in scratch and distroless images — they have no adduser command, but COPY creates the file directly. A separate runtime-fips named stage is kept for backward compatibility with --target runtime-fips.

2 Package Manager Swap PKG_MANAGER Designed

Swaps npm for a different JS/TS package manager inside the Dockerfile.

Why JS/TS can swap: npm, pnpm, yarn, and bun all read the same package.json file. The input file does not change — only which tool reads it changes. This is why all four are interchangeable with one --build-arg.

Package manager = tool that downloads and installs code libraries your project depends on.
Lock file = a file that records the exact version of every library installed, so the build is identical every time.

3 Build Tool Swap BUILD_TOOL Designed

Swaps Vite for a different JS/TS bundler inside the Dockerfile. Applies to JS/TS groups only (01–08, 13, 14).

Why only JS/TS: Bundling exists because JS/TS code runs in browsers. A browser loads files over the internet — 500 separate file requests is too slow. A bundler combines those 500 files into 2–3 optimized files before deployment.

Go, Python, Java, Rust, Ruby, PHP, Elixir, Swift, C# — these run on servers, not browsers. They compile to one binary or run source files directly. There are no separate files to bundle. The concept does not exist for these languages.

Bundler = tool that combines many JS/TS source files into fewer optimized files for delivery to a browser.
HMR (Hot Module Replacement) = feature that updates the browser instantly when you save a file during development, without a full page reload.

4 Compliance Preset COMPLIANCE Designed

Adds the security code your industry's regulations require — automatically, without you writing it. This includes middleware that runs on every request, response headers, startup checks, and log masking. Applies to all server groups.

Why you'd use this Your healthcare app must mask patient data in logs (HIPAA). Your payment app must enforce TLS and remove debug endpoints (PCI). Your Canadian app must consent-check every user request (PIPEDA). One arg handles all of this.
Compliance = rules that governments, regulators, or card networks require your software to follow. Violations can result in fines, audits, or loss of operating license.
Middleware = code that runs on every incoming HTTP request before your route handler. Compliance middleware checks headers, masks data, and writes audit logs.
PHI (Protected Health Information) = patient data covered by HIPAA law in the US.
PCI DSS = Payment Card Industry Data Security Standard — required for any service that handles card data.
⚠ Rule: COMPLIANCE=fips requires RUNTIME=ubi9. The FIPS crypto module only ships in the UBI9 image. Building with fips + any other RUNTIME will fail with an error message.

5 Observability Stack OBSERVABILITY Designed

Adds code that reports what your service is doing in production — how fast requests are, how many errors occur, where time is being spent, and what path each request took through your system. Applies to all server groups.

Why you'd use this Without observability, when something breaks in production you are blind. You cannot tell if it is slow, erroring, or completely down without external monitoring. With otel, a dashboard like Grafana shows you request rates, latencies, and error traces in real time.
OpenTelemetry (OTel) = open standard for collecting traces, metrics, and logs from services. Vendor-neutral — works with Grafana, Jaeger, Datadog, Dynatrace, and others.
Trace = a timeline of one request moving through your system — which functions ran, how long each took.
Metric = a number measured over time — request count per second, error rate, database query latency.
OTLP exporter = the component that sends your telemetry data to your monitoring backend.

6 Auth Strategy AUTH Designed

Adds a verification gate at the entry point of your API. Every HTTP request must prove who it is before it reaches your route handler. Unauthenticated requests are rejected with 401 Unauthorized before any business logic runs. Applies to all server groups.

Why you'd use this Without auth, anyone who knows your URL can call your API — read data, trigger actions, cause damage. With AUTH=jwt, only callers holding a valid signed token can get through. With AUTH=mtls, only callers with a specific client certificate are allowed — used in zero-trust defense and government systems.
JWT (JSON Web Token) = a signed string passed in the HTTP Authorization: Bearer … header. Your service validates the signature cryptographically — no database lookup required.
OAuth2 = industry-standard protocol for delegating access. Powers "Log in with Google" and service-to-service API access.
OIDC (OpenID Connect) = identity layer on top of OAuth2. Returns a verified user identity, not just an access token.
mTLS (Mutual TLS) = both the client and server present certificates. Client must have a specific cert issued by your CA. Used in defense and zero-trust architectures.
⚠ Rule: AUTH=mtls requires RUNTIME=ubi9 or RUNTIME=chainguard. mTLS requires FIPS-grade TLS libraries that only ship in those base images.

7 ORM / Database Layer ORM Designed

Adds a database connection library and a starter schema to your service so it can read and write to a database. Each language group gets the idiomatic database library for that language. Applies to pure API server groups 14–27 only.

Why you'd use this Without this axis, your service has no database. You would manually install the right library, write the connection config, and create a starter schema. With ORM=gorm on a Go service, GORM is installed, your PostgreSQL connection is configured, and a sample schema is ready to extend. With ORM=prisma on a Node.js service, Prisma client and schema are in place.
ORM (Object-Relational Mapper) = library that maps database rows to code objects. You work with structs/classes in your language — the ORM translates them to SQL queries.
Schema = the structure of your database: which tables exist, which columns they have, which are required.
Why ORM does not apply to groups 28–30: Protocol servers (gRPC, GraphQL, WebSocket) are transport layers — they pass messages. The service that owns the data and the database is upstream of the protocol server.

Real-world examples

Four worked examples. Each shows the full command, what each arg does, and what the running service has when it starts.

1 No customization — bare starter service

Scenario: You just want to run the starter and see it work. No compliance, no auth, no database.

docker build --target runtime -t my-service services/16-gin
docker run --rm -p 8080:8080 my-service
What the running service has Standard Debian base image · No compliance rules · No auth (all requests pass through) · No database · GET /health → {"status":"ok"} · GET /health/live and /health/ready → {"status":"ok"}
2 Healthcare Go API — HIPAA + JWT + OpenTelemetry + database

Scenario: Building a US healthcare API. Needs HIPAA compliance, JWT auth for all routes, traces visible in Grafana, and a PostgreSQL database.

docker build --target runtime \
  --build-arg COMPLIANCE=hipaa \
  --build-arg AUTH=jwt \
  --build-arg OBSERVABILITY=otel \
  --build-arg ORM=gorm \
  -t healthcare-api services/16-gin
  • COMPLIANCE=hipaa → PHI fields masked in all log output · audit trail middleware active · verbose errors disabled
  • AUTH=jwt → every request must include Authorization: Bearer <token> · missing or invalid token → 401
  • OBSERVABILITY=otel → OpenTelemetry SDK active · request traces sent to your OTLP backend (Grafana, Jaeger, etc.)
  • ORM=gorm → GORM installed · PostgreSQL connection configured · starter schema ready to extend
What the running service has Standard Debian base · HIPAA log masking and audit trail · JWT auth on all routes · OpenTelemetry traces and metrics · GORM + PostgreSQL connection
3 Hardened production Node.js API — zero CVEs + SOC 2 + JWT + OTel + Prisma

Scenario: SaaS platform deploying to production. Security team requires zero CVEs. SOC 2 audit required. Full observability and database needed.

docker build --target runtime \
  --build-arg RUNTIME=chainguard \
  --build-arg COMPLIANCE=soc2 \
  --build-arg AUTH=jwt \
  --build-arg OBSERVABILITY=otel \
  --build-arg ORM=prisma \
  -t prod-api services/14-express
  • RUNTIME=chainguard → zero CVEs in base image · supply-chain signed · SBOM generated automatically
  • COMPLIANCE=soc2 → immutable audit log · access control middleware · debug mode disabled
  • AUTH=jwt → Bearer token validation on all routes
  • OBSERVABILITY=otel → full traces, metrics, and logs via OpenTelemetry
  • ORM=prisma → Prisma client installed · schema and database connection ready
What the running service has Chainguard zero-CVE base · SOC 2 audit log · JWT auth · OpenTelemetry instrumentation · Prisma + database
4 Federal/defense Java API — FIPS crypto + mTLS client certificates

Scenario: Building a service for a US defense contractor. FIPS-validated cryptography required. Client certificate authentication required for every call. Note: FIPS and ubi9 must be used together.

docker build --target runtime \
  --build-arg RUNTIME=ubi9 \
  --build-arg COMPLIANCE=fips \
  --build-arg AUTH=mtls \
  -t defense-api services/17-spring
  • RUNTIME=ubi9 → Red Hat UBI9 base · required for FIPS — using fips without ubi9 = build error
  • COMPLIANCE=fips → FIPS-140-2 OpenSSL module active · non-FIPS cipher suites disabled at startup
  • AUTH=mtls → client certificate required on every request · requests without a valid cert → 401 · requires ubi9 or chainguard base
What the running service has UBI9 FIPS base · FIPS-140-2 crypto enforced · mTLS client certificate required on every request

Notes on * partial cells

Some groups get * instead of Y — the axis applies, but works differently than the standard HTTP middleware pattern.

Groups 01, 04, 05, 07 — COMPLIANCE and OBSERVABILITY *

Pattern: HTTP handler level — same as pure API servers.
Behavior: applies fully. The implementation is identical to groups 14–27.
Why *: reminder that these are full-stack frameworks with server-side rendering. Verify the HTTP handler entry point before adding middleware — it is not always a plain Express/Gin-style function.

Groups 01, 04, 05, 07 — AUTH *

What works: JWT Bearer middleware on /api/* routes within the framework.
What does not work: standard JWT middleware on SSR page routes — those use cookies and session-based auth, not Bearer tokens.
Next.js page routes: use Next-Auth for page-level auth. JWT Bearer middleware for /api/* routes only.
SvelteKit page routes: use hooks.server.ts sessions. JWT Bearer for API endpoints only.
Remix page routes: use Remix sessions for page auth. JWT Bearer for resource routes only.

Group 06 — PKG_MANAGER *

Applies via: lock file used by wrangler deploy — not a docker build --build-arg.
Does not apply: RUNTIME, BUILD_TOOL — group 06 (Cloudflare Workers) has no Dockerfile.
How to set: configure the packageManager field in package.json. Wrangler reads it automatically.
