What is this page?

The service-starters repo has 105 ready-to-run service templates — one for each framework and language. Out of the box, each template is a minimal "hello world" service with health endpoints.

This page explains 8 ways to customize any service — swapping the build image, swapping the runtime image, adding compliance rules, wiring up authentication, connecting a database — all with a single command. No editing Dockerfiles. No writing code.

Here is the full command. Each --build-arg is one customization:

docker build --target runtime \
  --build-arg BUILD_IMAGE=sdk \
  --build-arg RUNTIME=slim \
  --build-arg COMPLIANCE=hipaa \
  --build-arg AUTH=jwt \
  --build-arg OBSERVABILITY=otel \
  --build-arg ORM=gorm \
  -t my-service services/16-gin
  • BUILD_IMAGE=sdk → which image runs the compiler and SDK (supply chain trust boundary)
  • RUNTIME=slim → which base Linux image the service runs on (affects security and CVEs)
  • COMPLIANCE=hipaa → which regulatory rules to activate (HIPAA, PCI, FIPS, PIPEDA, etc.)
  • AUTH=jwt → which authentication method to enforce (JWT, OAuth2, API key, mTLS, etc.)
  • OBSERVABILITY=otel → which monitoring instrumentation to add (OpenTelemetry, Prometheus, Datadog)
  • ORM=gorm → which database library to include (GORM for Go, Prisma for Node, SQLAlchemy for Python, etc.)

Built-in features — no --build-arg needed

These features exist in every applicable service today. No configuration. No build-arg. Always present.

✓ Health Endpoints GET /health · /health/live · /health/ready Implemented

Every server service automatically exposes 3 HTTP health routes from the moment it starts.
No build-arg. No configuration. No code to write. They are wired in at the framework layer.

Why health endpoints matter
Kubernetes uses these routes to manage the container lifecycle.
Without them: Kubernetes cannot know if your service is alive or ready for traffic.
With them: Kubernetes automatically restarts broken pods and routes traffic correctly.
Liveness probe = Kubernetes checks this every few seconds. Non-200 → pod is restarted.
Readiness probe = Kubernetes checks this before sending traffic. Non-200 → pod is taken out of the load balancer rotation.
Probe = an automated HTTP call Kubernetes makes to the container to check its health.
Routes
RoutePurposeReturnsWhen Kubernetes uses this
GET /healthManual health check — call this yourself to verify the service is up200 {"status":"ok"}Not a probe target. For human/monitoring use.
GET /health/liveKubernetes liveness probe200 {"status":"ok"}Non-200 → Kubernetes restarts the pod
GET /health/readyKubernetes readiness probe200 {"status":"ok"}Non-200 → pod removed from load balancer rotation

Example:

# Test locally — replace 3000 with your service port
curl http://localhost:3000/health
# Returns: {"status":"ok"}

curl http://localhost:3000/health/live
# Returns: {"status":"ok"}

curl http://localhost:3000/health/ready
# Returns: {"status":"ok"}

Kubernetes probe config example (in your deployment YAML):

livenessProbe:
  httpGet:
    path: /health/live
    port: 3000
  initialDelaySeconds: 5
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /health/ready
    port: 3000
  initialDelaySeconds: 3
  periodSeconds: 5

The 8 axes — what each one does

Each axis is a --build-arg KEY=value flag. Click any axis to expand its values table and example command.

1 Build Stage Image BUILD_IMAGE Implemented

Swaps the base image used in the compilation stage of the Dockerfile. The build stage runs the compiler, SDK, and package manager — then is discarded. Only the runtime stage ships to production. This axis controls the supply chain trust boundary at build time.

Why it matters: A compromised build image can inject malicious code into your binary before the runtime stage strips everything else. All other axes plus health endpoints protect the running container — this axis protects what creates it.

Scope: Applies to all 99 services that have a Dockerfile. Does not apply to group 06 (Cloudflare Workers — no Dockerfile) and groups 09–12 (mobile/native — no Docker).

Why you'd use this
Use sdk when your security team requires using the official language maintainer's image (not Ubuntu + apt install).
Use sdk-alpine to reduce the build stage size — faster CI, smaller layer cache.
Use custom to pull from your internal registry instead of Docker Hub (air-gapped environments, vetted internal images).
Supply chain attack = injecting malicious code into software during the build process, before the final image is assembled.
Internal registry = your organization's private image store — images scanned, approved, and hosted internally.
SBOM (Software Bill of Materials) = signed list of every dependency in the image — generated automatically by some base images.

2 Runtime Variant RUNTIME Implemented

Swaps the base Linux image your service container runs on. Every Docker container starts from a base image — this axis lets you choose which one, trading off image size, security posture, and compliance eligibility. This is the only axis implemented today.

Applies to 28 of the 34 Tier 1 services — the 28 that build to Docker containers.
The other 6 Tier 1 services have no Docker container: 06-hono-edge (Cloudflare Workers), 09-react-native, 09-expo, 10-flutter, 11-swift-swiftui, 12-kotlin-jetpack (mobile/native apps).
--build-arg RUNTIME= is a Docker-only argument. When there is no Dockerfile: this arg does not exist.

Why you'd use this
Use fips when your industry legally requires FIPS-validated cryptography (US federal government, defense contractors, healthcare).
Use slim or distroless to reduce the attack surface — no shell, no package manager in the running image.
Use alpine for the smallest standard image with broad ecosystem support.
Use scratch for Go and Rust static binaries — the absolute minimum: zero OS, only your binary.
CVE (Common Vulnerability and Exposure) = a publicly documented security flaw in software.
FIPS-140-2 = US government standard for cryptographic modules. Required for federal IT systems.
Distroless = image with no shell and no package manager. Minimal attack surface. Default for Java and Kotlin.
Scratch = completely empty image — only what you COPY into it. Default for Go and Rust static binaries.
SBOM (Software Bill of Materials) = a signed list of every dependency in the image, generated automatically by some base images.
How it works inside every Dockerfile (3 steps)

Step 1 — Build stage creates a minimal user file.

Background: why does USER 1001 need a file?

Every Linux system has a file at /etc/passwd.
That file maps usernames to user IDs (UIDs).
When a container starts and hits USER 1001, Docker looks up UID 1001 in /etc/passwd.

When /etc/passwd has the entry for UID 1001: container starts, app runs as that user.
When /etc/passwd is missing the entry: container refuses to start.

The problem: four image types, four incompatible tools

Base image typeHow to add a userHas a shell?Can run RUN commands?
Alpineadduser -S -G app app✅ Yes✅ Yes
Debian / slimuseradd -u 1001 -r -g app app✅ Yes✅ Yes
scratch (empty image)No command exists❌ No shell❌ No
distroless (no shell)No command exists❌ No shell❌ No

scratch = a completely empty image. Zero OS. Zero tools. Only what you COPY into it.
distroless = an image with no shell and no package manager — only the language runtime.
Both accept COPY. Neither accepts RUN. RUN requires a shell to execute.

The solution: make the file in the build stage, COPY it everywhere

The build stage runs on ubuntu:24.04, which has a full shell.
The printf command writes a three-line text file to /tmp/app-passwd.
Each line is one user entry in the standard Unix passwd format.

RUN printf 'root:x:0:0:root:/root:/bin/sh\n\
nobody:x:65534:65534:nobody:/nonexistent:/bin/false\n\
app:x:1001:1001::/home/app:/sbin/nologin\n' > /tmp/app-passwd \
 && printf 'root:x:0:\nnobody:x:65534:\napp:x:1001:\n' > /tmp/app-group

What each field in an /etc/passwd line means:

LinenamepasswordUIDGIDcommenthomeshell

nobody:x:65534:65534:nobody:/nonexistent:/bin/falsenobodyx6553465534nobody/nonexistent/bin/false

/sbin/nologin as the shell = this user cannot log in interactively.
This is intentional — a service user must not be loginable.
The container process runs as UID 1001 but no human can SSH in as that user.

Later, in the runtime stage:
COPY --from=build /tmp/app-passwd /etc/passwd writes the file directly into the image layer.
No shell needed. No adduser needed. Works in scratch. Works in distroless.

The /etc/group file does the same thing for group IDs.
app:x:1001: = group named app, GID 1001, no extra members.
Some tools (nginx, Java) log warnings if the group entry is missing.

Step 2 — All base image options defined as named stages.

# Node.js example — each language has its own set
FROM node:22-alpine AS base-alpine
FROM node:22-slim   AS base-slim
FROM registry.access.redhat.com/ubi9/nodejs-22-minimal AS base-fips

Docker BuildKit (the modern Docker build engine) only pulls the stage that is selected. The other two are skipped entirely — no extra pull time.

Step 3 — ARG selects which base image becomes the runtime stage.

ARG RUNTIME=alpine
FROM base-${RUNTIME} AS runtime
WORKDIR /app
COPY --from=build /tmp/app-passwd /etc/passwd
COPY --from=build /tmp/app-group  /etc/group
COPY --from=build --chown=1001:1001 /app/dist ./dist
USER 1001

The /etc/passwd copy is what makes USER 1001 work in scratch and distroless images — they have no adduser command, but COPY creates the file directly. A separate runtime-fips named stage is kept for backward compatibility with --target runtime-fips.

Why you'd use this Without auth, anyone who knows your URL can call your API — read data, trigger actions, cause damage. With AUTH=jwt, only callers holding a valid signed token can get through. With AUTH=mtls, only callers with a specific client certificate are allowed — used in zero-trust defense and government systems.
JWT (JSON Web Token) = a signed string passed in the HTTP Authorization: Bearer … header. Your service validates the signature cryptographically — no database lookup required.
OAuth2 = industry-standard protocol for delegating access. Powers "Log in with Google" and service-to-service API access.
OIDC (OpenID Connect) = identity layer on top of OAuth2. Returns a verified user identity, not just an access token.
mTLS (Mutual TLS) = both the client and server present certificates. Client must have a specific cert issued by your CA. Used in defense and zero-trust architectures.
⚠ Rule: AUTH=mtls requires RUNTIME=fips.
mTLS requires FIPS-grade TLS libraries. RUNTIME=fips selects the UBI9 (Red Hat) base image which includes them.
Using AUTH=mtls with any other RUNTIME value will fail — the required TLS libraries are absent.

Notes on * partial cells

Some groups get * instead of Y — the axis applies, but works differently than the standard HTTP middleware pattern.

Groups 01, 04, 05, 07 — COMPLIANCE and OBSERVABILITY *

Pattern: HTTP handler level — same as pure API servers.
Behavior: applies fully. The implementation is identical to groups 14–27.
Why *: reminder that these are full-stack frameworks with server-side rendering. Verify the HTTP handler entry point before adding middleware — it is not always a plain Express/Gin-style function.

Groups 01, 04, 05, 07 — AUTH *

What works: JWT Bearer middleware on /api/* routes within the framework.
What does not work: standard JWT middleware on SSR page routes — those use cookies and session-based auth, not Bearer tokens.
Next.js page routes: use Next-Auth for page-level auth. JWT Bearer middleware for /api/* routes only.
SvelteKit page routes: use hooks.server.ts sessions. JWT Bearer for API endpoints only.
Remix page routes: use Remix sessions for page auth. JWT Bearer for resource routes only.

Group 06 — PKG_MANAGER *

Applies via: lock file used by wrangler deploy — not a docker build --build-arg.
Does not apply: RUNTIME, BUILD_TOOL — group 06 (Cloudflare Workers) has no Dockerfile.
How to set: configure the packageManager field in package.json. Wrangler reads it automatically.
