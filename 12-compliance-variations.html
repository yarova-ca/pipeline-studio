How compliance affects Dockerfiles

Read this first. It explains the model — why compliance changes specific Dockerfile lines, and how to apply it.

The additive compliance model

1
A Dockerfile is the blueprint for what is inside the container. The container image is part of the system. Compliance standards govern the whole system — including the container image.
2
You do not rewrite the Dockerfile per compliance standard. You start from the base Dockerfile in doc 11. Then you apply a compliance layer on top. The layer adds or replaces specific instructions.
3
Doc 11 already handles FIPS via the --target runtime-fips named target. That is one example of the compliance layer model. The base Dockerfile is unchanged. The FIPS layer swaps the runtime base image to UBI9.
4
The 5 compliance additions in section 5 are the remaining layers. Apply whichever your industry requires. Each addition is independent — you can apply one, several, or all five to the same Dockerfile.
5
Compliance enforcement that does NOT change the Dockerfile — data residency (GDPR, PIPEDA), image signing (FedRAMP), and registry air-gapping (NERC CIP) — is enforced at the CI/CD pipeline and infrastructure layer, not the Dockerfile itself.

The 5 concrete Dockerfile additions

Copy-paste additions for the most common compliance requirements. Apply on top of any doc 11 Dockerfile. Each is independent.

1. HEALTHCHECK instruction
Applies to: FedRAMP (required) · HIPAA (required) · SOC 2 (required) · NERC CIP (required)
Orchestrators (Kubernetes, ECS) use HEALTHCHECK to determine liveness. Compliance auditors require proof the container self-reports health status.
# Add inside the runtime stage, before CMD HEALTHCHECK --interval=30s --timeout=5s --start-period=15s --retries=3 \ CMD curl -f http://localhost:$PORT/health || exit 1
Replace $PORT with your framework port. For scratch/distroless runtimes: copy a static health-check binary instead of curl.
2. LABEL provenance
Applies to: FedRAMP (required) · CMMC (required) · SOX (required) · PCI DSS (recommended) · NERC CIP (recommended)
Every deployed image must be traceable to a commit, a build pipeline run, and an approver. Labels are the standard mechanism.
# Add inside the runtime stage ARG GIT_SHA=unknown ARG BUILD_DATE=unknown ARG COMPLIANCE=none LABEL org.opencontainers.image.created="$BUILD_DATE" \ org.opencontainers.image.revision="$GIT_SHA" \ org.opencontainers.image.title="myapp" \ org.compliance.standard="$COMPLIANCE" \ org.compliance.stig-version="V1R3" # Build with: # docker build --build-arg GIT_SHA=$(git rev-parse HEAD) \ # --build-arg BUILD_DATE=$(date -u +%Y-%m-%dT%H:%M:%SZ) \ # --build-arg COMPLIANCE=fedramp \ # --target runtime .
ARG values do not persist in the final image ENV — they only set LABEL values at build time. No secret exposure risk.
3. Non-root user with specific UID
Applies to: PCI DSS (mandatory) · HIPAA (mandatory) · FedRAMP (mandatory UID 1001) · CMMC (mandatory)
If a process running as root escapes the container, it has root on the host. A specific UID (not just "a non-root user") allows Kubernetes PodSecurityPolicy to enforce runAsNonRoot.
# Add inside the runtime stage before EXPOSE # Alpine / scratch pattern: RUN addgroup -g 1001 -S appgroup && adduser -u 1001 -S appuser -G appgroup USER 1001:1001 # Debian / Ubuntu pattern: RUN groupadd -g 1001 appgroup && \ useradd -u 1001 -g 1001 -s /sbin/nologin -M appuser USER 1001:1001
Doc 11 already sets USER. This addition makes the UID explicit (1001) for FedRAMP and CMMC auditors who check UID values, not just "non-root".
4. Base image swap for compliance tier
Applies to: FIPS/FedRAMP (UBI9 required) · PCI DSS (CIS/distroless required) · NERC CIP (air-gapped registry required)
The runtime base image determines which crypto libraries are available at runtime. FIPS requires the OS to use only FIPS-validated crypto — the base image controls this.
# Doc 11 already has --target runtime and --target runtime-fips. # Additional targets for other compliance tiers: # ── runtime-pci (PCI DSS: distroless, minimal attack surface) ────────────── FROM cgr.dev/chainguard/node:latest AS runtime-pci WORKDIR /app COPY --from=builder /app/dist ./dist USER 65532 EXPOSE 3000 CMD ["node", "dist/server.js"] # ── runtime-nerc (NERC CIP: air-gapped — replace registry.internal with yours) FROM registry.internal.yourco.com/base/node:22-alpine AS runtime-nerc WORKDIR /app COPY --from=builder /app/dist ./dist USER 1001:1001 EXPOSE 3000 CMD ["node", "dist/server.js"]
For NERC CIP: all FROM references must point to your internal air-gapped registry. Never pull from docker.io, gcr.io, or quay.io in a CIP environment.
5. Tool removal in builder stage
Applies to: PCI DSS (required) · CIS Benchmark Level 2 (required) · ISO 27001 (recommended)
Tools left in an image layer are accessible if the container is compromised. curl, wget, and package managers are attack vectors. Removing them from the runtime stage eliminates them.
# Builder stage: install tools, use them, then remove in same RUN layer FROM ubuntu:24.04 AS builder RUN apt-get update && \ apt-get install -y curl build-essential ca-certificates && \ # ... your build steps here ... \ apt-get remove -y curl build-essential && \ apt-get autoremove -y && \ rm -rf /var/lib/apt/lists/* /tmp/* /root/.cache # Runtime stage has none of these tools because it starts from a clean base FROM node:22-alpine AS runtime # curl is not present in this stage at all
Multi-stage builds (doc 11 pattern) already achieve this for the runtime stage. This addition applies to cases where a single-stage is unavoidable or the builder stage itself is audited.

Gap analysis — doc 11 vs compliance requirements

What doc 11 already handles, and what must be added for full compliance coverage.

✓ Already covered in doc 11
  • ✓ FIPS base image via --target runtime-fips (all 75 frameworks)
  • ✓ Non-root USER in every runtime stage
  • ✓ CA certificates copied for scratch/distroless runtimes
  • ✓ FIPS note in every verify block
  • ✓ HEALTHCHECK in all 75 runtime stages (--interval=30s --timeout=5s --retries=3)
  • ✓ LABEL provenance — org.opencontainers.image.revision and org.opencontainers.image.created in all runtime stages
  • ✓ Explicit numeric UID — USER 1001:1001 in all non-Deno frameworks
✗ One edge case remaining in doc 11
  • ✗ Fresh (Deno) uses USER deno — not a numeric UID (required: FedRAMP, CMMC). Fix: replace with USER 65532:65532 (Deno's system UID) when FedRAMP or CMMC compliance is required.

Data residency (GDPR, PIPEDA), image signing (FedRAMP, CMMC), registry air-gapping (NERC CIP), and SIEM log shipping (all standards) are enforced at the CI/CD and infrastructure layer — they cannot be expressed in a Dockerfile.
