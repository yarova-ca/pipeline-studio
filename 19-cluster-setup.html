Cluster Setup
What this covers: How to create and configure the 4-cluster platform (hub + dev + test + prod) on any cloud using Terraform and ArgoCD.
When to read: Before provisioning any cluster. Before registering a new cloud provider. When the platform needs a new environment.
Pillar 7 — Maintaining · Related: 16 Runbook · DECISIONS.md

1. What a cluster is — plain English

A cluster is a group of servers that run containers.

Your 105 services run as containers inside the cluster.

Kubernetes manages which server each container runs on.

You do not manually decide which server — Kubernetes does that automatically.

How your code gets from laptop to running service

Your laptop → GitHub → GitHub Actions → Container image in GHCR
↓
ArgoCD sees new image in GHCR
↓
Kubernetes pulls image into the cluster
↓
14-express is running at https://14-express.yarova.ca

Each step is automated. A git push to main triggers the full chain.

No manual kubectl apply needed after initial setup.

2. The 4-cluster architecture

Hub and spoke — plain English

The platform uses one hub cluster and three spoke clusters.

  • Hub cluster: runs ArgoCD only. No user services live here.
  • Dev cluster: all 105 services, 1 replica each, debug logging on.
  • Test cluster: all 105 services, 2 replicas, staging config.
  • Prod cluster: all 105 services, 3+ replicas, autoscaling, strict security.

Why separate clusters

Dev can be broken without affecting prod.

Test uses the same config as prod — this catches config bugs before they hit prod.

Hub runs management tools — if prod crashes, hub is unaffected and can redeploy.

Per-cluster configuration comparison

Config Hub Dev Test Prod
Runs services No Yes (1 replica) Yes (2 replicas) Yes (3+ replicas)
Autoscaling Off Off Off On
Kyverno mode Audit Audit Audit Enforce
TLS cert Self-signed Let's Encrypt staging Let's Encrypt staging Let's Encrypt prod
Node size Small Small Medium Large
Debug logging Off On Off Off

3. Terraform — what it does and why

Terraform is a tool that creates cloud infrastructure from code files.

You write what you want (e.g. a cluster with 3 nodes on EKS) in a .tf file.

Terraform creates it.

Run Terraform again: it checks what already exists, and only changes what's different.

State file: Terraform keeps a record of what it created.

The state file is stored remotely in S3 (AWS), GCS (GCP), or Azure Blob (Azure).

Without the state file, Terraform does not know what it already created.

Why Terraform instead of clicking in the cloud console: Clicking in the console creates infrastructure that cannot be reproduced exactly. Terraform creates identical clusters across dev, test, and prod from the same code. A new developer can clone the repo and recreate the entire platform from scratch.

Cloud selection decision

Decision: Start with GKE for the hub cluster.
Rejected: Starting with EKS for the hub cluster.
Why rejected: GKE Autopilot handles node management automatically, reducing operational burden for the management cluster. EKS requires explicit node pool configuration and Karpenter setup.
Tradeoff accepted: GKE vendor lock-in for the hub cluster. Spoke clusters can still use any cloud — EKS, AKS, OpenShift, or another GKE project.

4. Bootstrap sequence — exact commands

Note: Run these steps in order. Each step depends on the previous one completing successfully. Steps 3–5 repeat for each spoke (dev, test, prod).

Step 1 — Create hub cluster (GKE)

cd infra/clusters/terraform/hub
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars: set project_id, region
terraform init
terraform apply
# Output: hub-kubeconfig.yaml

Step 1 alternative — EKS hub

cd infra/clusters/terraform/hub
cp terraform.tfvars.eks.example terraform.tfvars
# Edit: set aws_account_id, region
terraform init
terraform apply

Step 2 — Install platform tools on hub

export KUBECONFIG=hub-kubeconfig.yaml
bash infra/clusters/platform/install.sh --env hub
# Installs in order:
#   cert-manager → ingress-nginx → Kyverno → ArgoCD → ESO → Prometheus
# Takes ~8 minutes
Why this install order: cert-manager must exist before ingress-nginx requests TLS certs. Kyverno must exist before ArgoCD, because ArgoCD deploys workloads that Kyverno validates. ESO must exist before ArgoCD deploys services that read secrets.

Step 3 — Create dev cluster

cd infra/clusters/terraform/dev
cp terraform.tfvars.gke.example terraform.tfvars
terraform apply
# Output: dev-kubeconfig.yaml

Step 4 — Register dev cluster with hub ArgoCD

bash infra/clusters/platform/register-spoke.sh \
  --hub-kubeconfig hub-kubeconfig.yaml \
  --spoke-kubeconfig dev-kubeconfig.yaml \
  --spoke-name dev \
  --environment dev
# ArgoCD on hub now manages dev cluster
# All 105 services deploy automatically within 5 minutes

Step 5 — Verify dev cluster

bash infra/clusters/platform/verify.sh --kubeconfig dev-kubeconfig.yaml
# Runs 10 health checks
# Expected output: all 10 pass
Repeat steps 3–5 for test and prod clusters. Use the matching terraform.tfvars example file for each environment.

5. Cloud-specific gotchas

EKS (Amazon)

IRSA (AWS identity federation for pods)
IRSA must be enabled before ESO can pull secrets from AWS Secrets Manager.
Without IRSA, the ESO pod has no AWS identity and all secret reads fail.
Karpenter
Karpenter requires its own IAM role with EC2 permissions.
Create the role before installing Karpenter, or the autoscaler will not start.
Load balancer
The AWS Load Balancer Controller is needed for AWS-native load balancers.
Alternative: use ingress-nginx with a classic NLB — simpler, less AWS-specific.
Cluster creation
eksctl creates the EKS cluster.
Terraform manages everything else after the cluster exists (node groups, IAM, add-ons).

GKE (Google)

GKE Autopilot
Node management is automatic — no node pool configuration needed.
Autopilot provisions and scales nodes based on pending pod resource requests.
Workload Identity
Enables pods to call GCP APIs without static service account key files.
Enable at cluster creation — it cannot be added to an existing cluster without node pool recreation.
Container registry
Artifact Registry is an optional alternative to GHCR for storing images.
GHCR works fine — no migration required unless you have a compliance reason to use GCP-only storage.

AKS (Azure)

Managed Identity
Azure's equivalent of IRSA.
Auto-configured when the cluster is created with --enable-managed-identity.
Without it, pods cannot call Azure APIs.
Secret store
Azure Key Vault Provider for Secrets Store CSI Driver is an alternative to ESO.
ESO is simpler — it uses the same interface across all clouds.
Use the CSI driver only if your org mandates it.
VMSS (Virtual Machine Scale Sets)
AKS uses VMSS for node autoscaling.
Terraform's azurerm_kubernetes_cluster manages VMSS automatically.
Do not manage VMSS directly — AKS owns that resource.

OpenShift

SCC (SecurityContextConstraints)
OpenShift's pod security system conflicts with Kyverno's non-root enforcement policy.
Fix: apply infra/clusters/openshift/scc-patch.yaml before installing Kyverno.
Without this patch, all pods are blocked on first deployment.
Routes
OpenShift uses Routes for native load balancing instead of Ingress objects.
The platform supports both — Routes work as a replacement for Ingress on OpenShift.
Pull secret
Required to access the Red Hat registry for OpenShift base images.
Download from console.redhat.com and store as a cluster secret before the first deployment.

6. Adding a new cloud later

When you want a 5th cluster on a new cloud provider:

  1. 1
    Add a Terraform module for the new cloud. infra/clusters/terraform/modules/NEW-CLOUD/
  2. 2
    Create an environment config that uses the module. infra/clusters/terraform/ENV/
  3. 3
    Create the cluster. terraform apply
  4. 4
    Register the cluster with hub ArgoCD. bash infra/clusters/platform/register-spoke.sh ...
  5. 5
    ArgoCD deploys all 105 services automatically to the new cluster. No manual kubectl apply needed. ArgoCD reads the same ApplicationSet definitions.

7. Disaster recovery

When a spoke cluster fails

Spoke cluster failure
Trigger: A dev, test, or prod cluster becomes unreachable or all nodes fail.
System: Services on that cluster are unavailable. Hub cluster is unaffected. Other spokes continue running.
Recovery: Run terraform apply in the cluster's Terraform directory. Recreates the cluster with the same config.
Then: Run register-spoke.sh to reconnect to hub. ArgoCD redeploys all services.
Recovery time: ~15 minutes.

When hub cluster fails

Hub cluster failure
Trigger: The hub cluster becomes unreachable or all hub nodes fail.
System: Existing services on all spoke clusters continue running. ArgoCD is not in the request path — user traffic is unaffected.
Impact: New deployments are paused until hub recovers. No new images can be rolled out.
Recovery: Run terraform apply in infra/clusters/terraform/hub, then run install.sh --env hub.
Recovery time: ~20 minutes.
Why hub failure doesn't take down prod: ArgoCD on the hub deploys changes to spokes, but running pods on spokes do not communicate with hub on every request. Hub failure = no new deploys. Hub failure ≠ prod outage.

8. GitHub Actions for Terraform

All Terraform operations run through CI after initial bootstrap.

No manual terraform apply from a laptop is needed after setup.

On a pull request touching infra/clusters/terraform/

  1. 1
    GitHub Actions runs terraform plan.
  2. 2
    Plan output is posted as a PR comment.
  3. 3
    Reviewer sees exactly what will change before approving the merge.

On merge to main

  1. 1
    GitHub Actions runs terraform apply.
  2. 2
    The cluster is created or updated automatically.
  3. 3
    ArgoCD detects the updated cluster registration and syncs all services.
Warning: Merging to main triggers a real terraform apply. Always review the plan comment in the PR before approving a merge that changes cluster node counts, machine types, or network config.
