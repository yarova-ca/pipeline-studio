Pipeline Schema
What this covers: The 6-phase CI/CD pipeline — every phase, every stage, trigger conditions, outputs, and why each phase boundary exists.
When to read: First doc to read. Before adding any stage or changing stage order.
Pillar 4 — Pipeline & Tools  ·  Related: 03 Stage types  ·  04 Tool catalog  ·  05 Invariants
Decision: 6 phases (not 4, not 8)
Rejected: 4 phases — too coarse. Cannot isolate a security gate failure from a build failure. A failed Phase 2 and a failed Phase 3 look the same.
Rejected: 8 phases — too granular. Adds cognitive overhead and more YAML per pipeline without adding security enforcement value.
Accepted tradeoff: 6 phases adds ~3 min to total pipeline runtime vs a 4-phase baseline. That cost is accepted because security gate isolation is non-negotiable.
If Phase 2 PR Gate fails:
Trigger: SAST or SCA scanner returns a HIGH or CRITICAL severity finding.
System: PR is blocked. No merge is allowed until the finding is resolved or suppressed with a written justification.
User sees: GitHub check status "failed" — shows scanner name (CodeQL / Trivy) and finding count.
User can: Fix the vulnerability in the branch and re-push, or add the finding to the suppression list with justification and security team sign-off.

Acronym definitions — terms used in this doc

SCA — Software Composition Analysis. Scans third-party dependencies for known CVEs.

SAST — Static Application Security Testing. Scans source code for insecure patterns without running the code.

DAST — Dynamic Application Security Testing. Runs the application and probes it for vulnerabilities over HTTP.

IaC — Infrastructure as Code. Terraform, Helm, Kubernetes manifests, CloudFormation, etc.

SBOM — Software Bill of Materials. A machine-readable list of every component in the build artifact.

SLSA — Supply-chain Levels for Software Artifacts. A provenance attestation proving the artifact was built by a trusted pipeline from a known source.

OIDC — OpenID Connect. Used here for keyless CI authentication — each job gets a short-lived token, no stored credentials.

Phase summary — trigger conditions and outputs

Phase 0 — Bootstrap

Runs once when the repository is created. Not triggered by PRs or pushes.

Output: the hard constraints (branch protection, OIDC, registry policy, Kyverno) that every other phase depends on.

Phase 1 — Local Dev

Runs on the developer machine. Triggered by git commit.

Output: commit proceeds only if pre-commit hooks pass and no secret is detected.

Phase 2 — PR Gate

Runs on every push to an open PR. Triggered by: pull_request event.

Output: PR is mergeable only when all 10 stages pass and a human approves.

5 security scans (SCA, SAST, License, IaC, Secrets) run in parallel. Build waits for all 5.

Phase 3 — Main Build

Runs on merge to main. Triggered by: push to main.

Output: signed container image in registry with SBOM attestation, SLSA provenance, and semantic version tag.

5 security scans run in parallel. Build waits for all 5. Sign depends on container scan. Attestations depend on sign.

Registry

Immediately after Phase 3 completes. The registry holds the canonical artifact for all promotions.

Output: image digest with signature, SBOM attestation, and SLSA provenance all attached. No promotion starts until all three are present.

Promotions

Same 5 stages across DEV, TEST, and STAGING. PROD replaces stages 4–5 with Argo Rollouts and Monitor.

Trigger: GitOps PR merged into the target environment overlay. ArgoCD or Flux reconciles.

Gate between each environment: explicit approval required. Test pass alone does not open the gate.

Phase 0 — Bootstrap

Runs once, not per PR. Sets the constraints every other phase depends on.
Why once, not per PR: Bootstrap configures repo-level and cluster-level controls — branch protection, OIDC trust, admission policies. These are not per-commit artifacts. Running them per PR would add ~4 min of idempotent setup work with no security gain.
DevSecOps — security control
DevOps — infrastructure / workflow
PHASE 0 — Bootstrap (runs once, not per PR) ├── Branch protection — no direct push to main ├── CODEOWNERS — auto-assign reviewers by path ├── OIDC trust — CI talks to cloud, no stored passwords ├── Registry access — push scoped to exact repo + branch ├── Kyverno admission policy — cluster rejects unsigned images └── Dependency update automation — Renovate / Dependabot configured

Phase 1 — Local Dev

Runs on developer machine. Triggered on git commit.
Why pre-commit hooks, not CI-only: CI feedback takes 3–8 minutes round-trip (push → GitHub Actions runner → result). Pre-commit hooks return in under 2 seconds. Secret leaks caught in CI have already been pushed to the remote. Pre-commit catches them before the network hop.
DevSecOps — security control
DevOps — infrastructure / workflow
PHASE 1 — Local Dev (developer machine) ├── IDE plugins — linters, formatters, real-time hints ├── Pre-commit hooks — gitleaks, hadolint, checkov └── Local secret scan — blocks commit if secret pattern detected

Phase 2 — PR Gate

Triggers on every PR push. No merge until all 10 stages green and human approves.
Why no image push on PR: An image pushed from a PR branch has not been reviewed or approved. It cannot be signed with a main-branch OIDC identity. Pushing it would produce an unsigned, unattested artifact that looks like a production image but is not. The build step on PR produces a tarball only — discarded after scanning.

Phase 3 — Main Build

Triggers on merge to main. Produces the signed, attested, promotion-ready image.
Why image signing happens here, not in Phase 2: cosign keyless signing uses the GitHub Actions OIDC token issued to the refs/heads/main branch identity. PR branches produce a different OIDC subject (refs/pull/N/merge). Signing a PR image would produce an attestation with the wrong subject — Kyverno admission policies would reject it at deploy time anyway.
