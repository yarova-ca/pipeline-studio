libc — what it is and why it matters

libc (C library) is the standard library that links to almost every Linux binary. It provides system calls, string functions, file I/O, and memory management.

glibc

GNU C Library — the default on Debian, Ubuntu, RHEL, and most distros. Widest binary compatibility. Pre-compiled binaries targeting glibc run on glibc systems without modification.

musl

Lightweight alternative used by Alpine Linux. ~5× smaller than glibc. Strict POSIX compliance. Problem: binaries compiled against glibc do not run on musl — they segfault or produce wrong output. Affects Python native extensions, Node.js native modules, and any C/C++ binary that was compiled on glibc.

none (scratch)

No libc at all. Only statically-linked binaries work. Go binaries compiled with CGO_ENABLED=0 and Rust binaries with static linking are the main use case.

FIPS 140-2 — what it is and when it is required

FIPS 140-2 / 140-3 (Federal Information Processing Standard) is a US government standard for cryptographic modules. It certifies that the specific implementation of AES, SHA, RSA, and other algorithms meets NSA/NIST requirements.

FIPS 140-2 vs 140-3

FIPS 140-2 was the previous standard. NIST closed new FIPS 140-2 submissions in September 2026 — existing validated modules remain valid indefinitely. New certifications are now issued under FIPS 140-3 (ISO/IEC 19790). For compliance purposes, both 140-2 and 140-3 validated modules are accepted by US government auditors.

When required

US federal contracts (FedRAMP Moderate and High), DoD IL2–IL6, ITAR-controlled systems, and some financial sector contracts (FFIEC, OCC guidance) require FIPS 140-2 or 140-3 validated modules.

What "FIPS" means in this table

Yes: the vendor ships a FIPS 140-2 or 140-3 validated module and it is active by default or enabled via a supported configuration. Partial: FIPS mode is available but not yet independently certified for this exact distro release, or certification is in progress.

DISA STIG — what it is and who requires it

DISA STIG (Defense Information Systems Agency Security Technical Implementation Guide) is a configuration baseline published by the US DoD. It specifies exact settings — file permissions, service states, kernel parameters, audit rules — for hardening a system.

When required

DoD IL4 and above. FedRAMP High (STIG is one accepted hardening baseline). Some US federal civilian contracts. Financial institutions under OCC supervision in regulated environments.

What "DISA STIG available" means

Yes: an official STIG has been published by DISA for this distro and version. It is regularly updated. Partial: RHEL STIG applies to this distro, or a vendor-maintained hardening guide exists, but no official DISA STIG has been published for this specific distro.

CIS Benchmark — what it is and who uses it

CIS Benchmark (Center for Internet Security) is a set of hardening recommendations — not as strict as DISA STIG, but widely used in commercial sectors (SOC 2, ISO 27001, PCI-DSS environments).

Levels

Level 1: conservative hardening with minimal performance impact. Level 2: defense-in-depth, may affect usability. Most environments use Level 1.

Who uses it

Financial services, healthcare, technology companies under SOC 2 audits. Cloud providers (AWS, Azure, GCP) publish pre-hardened AMIs/images that pass CIS benchmarks.

OCI image — what it is

OCI (Open Container Initiative) is the standard format for container images. All major registries (Docker Hub, GHCR, ECR, GCR, ACR) store OCI images. Docker images are OCI-compatible.

Min image size in this table is the compressed OCI image size pulled from the official registry for the smallest available variant of the distro (e.g., alpine:3.20, debian:12-slim).

musl vs glibc — the Alpine breakage pattern

The most common Alpine pitfall: a library or binary compiled against glibc fails silently or crashes on Alpine (musl).

Affected cases

Python packages with C extensions (cryptography, lxml, numpy) — must rebuild from source or use manylinux wheel. Node.js native modules (bcrypt, node-canvas, sharp) — require compilation on musl. Java: works (JVM is statically linked). Go: works when CGO_ENABLED=0. Rust: works with musl target.

Detection

Run ldd /usr/local/bin/your-binary inside the Alpine container. If it outputs not a dynamic executable or no dynamic linker, the binary will fail. If it lists libc.so.6, it requires glibc and will not run on Alpine.

Multi-arch and docker buildx — ARM64 vs x86_64

All 22 distros in this catalog publish official images for both x86_64 (Intel/AMD) and ARM64 (Apple Silicon, AWS Graviton). As of 2026, this is standard practice.

x86_64 (amd64)

Intel and AMD processors. Universal default for GitHub Actions runners, GitLab CI, and most CI providers. Always available.

ARM64 (aarch64)

Apple Silicon (M1/M2/M3), AWS Graviton 2/3/4, Ampere Altra. Graviton 3 is ~40% cheaper than equivalent x86_64 EC2 and 60% more energy efficient. GitHub Actions offers ARM64 runners (larger plan required).

Multi-arch OCI manifest

A multi-arch image is a single tag (alpine:3.21) that resolves to the correct architecture automatically. docker pull alpine:3.21 on ARM64 gets the ARM64 layer; on x86_64 gets the x86_64 layer.

Building multi-arch images

Use docker buildx build --platform linux/amd64,linux/arm64 -t your-image:tag . — requires Docker Desktop or a Docker daemon with buildx enabled. CI providers: GitHub Actions supports this natively; add platforms: linux/amd64,linux/arm64 to your workflow.

When ARM64 matters for base image selection

If your CI runs on ARM64 (Graviton) and your Dockerfile uses a musl-based base (Alpine), test on both architectures — musl behavior is identical across architectures, but native modules compiled at build time must match the target arch.

Container-optimized base images

8 images — designed for use as the FROM line in a Dockerfile. Primary goal: minimal size and minimal attack surface.

These are not full operating systems. They are base layers for container images.
scratch and Distroless have no shell — debugging requires a debug variant or a sidecar. Choose by compliance requirement first: FIPS → UBI-micro/UBI-minimal. SBOM-native → Wolfi. Smallest image → scratch (static) or Alpine. Azure workloads → Azure Linux.
UBI (Universal Base Image) — what it is

UBI (Red Hat Universal Base Image) is a free-to-redistribute subset of RHEL. It includes RHEL's FIPS-certified crypto module and receives the same security updates as RHEL.

Three tiers: micro (no package manager, ~27MB), minimal (microdnf, ~37MB), standard (dnf, ~220MB). All three share the same FIPS-certified crypto module.

UBI does not require a Red Hat subscription to pull or run. A subscription is required to install additional RHEL packages.

Distroless — what it is

Distroless images (gcr.io/distroless) are built by Google. They contain only the application runtime — no shell, no package manager, no coreutils. Built on Debian.

Available variants: static (~2MB, no libc), base (~20MB, glibc), java17, python3, nodejs. The static variant is for Go/Rust. The language variants include the runtime only.

Wolfi / Chainguard — what it is

Wolfi is a Linux distribution designed specifically for container security. Built by Chainguard. Uses apk (same format as Alpine) but targets glibc instead of musl — solving the Alpine binary-compat problem while keeping image sizes small.

Every package in Wolfi ships with an SBOM (Software Bill of Materials) and SLSA Level 3 provenance. This means every dependency is traceable to its source and build process. Required for SSDF and EO 14028 supply chain compliance.

Free tier: wolfi-base available publicly. FIPS tier: paid Chainguard Images subscription required.

CA certificates — what breaks without them

Images with no CA certificate bundle

scratch and gcr.io/distroless/static-debian12 ship with no CA cert bundle. The CA certs column shows "No" for these two.

What fails without CA certs

Any app making outbound HTTPS calls: API clients, npm install, pip install, cargo registry calls, apt-get over HTTPS. The error appears as connection refused, SSL_ERROR_HANDSHAKE_FAILURE, or a generic certificate error — not a clear "no certs" message.

Fix for scratch

Copy certs from a build stage in a multi-stage Dockerfile: COPY --from=alpine /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/. Add this immediately after FROM scratch.

Fix for distroless/static-debian12

Switch to gcr.io/distroless/base-debian12. The base variant adds ~18MB but includes CA certs, glibc, and libssl. Use base if your app makes HTTPS calls or uses glibc networking.

Alpine is fine

Alpine ships ca-certificates by default. Alpine CI images and Alpine runtime images can make HTTPS calls without any extra steps.

General purpose

6 distros — full OS with package managers, shells, and broad ecosystem support. Used as container bases, CI runners, and VMs.

Ubuntu 24.04 LTS is the default GitHub Actions runner OS as of 2026 (ubuntu-latest moved from 22.04 to 24.04 in January 2025).
Debian slim is the most commonly used container base for glibc-dependent runtimes (Node.js, Python, Ruby, PHP) when image size matters and FIPS is not required. Amazon Linux 2023 is optimal only for AWS-native workloads. Fedora's 13-month lifecycle makes it unsuitable as a container base or production server OS.
Ubuntu FIPS — what "Ubuntu Pro" means

Ubuntu Pro (formerly Ubuntu Advantage) is Canonical's paid subscription. It unlocks FIPS 140-3 validated modules (140-2 for 22.04, 140-3 for 24.04), DISA STIG tooling, and Expanded Security Maintenance (ESM) beyond the 5-year standard LTS window.

Cost: free for up to 5 personal machines. Enterprise pricing applies per-node. AWS/Azure/GCP marketplace instances can subscribe per-hour.

Without Ubuntu Pro: Ubuntu 22.04 LTS has no FIPS certification. The standard OpenSSL build in Ubuntu is not FIPS-validated.

Amazon Linux 2023 — EOL and AWS lock-in

Amazon Linux 2023 (AL2023) replaces Amazon Linux 2 (EOL 2025). AL2023 uses dnf (same as Fedora/RHEL), not yum.

AL2023 packages lag RHEL 9 by one release cycle — packages are pulled from Fedora after they ship in RHEL, giving AL2023 newer packages than RHEL but with less enterprise testing.

AL2023 is optimized for EC2 and Lambda container images. Kernel is patched for AWS Nitro hypervisor. Not recommended for on-premises or non-AWS environments — packages and kernel patches are AWS-specific.

RHEL family — enterprise

5 distros — all binary-compatible with RHEL 9. Differ by vendor, FIPS certification status, and cost.

Only RHEL 9 and Oracle Linux 9 have independently certified FIPS 140-2 modules as of 2026.
Rocky Linux 9 and AlmaLinux 9 use the same crypto code as RHEL 9 but do not hold their own FIPS certificate — regulators who require the certificate number must use RHEL or Oracle Linux. CentOS Stream 9 is the upstream of RHEL 9 — it receives changes before RHEL QA finishes. Use it for staging and dev, not production.
RHEL binary compatibility — what it means

All 5 distros in this section are binary-compatible with RHEL 9. A binary compiled on RHEL 9 runs on Rocky 9, Alma 9, Oracle 9, and CentOS Stream 9 without recompilation.

Package managers (dnf) use the same RPM format. Package names and paths are identical. Configuration files are in the same locations.

Not a guarantee: third-party software vendors certify on RHEL. They do not independently certify on Rocky, Alma, or Oracle Linux. A vendor's RHEL certification does not transfer to these clones.

CentOS Stream 9 — the upstream shift explained

CentOS Linux 8 was discontinued in December 2021. CentOS Stream 9 replaced it — but CentOS Stream is the upstream of RHEL, not a downstream clone.

This means: CentOS Stream 9 receives changes before RHEL 9 QA has finished validating them. It is a rolling release. It is not production-safe for regulated environments.

Use case: staging or CI runner where RHEL-compatible behavior is needed without a subscription, and where receiving pre-release RHEL patches is acceptable.

Immutable / Kubernetes-native node OS

3 distros — designed as Kubernetes node operating systems. Not container base images. Not general-purpose servers.

These are not used as Dockerfile FROM images. They are the OS running on the Kubernetes worker node itself.
Talos Linux has no shell and no SSH — the most locked-down option. All management is via API. Choose by cloud: EKS → Bottlerocket. GKE → Google COS (Section 1). AKS → Azure Linux or Flatcar. Self-managed → Flatcar or Talos.
Immutable OS — what it means for a K8s node

An immutable OS has a read-only root filesystem. You cannot install packages or modify configuration files on a running node. Updates happen atomically — the entire OS is swapped to a new image, then the node reboots.

Benefits: no configuration drift, no manual package installs that bypass audit, automatic rollback if the new image fails to boot.

Tradeoff: if you need to install a kernel module, a debug tool, or a monitoring agent that runs on the host OS, you cannot use a standard package install. It must be a DaemonSet (running as a container) or the image must be rebuilt.

Talos Linux — no shell, no SSH explained

Talos Linux has no shell and no SSH daemon. The only management interface is the Talos API (gRPC). talosctl is the only way to interact with the node.

This eliminates the entire class of attacks that require shell access: no kubectl exec into the node, no SSH brute force, no shell injection via misconfigured services.

Consequence: if a process is broken and you need to debug it interactively, you cannot. You must collect logs via the API or rebuild the cluster. Accepted tradeoff in high-security environments.
