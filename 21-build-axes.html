1. How build axes work

Every service Dockerfile declares 8 ARG directives near the top of the file.

Each ARG has a default value — a plain docker build . uses all defaults.

Passing --build-arg AXIS=VALUE overrides that axis only. All other axes keep their defaults.

Note: Axes are independent. Combine any values. COMPLIANCE=hipaa OBSERVABILITY=prometheus AUTH=jwt is valid. No axis constrains another, with one exception: COMPLIANCE=fips requires RUNTIME=fips — the build guard enforces this and exits with an error if they mismatch.

What happens during a build

  1. 1 Docker resolves all 8 ARG values — either from --build-arg flags or from the defaults declared in the Dockerfile. This happens before any COPY or RUN instruction executes.
  2. 2 The builder stage selects the base image using BUILD_IMAGE and RUNTIME to pick the correct tagged image from the base image registry. Example: BUILD_IMAGE=node RUNTIME=alpine → node:22-alpine as the build base.
  3. 3 Variant-specific source files are COPY'd into src/<axis>/active/ for each axis. The correct variant directory replaces the active/ placeholder. Only one variant per axis exists in the final image.
  4. 4 Dependencies are installed using the package manager selected by PKG_MGR. The Dockerfile uses the ARG value to invoke the correct install command: npm ci, pnpm install --frozen-lockfile, poetry install, etc.
  5. 5 The build tool compiles the source using the tool selected by BUILD_TOOL (Node backends only). Python, Go, and others use their own fixed compile or bundle steps and do not use this axis.
  6. 6 The final runtime image is assembled — only compiled output and runtime dependencies are copied from the builder stage. Dev dependencies, build tools, and source maps are excluded. The runtime image is lean.
Why conditional COPY instead of runtime branching: The variant code is selected once at build time, not at runtime. The compiled image contains exactly one compliance middleware, one auth handler, one ORM wrapper. There is no if (process.env.COMPLIANCE === 'hipaa') check at runtime. Zero runtime overhead. Zero chance of loading the wrong variant under load.

ARG BUILD_IMAGE=node
ARG RUNTIME=alpine
ARG PKG_MGR=npm
ARG BUILD_TOOL=tsc
ARG COMPLIANCE=standard
ARG OBSERVABILITY=none
ARG AUTH=all
ARG ORM=prisma

# ...builder stage selects base image...
FROM ${BUILD_IMAGE}:22-${RUNTIME} AS builder

# ...variant COPY — runs once at build time...

Warning: The active/ directory is not committed to the repo. It is a build-time artifact only. Never import from src/compliance/active/ in your own application code — import from the axis-specific path in development, or rely on the compiled output in production.

3. Each axis — full detail

BUILD_IMAGE Language / runtime family default: node Applies to: all services

Selects the base Docker image family used in both the builder stage and the runtime stage. The value maps directly to the Docker Hub official image name.

Combined with RUNTIME to form the full image tag. Example: BUILD_IMAGE=node RUNTIME=alpine → node:22-alpine.

Value Builder base Runtime base Version pinned Notes
node node:22-alpine node:22-alpine 22 LTS Default. Used for all TypeScript/Express/Fastify backends.
python python:3.12-slim python:3.12-slim 3.12 Used for FastAPI, Django, Flask services.
golang golang:1.22-alpine gcr.io/distroless/static-debian12 1.22 Builder stage compiles binary. Runtime is distroless — no shell, minimal attack surface.
java eclipse-temurin:21-jdk-alpine eclipse-temurin:21-jre-alpine 21 LTS Builds JAR in builder, runs JRE-only in runtime.
dotnet mcr.microsoft.com/dotnet/sdk:8.0-alpine mcr.microsoft.com/dotnet/aspnet:8.0-alpine 8.0 LTS Publishes self-contained binary. Runtime is ASP.NET Core image.
rust rust:1.77-alpine alpine:3.19 1.77 stable Compiles release binary. Runtime is bare Alpine — binary is statically linked.
elixir elixir:1.16-alpine alpine:3.19 1.16 Builds Mix release. Runtime runs the release tarball on bare Alpine.
ruby ruby:3.3-alpine ruby:3.3-alpine 3.3 Used for Rails API services. Puma runs in runtime stage.
php php:8.3-fpm-alpine php:8.3-fpm-alpine 8.3 PHP-FPM. Compose installs vendor in builder. Runtime runs FPM.
Gotcha: When BUILD_IMAGE=golang, the runtime stage is distroless — there is no shell. docker exec and kubectl exec will fail. Use --build-arg RUNTIME=slim only during development to get a shell for debugging.
RUNTIME OS variant / compliance mode default: alpine Applies to: all services

Selects the OS variant appended to the base image tag. Also gates FIPS compliance — RUNTIME=fips is required when COMPLIANCE=fips.

Value OS Image size Shell available FIPS OpenSSL Use when
alpine Alpine Linux 3.19 Smallest (~5MB base) Yes (sh) No Default for all services. Smallest attack surface with shell access.
slim Debian 12 Slim Medium (~30MB base) Yes (bash) No When a package requires glibc (native modules, some ML libs). Alpine uses musl.
fips Ubuntu 22.04 FIPS Largest (~75MB base) Yes (bash) Yes — OpenSSL FIPS module active Required when COMPLIANCE=fips. US federal workloads. NIST 140-2 cryptography.
If COMPLIANCE=fips but RUNTIME≠fips
Trigger: Build runs with --build-arg COMPLIANCE=fips but RUNTIME is alpine or slim.
System: The Dockerfile build guard runs a RUN check: if COMPLIANCE=fips and RUNTIME!=fips, it exits with code 1 and message: ERROR: COMPLIANCE=fips requires RUNTIME=fips. Rebuild with --build-arg RUNTIME=fips
User sees: Docker build fails with non-zero exit code. The error message is printed to stdout.
User can: Add --build-arg RUNTIME=fips to the build command and retry.
PKG_MGR Package manager default: npm (Node) / pip (Python) Applies to: Node, Python services

Selects which package manager installs dependencies in the builder stage. The Dockerfile uses the value to call the correct install command and lockfile.

Go, Rust, Java, and .NET use their own fixed package managers (go mod, cargo, gradle, dotnet restore). This axis has no effect on those languages.

Node — valid values

Value Install command Lockfile used Notes
npm npm ci package-lock.json Default. ci enforces lockfile — no version drift in builds.
pnpm pnpm install --frozen-lockfile pnpm-lock.yaml Faster installs, hard-linked node_modules. Requires corepack enable in Dockerfile.
yarn yarn install --frozen-lockfile yarn.lock Yarn Classic (v1). Yarn Berry (v2+) not supported — PnP mode breaks the build.
bun bun install --frozen-lockfile bun.lockb Fastest installs. Use only if production runtime also uses Bun — bun:latest image required.

Python — valid values

Value Install command Lockfile used Notes
pip pip install --no-cache-dir -r requirements.txt requirements.txt Default for Python. Simple and universal.
poetry poetry install --no-dev --no-interaction poetry.lock Strict dependency resolution. Requires pyproject.toml present.
uv uv pip install -r requirements.txt requirements.txt Rust-based pip replacement. 10-100x faster installs. Uses same requirements.txt format.
Warning: The lockfile must match the package manager. Building with PKG_MGR=pnpm when only package-lock.json exists (no pnpm-lock.yaml) fails with a lockfile not found error. Generate the correct lockfile locally before changing this axis.
BUILD_TOOL TypeScript compiler / bundler default: tsc Applies to: Node backends only

Selects the TypeScript compilation or bundling tool for Node backend services. Has no effect on Python, Go, Java, .NET, Rust, Elixir, Ruby, or PHP services.

Value Tool Build command Output Build time Source maps Use when
tsc TypeScript compiler tsc -p tsconfig.json dist/ — CommonJS .js files Slowest (~10-40s) Yes (configurable) Default. Best type checking. Required for libraries that export types.
esbuild esbuild esbuild src/index.ts --bundle --platform=node --outfile=dist/index.js dist/index.js — single bundled file Fast (<1s) Yes (inline) When build speed matters. Lambda deployments. Does not type-check.

4. COMPLIANCE axis — what each value adds

Default: standard — no extra middleware. Each non-standard value adds middleware that runs on every HTTP request.

How compliance middleware is wired: Each value copies its source into src/compliance/active/ at build time. The main application entry point imports from src/compliance/active/index and calls applyCompliance(app) (Node) or apply_compliance(app) (Python) once during startup. The middleware then runs on every request automatically.
standard No extra middleware

No compliance middleware is added. Commercial baseline — no regulated-industry requirements.

applyCompliance(app) is a no-op. Zero overhead. Zero extra log fields.

Use when: SaaS, internal tools, dev environments, any service without regulated-industry data.

hipaa HIPAA — Healthcare data

Applies to: Any service handling Protected Health Information (PHI).

Middleware runs on every HTTP request:

1 Audit logging: Every request logs actor (from JWT sub claim or API key ID), action (HTTP method + path), resource (extracted from path), request-id (UUID generated per request). Log destination is stdout — append-only. Log stream must be collected by an immutable log aggregator (Loki, CloudWatch, Splunk) to satisfy HIPAA §164.312(b).
2 PHI masking in logs: The following field names are redacted to [REDACTED] in all log output: ssn, dob, mrn, diagnosis, patientName, address, phone. Masking is applied to request body, query params, and response body before logging. Field name matching is case-insensitive.
3 Verbose error suppression: Node: app.set('env', 'production') is called — Express suppresses stack traces in error responses. Python: FastAPI debug=False — detail field in HTTPException responses is suppressed for 5xx errors.
If audit log write fails
Trigger: stdout write fails (disk full, container log driver error).
System: HIPAA middleware catches the write error and returns HTTP 500 to the caller. The request is not processed. No data is written without an audit trail.
User sees: HTTP 500 — {"error":"Audit logging unavailable. Request rejected."}
User can: Retry. Operator must resolve the log collection issue before requests are accepted.
pci PCI DSS — Payment card data

Applies to: Any service in the cardholder data environment (CDE) or that transmits/stores card data.

Middleware runs on every HTTP request:

1 Startup secret strength check: Runs once at process start (not per-request). Iterates all environment variables. Any variable whose name matches SECRET, KEY, TOKEN, or PASS (case-insensitive) with a value shorter than 16 characters causes a fatal error and process exit. PCI DSS Req 8.3.6 requires minimum 12-character secrets; 16 is the enforced floor.
2 TLS enforcement: Any request where X-Forwarded-Proto header equals http (case-insensitive) is rejected with HTTP 403 Forbidden. Response body: {"error":"HTTPS required (PCI DSS Req 4.2.1)"}. Requests without the header are passed through — assumes TLS terminated upstream with header forwarded.
3 Debug endpoint blocking: Any request whose path starts with /debug returns HTTP 404 Not Found. PCI DSS Req 6.3.3 prohibits debug or test interfaces in production cardholder environments.
4 PCI response headers added to every response: X-PCI-DSS: enforced, Strict-Transport-Security: max-age=63072000; includeSubDomains; preload, X-Frame-Options: DENY, X-Content-Type-Options: nosniff.
Warning: The startup secret check runs before the HTTP server starts accepting connections. A misconfigured secret causes the container to crash-loop immediately. Check all environment variable values before deploying with COMPLIANCE=pci.
pipeda PIPEDA — Canadian personal information

Applies to: Any service collecting or processing personal information about Canadian residents.

Middleware runs on every HTTP request:

1 Consent enforcement: Any POST, PUT, or PATCH request that does not include the header X-Consent: granted is rejected with HTTP 451 Unavailable For Legal Reasons. Response body: {"error":"Consent required under PIPEDA Schedule 1 Principle 3"}. GET and DELETE requests pass through without consent header check.
2 Data residency header: Every response includes X-Data-Residency: CA. This header signals to API consumers and auditors that data is processed in Canada. It does not enforce residency — infrastructure deployment location does. The header serves as an audit signal.
3 PII masking in logs: The following field names are redacted to [REDACTED] in all log output: email, name, phone, address, sin (Social Insurance Number). Applied to request body, query params, and response body before logging.
Note: PIPEDA consent must be collected by your application UI before the API call is made. The middleware validates that consent was obtained and signaled — it does not collect consent itself.
fips FIPS 140-2 — US federal cryptography

Applies to: US federal agency workloads. Any service subject to FISMA, FedRAMP, or DoD requirements.

Requires: RUNTIME=fips — the build guard blocks the build if this is not set.

Middleware runs at startup only (not per-request):

1 OpenSSL FIPS mode check: On startup, the middleware calls the OpenSSL FIPS mode query. Node: crypto.getFips() — returns 1 when FIPS mode is active. Python: ssl.OPENSSL_VERSION is checked for fips substring. Go: go-fips build tag enforces FIPS at compile time; runtime check confirms module is active.
2 When FIPS mode is not active: The middleware throws a fatal error and the process exits with code 1. Node error message: FATAL: OpenSSL FIPS mode not active. Verify RUNTIME=fips image is deployed and OPENSSL_FIPS=1 is set. The HTTP server never starts. The container crash-loops. Kubernetes marks the pod as CrashLoopBackOff.
If FIPS mode check fails
Trigger: Container starts but RUNTIME=fips image was not used, or FIPS kernel module is not loaded.
System: Process exits with code 1 before HTTP server binds. Container restarts. Kubernetes marks pod CrashLoopBackOff after 3 restarts.
User sees: Service unavailable. Pod status shows CrashLoopBackOff.
User can: Check pod logs for FATAL message. Confirm RUNTIME=fips image was used. Confirm OPENSSL_FIPS=1 environment variable is set.
soc2 SOC 2 — Service organization controls

Applies to: SaaS services undergoing SOC 2 Type I or Type II audit. B2B services where customers require SOC 2 report.

Middleware runs on every HTTP request:

1 Immutable audit logging: Every request emits a structured log event with event type soc2_audit and fields: actor (user ID from auth context), action (HTTP method + normalized path), resource (entity type extracted from path), timestamp (ISO 8601 UTC), request_id (UUID v4). The log stream is stdout — must be collected by an append-only log aggregator for immutability.
2 Verbose error suppression: Same as HIPAA. Stack traces and internal error details are suppressed in HTTP responses. SOC 2 CC6.1 requires limiting disclosure of system information.
cmmc CMMC — Defense supply chain

Applies to: US DoD contractors. Any service handling Controlled Unclassified Information (CUI) or defense supply chain data.

Middleware runs on every HTTP request:

1 CMMC audit logging: Every request emits a structured log event with event type cmmc_audit and fields: actor, action, resource, timestamp. CMMC Practice AU.L2-3.3.1 requires auditable events for all system access.
2 Debug endpoint blocking: Any path starting with /debug returns HTTP 404. CMMC Practice CM.L2-3.4.6 requires minimal functionality — no diagnostic interfaces in production.
3 Egress isolation flag: Sets NO_EGRESS=true in the process environment. Application code can read this flag to disable outbound network calls to non-approved destinations. CMMC Practice SC.L2-3.13.5 requires boundary protection. Note: the flag alone does not block egress — network policy in the cluster enforces the actual boundary.
nerc NERC CIP — Energy grid / OT-IT boundary

Applies to: Electric utilities, energy grid operators, bulk electric system owners. Any service crossing the OT-IT boundary.

Middleware runs on every HTTP request:

1 OT-IT boundary headers: Adds X-OT-IT-Boundary: enforced and X-Network-Isolation: on to every response. NERC CIP-005 requires electronic security perimeter documentation. These headers serve as audit trail markers for intermediate security controls.
2 Egress isolation flag: Sets NO_EGRESS=true in the process environment. Same mechanism as CMMC. NERC CIP-007 requires limiting open network access from BES cyber systems.
Note: NERC CIP compliance requires far more than these middleware controls — physical security, personnel training, supply chain risk management. This middleware covers only the software-layer network boundary signals. Full NERC CIP compliance requires a complete CIP program.

5. OBSERVABILITY axis — what each value instruments

Default: none — zero instrumentation, zero overhead. Each value adds instrumentation that initializes at service startup.

none No instrumentation

No observability code is included in the image.

initObservability() is a no-op. Zero imported packages. Zero startup overhead. Zero memory overhead.

Use when: Development, test environments, or services where observability is handled at the infrastructure layer (service mesh, eBPF).

otel OpenTelemetry — vendor-neutral distributed tracing

What it adds: Distributed tracing via the OpenTelemetry SDK. Traces are exported to the OTLP endpoint.

Required environment variable: OTEL_EXPORTER_OTLP_ENDPOINT=http://otel-collector:4318/v1/traces

Node implementation

// src/observability/otel/index.ts (Node)
import { NodeSDK } from '@opentelemetry/sdk-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';

export function initObservability() {
  const sdk = new NodeSDK({
    traceExporter: new OTLPTraceExporter({
      url: process.env.OTEL_EXPORTER_OTLP_ENDPOINT,
    }),
    instrumentations: [getNodeAutoInstrumentations()],
  });
  sdk.start();
}

getNodeAutoInstrumentations() auto-instruments: Express/Fastify HTTP, http/https modules, DNS, pg, mysql2, redis, gRPC.

Python implementation

# src/observability/otel/__init__.py (Python)
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor

def init_observability(app):
    provider = TracerProvider()
    exporter = OTLPSpanExporter(
        endpoint=os.environ['OTEL_EXPORTER_OTLP_ENDPOINT']
    )
    provider.add_span_processor(BatchSpanProcessor(exporter))
    trace.set_tracer_provider(provider)
    FastAPIInstrumentor.instrument_app(app)

FastAPIInstrumentor auto-instruments every HTTP request — creates a span per request with route, method, and status code.

Go implementation

// src/observability/otel/otel.go (Go)
func InitObservability(ctx context.Context) func(context.Context) error {
    exporter, _ := otlptracehttp.New(ctx,
        otlptracehttp.WithEndpoint(os.Getenv("OTEL_EXPORTER_OTLP_ENDPOINT")),
    )
    tp := sdktrace.NewTracerProvider(
        sdktrace.WithBatcher(exporter),
        sdktrace.WithResource(resource.NewWithAttributes(
            semconv.SchemaURL,
            semconv.ServiceName(os.Getenv("SERVICE_NAME")),
        )),
    )
    otel.SetTracerProvider(tp)
    return tp.Shutdown
}
If OTLP endpoint is unreachable
Trigger: OTEL_EXPORTER_OTLP_ENDPOINT is not set, or the otel-collector pod is down.
System: The OTLP exporter buffers spans in memory. After the buffer fills (default 2048 spans), new spans are dropped silently. The service continues to operate normally.
User sees: No user-visible impact. Traces are missing from the collector.
User can: Check otel-collector pod health. Restart the collector to resume trace export.
prometheus Prometheus — metrics scrape endpoint

What it adds: A /metrics endpoint that Prometheus scrapes. Exposes default system metrics plus application-prefix custom metrics.

Node implementation

// src/observability/prometheus/index.ts (Node)
import client from 'prom-client';

export function initObservability(app: Express) {
  client.collectDefaultMetrics({ prefix: 'app_' });

  app.get('/metrics', async (req, res) => {
    res.set('Content-Type', client.register.contentType);
    res.end(await client.register.metrics());
  });
}

collectDefaultMetrics exposes: process CPU, memory, event loop lag, garbage collection, active handles, file descriptors. All prefixed app_.

Python implementation

# src/observability/prometheus/__init__.py (Python)

datadog Datadog — APM and distributed tracing

What it adds: Datadog APM tracing via the Datadog tracer. Traces sent to the Datadog Agent.

Required environment variables:

  • DD_SERVICE — service name shown in Datadog APM
  • DD_ENV — environment tag (e.g., production, staging)
  • DD_AGENT_HOST — hostname of the Datadog Agent (default: localhost)

Node implementation

// src/observability/datadog/index.ts (Node)
// Must be the FIRST import in the entry point — before any other require
import tracer from 'dd-trace';

export function initObservability() {
  tracer.init({
    service: process.env.DD_SERVICE,
    env: process.env.DD_ENV,
    logInjection: true,
  });
}

logInjection: true adds dd.trace_id and dd.span_id to every log line — enables log correlation in Datadog Log Management.

Python implementation

# src/observability/datadog/__init__.py (Python)
import ddtrace

def init_observability(app):
    ddtrace.patch_all()
    # ddtrace.patch_all() auto-instruments:
    # requests, aiohttp, fastapi, sqlalchemy, redis, celery
If Datadog Agent is unreachable
Trigger: DD_AGENT_HOST is wrong, or the Datadog Agent daemonset pod is down on the node.
System: dd-trace buffers spans in memory silently. No error is thrown. No crash. When the buffer fills, spans are dropped silently.
User sees: No user-visible impact. APM traces stop appearing in Datadog.
User can: Check DD_AGENT_HOST value. Verify Datadog Agent daemonset pod is running on the same node as the service pod.
Warning: dd-trace must initialize before any other module is imported. If initObservability() is called after Express/Fastify imports, some instrumentation patches miss. The application entry point must import the observability module first.

6. AUTH axis — what each value includes

Default: all — tries JWT first, then API key. Each value controls how the requireAuth middleware validates incoming requests.

Shared contract: All auth values expose a requireAuth middleware function with the same signature. When auth succeeds, it sets req.user (Node) or request.state.user (Python/FastAPI) with fields: id, email, name, role. When auth fails, it returns HTTP 401.
none No authentication — all requests pass

requireAuth calls next() immediately. No headers are read. No tokens are validated.

req.user is set to null. Application code must not depend on a user object being present.

Use when: Internal microservice-to-microservice calls where the gateway handles auth. Health check endpoints. Development-only services.

Warning: Do not deploy AUTH=none services on a public-facing port. Any caller can reach any endpoint with no credential check.
jwt JWT — JSON Web Token validation

Required environment variable: JWT_SECRET — the HMAC signing secret for HS256. Minimum 32 characters. Must match the secret used when issuing tokens.

Request flow

  1. 1 Middleware reads Authorization header. Expects Bearer <token> format. When header is absent or does not start with Bearer : returns HTTP 401 {"error":"Authorization header required"}.
  2. 2 Calls jwt.verify(token, process.env.JWT_SECRET) (Node) or jwt.decode(token, JWT_SECRET, algorithms=['HS256']) (Python). When token signature is invalid or expired: returns HTTP 401 {"error":"Invalid or expired token"}.
  3. 3 Sets req.user = { id: payload.sub, email: payload.email, name: payload.name, role: payload.role }. Calls next(). Request proceeds to the route handler.
Condition HTTP status Response body
Valid token Pass through req.user populated — route handler executes
Missing Authorization header 401 {"error":"Authorization header required"}
Malformed header (not Bearer format) 401 {"error":"Bearer token required"}
Invalid signature 401 {"error":"Invalid or expired token"}
Expired token (exp claim past) 401 {"error":"Invalid or expired token"}
oauth2 OAuth 2.0 — Bearer access token with issuer validation

Required environment variable: OAUTH_ISSUER — the expected issuer claim in the access token (e.g., https://auth.example.com).

The OAuth2 middleware validates the Bearer token's iss claim against OAUTH_ISSUER. It does not validate the signature cryptographically — it trusts the token was issued by the configured issuer and validates the claim string match only.

For full JWKS-based signature validation (recommended for production), extend the variant to use jwks-rsa (Node) or python-jose (Python) to fetch and cache the public key from the issuer's JWKS endpoint.

When validation passes: sets req.user with the same structure as the JWT variant — id from sub claim, email, name, role.

Condition HTTP status Response body
Valid token, issuer matches Pass through req.user populated
Missing Bearer token 401 {"error":"Bearer token required"}
Issuer mismatch 401 {"error":"Token issuer invalid"}
Expired token 401 {"error":"Token expired"}
apikey API Key — database-validated key

Reads: X-API-Key header.

Looks up the key value in the database table api_keys. The table must have columns: key (string, indexed), user_id, email, name, role, active (boolean), expires_at (nullable timestamp).

Condition HTTP status Response body
Key found, active=true, not expired Pass through req.user populated from DB row
Missing X-API-Key header 401 {"error":"X-API-Key header required"}
Key not found in database 401 {"error":"Invalid API key"}
Key found but active=false 401 {"error":"API key disabled"}
Key found but expires_at is past 401 {"error":"API key expired"}
Database unreachable during lookup 503 {"error":"Auth service temporarily unavailable"}
all All methods — JWT first, then API key (default)

Default value. Tries JWT validation first. When JWT fails, tries API key lookup. When both fail, returns 401.

This value supports both human users (who carry JWTs from login) and service accounts (which carry API keys for machine-to-machine calls) without requiring two separate deployments.

Decision flow

Request arrives
    ↓
Has Authorization: Bearer header?
    ├── Yes → Try JWT.verify()
                ├── Valid → set req.user, call next()
                └── Invalid → Try X-API-Key header
    └── No → Try X-API-Key header
                   ├── Key found + valid → set req.user, call next()
                   └── No key or invalid → 401 Unauthorized

7. ORM axis — what each value includes

Selects the database access layer. Each value provides a consistent interface with three operations: ping() (health check), findUserByApiKey(key) (used by AUTH=apikey), and disconnect() (shutdown).

Why a consistent interface: The auth middleware, health check endpoint, and graceful shutdown handler all use the ORM wrapper without knowing which ORM is active. Swapping ORM values does not require changing any application code outside src/db/active/.

Node ORM options

prisma Prisma — default for Node 
// src/db/prisma/index.ts
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient();

export async function ping(): Promise<void> {
  await prisma.$queryRaw`SELECT 1`;
}

export async function findUserByApiKey(key: string) {
  return prisma.apiKey.findUnique({
    where: { key, active: true },
    include: { user: true },
  });
}

export async function disconnect(): Promise<void> {
  await prisma.$disconnect();
}

Requires prisma/schema.prisma in the project root. Run npx prisma generate to generate the client before building. The generated client is included in the build output.

typeorm TypeORM — decorator-based ORM 
// src/db/typeorm/index.ts
import { DataSource } from 'typeorm';
import { ApiKey } from '../entities/ApiKey';

const dataSource = new DataSource({
  type: 'postgres',
  url: process.env.DATABASE_URL,
  entities: [ApiKey],
  synchronize: false, // never true in production
});

export async function ping(): Promise<void> {
  await dataSource.query('SELECT 1');
}

export async function findUserByApiKey(key: string) {
  return dataSource.getRepository(ApiKey).findOne({
    where: { key, active: true },
    relations: ['user'],
  });
}

export async function disconnect(): Promise<void> {
  await dataSource.destroy();
}

Requires entity classes decorated with @Entity(). synchronize: false is enforced — do not set to true in any environment that has real data.

drizzle Drizzle ORM — type-safe SQL builder 
// src/db/drizzle/index.ts
import { drizzle } from 'drizzle-orm/node-postgres';
import { Pool } from 'pg';
import { sql } from 'drizzle-orm';
import { apiKeys } from '../schema';

const pool = new Pool({ connectionString: process.env.DATABASE_URL });
const db = drizzle(pool);

export async function ping(): Promise<void> {
  await db.execute(sql`SELECT 1`);
}

export async function findUserByApiKey(key: string) {
  const result = await db
    .select()
    .from(apiKeys)
    .where(eq(apiKeys.key, key))
    .limit(1);
  return result[0] ?? null;
}

export async function disconnect(): Promise<void> {
  await pool.end();
}

Requires schema file at src/db/schema.ts with Drizzle table definitions. No code generation step — schema is the source of truth.

none No ORM — no-op stub (Node) 
// src/db/none/index.ts
export async function ping(): Promise<void> {
  // no-op — no database configured
}

export async function findUserByApiKey(_key: string) {
  return null; // always returns null — AUTH=apikey will always fail
}

export async function disconnect(): Promise<void> {
  // no-op
}

Use when: The service does not use a database. AUTH=apikey will always reject requests when ORM=none because findUserByApiKey always returns null. Do not use ORM=none with AUTH=apikey or AUTH=all.

Python ORM options

sqlalchemy SQLAlchemy — default for Python 
# src/db/sqlalchemy/__init__.py
from sqlalchemy.ext.asyncio import create_async_engine, AsyncSession
from sqlalchemy.orm import sessionmaker
from sqlalchemy import text
import os

engine = create_async_engine(os.environ['DATABASE_URL'])
AsyncSessionLocal = sessionmaker(engine, class_=AsyncSession, expire_on_commit=False)

async def get_db():
    async with AsyncSessionLocal() as session:
        yield session

async def ping() -> None:
    async with engine.connect() as conn:
        await conn.execute(text('SELECT 1'))

async def find_user_by_api_key(key: str):
    async with AsyncSessionLocal() as session:
        result = await session.execute(
            select(ApiKey).where(ApiKey.key == key, ApiKey.active == True)
        )
        return result.scalar_one_or_none()

Async SQLAlchemy with PostgreSQL. Requires asyncpg driver (DATABASE_URL must use postgresql+asyncpg:// scheme).

tortoise Tortoise ORM — async-native Python ORM 
# src/db/tortoise/__init__.py
from tortoise import Tortoise
from tortoise.contrib.fastapi import register_tortoise
import os

async def init_db(app) -> None:
    await Tortoise.init(
        db_url=os.environ['DATABASE_URL'],

Tortoise ORM is designed for async-first Python. Simpler query API than SQLAlchemy. Requires models defined with class Model(tortoise.Model).

Go ORM options

gorm GORM — default for Go 
// src/db/gorm/gorm.go
package db

import (
    "gorm.io/driver/postgres"
    "gorm.io/gorm"
    "os"
)

var DB *gorm.DB

func Init() error {
    var err error
    DB, err = gorm.Open(postgres.Open(os.Getenv("DATABASE_URL")), &gorm.Config{})
    if err != nil {
        return err
    }
    return DB.AutoMigrate(&ApiKey{})
}

func Ping() error {
    sqlDB, err := DB.DB()
    if err != nil {
        return err
    }
    return sqlDB.Ping()
}

func FindUserByApiKey(key string) (*ApiKey, error) {
    var apiKey ApiKey
    result := DB.Where("key = ? AND active = ?", key, true).First(&apiKey)
    if result.Error != nil {
        return nil, result.Error
    }
    return &apiKey, nil
}

func Disconnect() error {
    sqlDB, _ := DB.DB()
    return sqlDB.Close()
}

AutoMigrate only adds missing columns — it does not drop or rename. Safe to run on every startup.

ent ent — graph-based entity framework for Go 
// src/db/ent/ent.go
package db

import (
    "context"
    "entgo.io/ent/dialect/sql"
    "myservice/ent"
    _ "github.com/lib/pq"
    "os"
)

var Client *ent.Client

func Init(ctx context.Context) error {
    drv, err := sql.Open("postgres", os.Getenv("DATABASE_URL"))
    if err != nil {
        return err
    }
    Client = ent.NewClient(ent.Driver(drv))
    return Client.Schema.Create(ctx)
}

func Ping() error {
    _, err := Client.ApiKey.Query().Count(context.Background())
    return err
}

func FindUserByApiKey(ctx context.Context, key string) (*ent.ApiKey, error) {
    return Client.ApiKey.
        Query().
        Where(apikey.Key(key), apikey.Active(true)).
        Only(ctx)
}

func Close() error {
    return Client.Close()
}

Requires ent schema generation: go generate ./ent/.... Run this before the Docker build or include it in the Dockerfile builder stage.

9. Combining axes — compatibility matrix

Most combinations are valid. Three constraints exist. One combination requires special handling.

Combination Valid? Constraint / note
COMPLIANCE=fips + RUNTIME=alpine No Build guard exits with fatal error. Alpine does not have FIPS-validated OpenSSL. Must use RUNTIME=fips.
COMPLIANCE=fips + RUNTIME=slim No Same as above. Slim is Debian, not FIPS-validated. Must use RUNTIME=fips.
AUTH=apikey + ORM=none Builds Build succeeds. At runtime, every API key lookup returns null. Every request fails with 401. Do not use in production.
BUILD_IMAGE=golang + RUNTIME=alpine Yes Builder uses golang:1.22-alpine. Runtime uses distroless (RUNTIME value ignored for Go runtime stage). No shell in the container.
BUILD_IMAGE=node + BUILD_TOOL=esbuild or swc Yes Valid. No type checking runs. Add tsc --noEmit to CI pipeline separately.
COMPLIANCE=hipaa + OBSERVABILITY=prometheus Yes Valid and common. HIPAA audit logs go to stdout. Prometheus metrics go to /metrics. No conflict.
COMPLIANCE=pci + OBSERVABILITY=datadog Yes — with note Valid. Ensure Datadog Agent is deployed inside the CDE network boundary. Spans contain request metadata — verify no cardholder data leaks into span attributes.
PKG_MGR=bun + BUILD_IMAGE=node Yes — with note Valid for install only. If the runtime also uses Bun (not Node), swap BUILD_IMAGE=bun. Using Bun to install but Node to run is valid but uncommon.
BUILD_IMAGE=python + BUILD_TOOL=tsc Ignored BUILD_TOOL has no effect on Python services. The axis is silently ignored. No error.
Any + COMPLIANCE=nerc or cmmc Yes Valid with all combinations. NO_EGRESS=true environment variable is set by the middleware. Cluster NetworkPolicy must independently enforce egress blocking.

11. Adding a new compliance value — extensibility guide

The Dockerfile COPY picks up new values automatically. No Dockerfile change is needed. Five steps.

  1. 1 Create the source files for the new value. mkdir -p src/compliance/<new-value> One directory per value. The Dockerfile's COPY src/compliance/${COMPLIANCE}/ instruction resolves to this path.
  2. 2 Implement the compliance function matching the existing signature.

    Node (src/compliance/<new-value>/index.ts):

    export function applyCompliance(app: Express): void { // add middleware using app.use() // add startup checks here }

    Python (src/compliance/<new-value>/__init__.py):

    def apply_compliance(app) -> None: # add FastAPI middleware using app.add_middleware() # add startup event handlers using @app.on_event("startup")

    Go (src/compliance/<new-value>/compliance.go):

    func Apply(r *chi.Mux) { r.Use(complianceMiddleware) } The function signature is the contract. Application code calls this function once at startup — it does not know which value is active.
  3. 3 Test the new value locally. docker build --build-arg COMPLIANCE=<new-value> -t test:compliance-new . Confirm the COPY succeeds, dependencies install, and the service starts. Check logs for startup events from the new middleware.
  4. 4 Update the feature matrix. Add the new value to the COMPLIANCE row in 17-feature-matrix.html. Auditors and consultants use the matrix to understand available options.
  5. 5 Document the middleware specification. Add the new value's middleware spec to 12-compliance-variations.html. Follow Template A (failure mode block) for every failure mode the middleware introduces.

Adding a new observability value

Same 5-step process. Target directory: src/observability/<new-value>/. Function signature: initObservability(app) (Node/Python) or InitObservability(ctx) (Go).

Adding a new auth value

Same 5-step process. Target directory: src/auth/<new-value>/. Must export a requireAuth middleware. Must set req.user with { id, email, name, role } on success. Must return HTTP 401 on failure.

Adding a new ORM value

Same 5-step process. Target directory: src/db/<new-value>/. Must export three functions: ping(), findUserByApiKey(key), disconnect(). All three must match the existing async signatures.

13. CI pipeline integration

How to pass build axes through a CI system. GitHub Actions example and Kubernetes deployment pattern.

GitHub Actions — parameterized build

name: Build and push service image

on:
  push:
    branches: [main]

env:
  COMPLIANCE: ${{ vars.COMPLIANCE || 'standard' }}
  OBSERVABILITY: ${{ vars.OBSERVABILITY || 'none' }}
  AUTH: ${{ vars.AUTH || 'all' }}
  ORM: ${{ vars.ORM || 'prisma' }}

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build Docker image with axes
        run: |
          docker build \
            --build-arg COMPLIANCE=${{ env.COMPLIANCE }} \
            --build-arg OBSERVABILITY=${{ env.OBSERVABILITY }} \
            --build-arg AUTH=${{ env.AUTH }} \
            --build-arg ORM=${{ env.ORM }} \
            --label "git.sha=${{ github.sha }}" \
            --label "git.ref=${{ github.ref }}" \
            -t $IMAGE_NAME:${{ github.sha }} .

      - name: Verify build axes in image labels
        run: |
          COMPLIANCE_ACTUAL=$(docker inspect $IMAGE_NAME:${{ github.sha }} \
            --format '{{ index .Config.Labels "build.axis.compliance" }}')
          echo "COMPLIANCE_ACTUAL=${COMPLIANCE_ACTUAL}"
          if [ "${COMPLIANCE_ACTUAL}" != "${{ env.COMPLIANCE }}" ]; then
            echo "ERROR: compliance axis mismatch"
            exit 1
          fi

Set COMPLIANCE, OBSERVABILITY, AUTH, and ORM as GitHub Actions Variables (repository or environment level) — not secrets. Axis values are not sensitive.

Kubernetes — separate deployments per compliance tier

Run separate images per compliance tier. Do not use a single image with a runtime env var to switch compliance modes — the variant code must be compiled in at build time.

# Build three tier-specific images from the same source
docker build --build-arg COMPLIANCE=standard -t myapi:standard .
docker build --build-arg COMPLIANCE=hipaa    -t myapi:hipaa    .
docker build --build-arg COMPLIANCE=pci      -t myapi:pci      .

# Deploy each to its own namespace
kubectl -n commercial  set image deployment/myapi myapi=myapi:standard
kubectl -n healthcare  set image deployment/myapi myapi=myapi:hipaa
kubectl -n payments    set image deployment/myapi myapi=myapi:pci
Decision: one image per compliance tier, not runtime env switching.
Rejected: single image with COMPLIANCE env var read at runtime to load the right middleware.
Why rejected: A single image would include all 8 compliance middleware variants. Any of them could be activated by changing an env var — compliance boundaries are only as strong as environment variable access controls. A misconfigured deployment could run HIPAA middleware in a standard environment or vice versa.
Tradeoff accepted: more images to build and push (~3-8x image count). CI time increases by image count × build time.

14. Failure modes — wrong axis values

What happens when an invalid or incompatible axis value is passed. Every failure mode resolved.

Failure condition When it occurs Error seen Fix
COMPLIANCE=fips RUNTIME=alpine Docker build ERROR: COMPLIANCE=fips requires RUNTIME=fips — build exits non-zero Add --build-arg RUNTIME=fips
Unknown compliance value (e.g. COMPLIANCE=gdpr) Docker build COPY step COPY failed: file not found in build context or excluded by .dockerignore: src/compliance/gdpr/ Use only listed values. Create the variant first if adding a new one.

16. Quick build command reference

Copy-paste commands for common build patterns. All use the short form (omit axes that stay at default).

Compliance-only builds (default language: Node, default obs/auth/orm)

# Standard (no compliance middleware)
docker build -t svc:standard .

# HIPAA
docker build --build-arg COMPLIANCE=hipaa -t svc:hipaa .

# PCI DSS
docker build --build-arg COMPLIANCE=pci -t svc:pci .

# PIPEDA (Canadian)
docker build --build-arg COMPLIANCE=pipeda -t svc:pipeda .

# FIPS (requires RUNTIME=fips)
docker build --build-arg COMPLIANCE=fips --build-arg RUNTIME=fips -t svc:fips .

# SOC 2
docker build --build-arg COMPLIANCE=soc2 -t svc:soc2 .

# CMMC
docker build --build-arg COMPLIANCE=cmmc -t svc:cmmc .

# NERC CIP
docker build --build-arg COMPLIANCE=nerc -t svc:nerc .

Observability builds

# OpenTelemetry
docker build --build-arg OBSERVABILITY=otel -t svc:otel .

# Prometheus
docker build --build-arg OBSERVABILITY=prometheus -t svc:prom .

# Datadog
docker build --build-arg OBSERVABILITY=datadog -t svc:dd .

# HIPAA + Prometheus + JWT (common healthcare API stack)
docker build \
  --build-arg COMPLIANCE=hipaa \
  --build-arg OBSERVABILITY=prometheus \
  --build-arg AUTH=jwt \
  -t svc:hipaa-prom-jwt .

Language + package manager builds

# Python with Poetry
docker build \
  --build-arg BUILD_IMAGE=python \
  --build-arg PKG_MGR=poetry \
  -t svc:py-poetry .

# Node with pnpm and esbuild (fast build)
docker build \
  --build-arg PKG_MGR=pnpm \
  --build-arg BUILD_TOOL=esbuild \
  -t svc:pnpm-esbuild .

# Go (distroless runtime — no shell)
docker build \
  --build-arg BUILD_IMAGE=golang \
  -t svc:go .

# Go + debug shell (for local dev only)
docker build \
  --build-arg BUILD_IMAGE=golang \
  --build-arg RUNTIME=slim \
  -t svc:go-dev .

Verify a built image after build

# Check all axis labels
docker inspect svc:hipaa --format '{{ json .Config.Labels }}' | jq .

# Check which compliance variant is in the image
docker run --rm --entrypoint sh svc:hipaa \
  -c "ls src/compliance/active/ && cat src/compliance/active/index.js | head -20"

# Run the service locally with required env vars
docker run --rm \
  -e JWT_SECRET=my-test-secret-at-least-32-chars \
  -e DATABASE_URL=postgresql://localhost:5432/testdb \
  -p 3000:3000 \
  svc:hipaa
