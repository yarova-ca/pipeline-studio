The one-paragraph summary

pipeline-studio is a monorepo of 106 production-ready service starters, each covering one framework. Every starter ships with a Dockerfile, Helm chart, Kustomize overlays, Backstage catalog entry, and GitHub Actions CI — ready to clone and deploy.

Each starter accepts 8 Docker build arguments (axes) that control the base image, runtime, package manager, build tool, compliance profile, observability stack, auth method, and ORM. Changing an axis at build time produces a different image — no code change required.

A new team copies a starter, sets the axes they need, and has a production-grade service in 30 minutes. Without pipeline-studio, building CI, Dockerfiles, and Helm charts from scratch takes 2–4 days per service.

1. What is pipeline-studio?

The complete definition — every component, why it exists, what it replaces.

pipeline-studio is a monorepo of 106 production-ready service starters.

Each starter covers one framework: Next.js, FastAPI, Gin, Spring Boot, Rails, Django, NestJS, Actix, and more.

Every starter ships with five artifacts, all complete and production-grade out of the box:

Artifact What it is Location
Dockerfile Multi-stage build. Accepts all 8 build axes as ARG. Produces a minimal production image. services/<name>/Dockerfile
Helm chart Parameterized Kubernetes deployment, service, ingress, HPA, and PodDisruptionBudget. services/<name>/helm/
Kustomize overlays Environment overlays for dev, staging, and prod. Patches image tags and replica counts. services/<name>/kustomize/
Backstage catalog entry catalog-info.yaml declaring the component, owner, system, and links. services/<name>/catalog-info.yaml
GitHub Actions CI Build, test, lint, SBOM, and push workflow. Parameterized by the same 8 build axes. .github/workflows/<name>.yml
Why this exists: Building a Dockerfile, Helm chart, Kustomize overlays, and CI pipeline from scratch for one service takes 2–4 days of engineering time. Across a portfolio of 10 services, that is 20–40 days. pipeline-studio reduces each service to a 30-minute copy-and-configure operation.

2. The 8 build axes — every service accepts these Docker build args

Build axes are Docker ARG values passed at build time. They control what goes into the image without changing source code.

Axis Default What it controls
BUILD_IMAGE node Base image family. Sets the runtime language and toolchain for the build stage.
RUNTIME alpine Final image variant: alpine (smallest), slim (standard Debian), fips (FIPS-compliant OpenSSL).
PKG_MGR npm Package manager. npm / pnpm / yarn / bun for Node; pip / poetry / uv for Python.
BUILD_TOOL tsc Compiler or bundler for Node backends: tsc (TypeScript), esbuild (fast bundler), swc (Rust-based transpiler).
COMPLIANCE standard Compliance profile compiled into the image: standard / hipaa / pci / pipeda / fips / soc2 / cmmc / nerc.
OBSERVABILITY none Telemetry SDK bundled into the image: none / otel (OpenTelemetry) / prometheus / datadog.
AUTH all Auth middleware compiled in: none / jwt / oauth2 / apikey / all.
ORM Varies by ecosystem ORM variant compiled into the image. Default: prisma (Node) / sqlalchemy (Python) / gorm (Go).

Example — build a HIPAA-compliant Node service with Prometheus metrics:

docker build \
  --build-arg COMPLIANCE=hipaa \
  --build-arg OBSERVABILITY=prometheus \
  --build-arg PKG_MGR=pnpm \
  .
Note: Axes are additive. You can set any combination. The Dockerfile uses ARG branching to include or exclude the relevant packages and config at build time. The final image contains only what the axes request.

3. The three canonical reference services

When a build axis is added or changed, these three services are the first to be updated and verified. They are the ground truth for how axes behave.

Service Stack Ecosystem Why canonical
14-express Node / TypeScript / Express npm / pnpm / yarn / bun Most build axes are first tested here. Covers all four Node package managers and all three Node build tools.
15-fastapi Python / FastAPI / Uvicorn pip / poetry / uv Canonical Python backend. Covers all three Python package managers. First to test Python-specific axis behavior.
16-gin Go / Gin / net/http go modules Canonical Go backend. Go has one package manager and one build tool. Tests axes that must work across all three ecosystems.
Why these three: Together they cover the three most common enterprise ecosystems — npm, pip, and go modules. A change that works on all three is safe to propagate to the remaining 103 services. A change that breaks any one of them is not ready.
Rule: When you add or modify a build axis, verify it against 14-express, 15-fastapi, and 16-gin before opening a PR. All three must build successfully with the new axis value.

5. When to use each RUNTIME value

RUNTIME controls the final image base. The wrong value causes either oversized images or build failures. Use this decision block every time.

When RUNTIME=alpine (default): smallest image, use for all services unless a specific reason to change.
Produces a musl libc base. Alpine images are 5–10x smaller than Debian-based images.
Switch away from alpine only when the build log shows a musl libc error.
When RUNTIME=slim: a native module fails on alpine.
Trigger: docker build output contains a line like Error loading shared library libssl.so: No such file or directory or cannot open shared object file: musl not found.
slim uses Debian slim (glibc). Most native modules that fail on alpine will succeed on slim.
Tradeoff: slim images are ~80MB larger than alpine equivalents.
Warning — RUNTIME=fips has a hard rule:
RUNTIME=fips is only valid when COMPLIANCE=fips is also set, OR when deploying to a federal environment that mandates FIPS-compliant OpenSSL.

When COMPLIANCE=fips is set but RUNTIME is not also set to fips: the build will fail with an explicit error.
The error message is: COMPLIANCE=fips requires RUNTIME=fips. Set --build-arg RUNTIME=fips.
RUNTIME value Base Image size When to use
alpine Alpine Linux (musl libc) Smallest — typically 20–80MB Default. Use unless build fails with a musl error.
slim Debian slim (glibc) Medium — typically 80–200MB Native module fails on alpine (musl libc error in docker build output).
fips FIPS-validated OpenSSL base Medium — typically 100–220MB COMPLIANCE=fips is set, OR deploying to a federal environment. Must pair with COMPLIANCE=fips or build fails.

7. 5-minute environment check

Run all six commands before any other work. Every command must pass. One failure blocks you.

Hard stop: Do not proceed to 20 — Local dev or any other doc until every command below returns the expected output. A missing tool will cause a confusing failure later in a build or deploy step.
  • ✓
    Docker daemon is running docker info Expected: output showing server version, running containers, and storage driver. No error text. When this fails: Docker Desktop is not running. Start it and wait for the whale icon to show as running.
  • ✓
    kubectl client v1.28 or later kubectl version --client Expected: Client Version: v1.28.x or higher. Server connection is not required at this step. When this fails: install from kubernetes.io/docs/tasks/tools
  • ✓
    Helm v3.14 or later helm version Expected: version.BuildInfo{Version:"v3.14.x"} or higher. When this fails: install from helm.sh/docs/intro/install
  • ✓
    Node.js v22 or later node --version Expected: v22.x.x or higher. When this fails: install from nodejs.org/en/download or use nvm install 22
  • ✓
    Go 1.23 or later go version Expected: go version go1.23.x or higher. When this fails: install from go.dev/doc/install
  • ✓
    Python 3.12 or later python3 --version Expected: Python 3.12.x or higher. When this fails: install from python.org/downloads or use pyenv install 3.12

Run all six as a single paste to see all results at once:

docker info              # must show running daemon
kubectl version --client # must show v1.28+
helm version             # must show v3.14+
node --version           # must show v22+
go version               # must show go1.23+
python3 --version        # must show 3.12+
When all six pass: proceed to 20 — Local dev.
When any fail: install the missing tool using the link in the check above. Re-run the command. Do not proceed until it passes.
Why strict: a missing tool produces a cryptic error deep inside a build step, not a clear "tool not found" message at the start. The 5-minute check here saves 30+ minutes of debugging later.
