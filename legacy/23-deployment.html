1. The deployment flow — overview

Every step from git push to running pod. Every step is automatic. No manual kubectl apply in normal operation.

git push → GitHub Actions (builds & pushes image) → GHCR (image stored)
↓ developer edits values.yaml with new SHA
ArgoCD (detects Git diff, syncs within 3 min) → Helm chart applied → Kubernetes rolling update → pod running
Note: ArgoCD (a GitOps tool that syncs Kubernetes cluster state from a Git repository) is the only actor that applies changes to the cluster. Developers do not run kubectl apply or helm upgrade in normal operation. Git is the single source of truth.

The two phases

Phase What happens Who triggers it How long it takes
Phase 1 — Build GitHub Actions builds the Docker image, pushes to GHCR, signs with cosign, generates SBOM, runs Trivy scan. Automatic on push to main or PR to main touching services/**. ~3–6 minutes depending on service size and cache hit rate.
Phase 2 — Deploy Developer edits values.yaml with the new image SHA. ArgoCD detects the change and applies the Helm chart. Kubernetes performs a rolling update. Developer edits the values file and pushes. ArgoCD syncs automatically. ArgoCD sync: up to 3 minutes. Rolling update: 30–120 seconds depending on pod readiness.
Why these are two separate phases: The build phase is triggered by code changes. The deploy phase is triggered by a deliberate configuration change. Separating them means a build can succeed but not be deployed until a human edits the values file. This prevents accidental auto-promotion to staging or prod.

What ArgoCD does (plain English)

ArgoCD watches a Git repository. Every 3 minutes it compares what is in Git against what is running in the cluster. If they differ, it applies the diff using Helm. The cluster always converges back to what Git says.

If someone runs kubectl apply manually and bypasses Git, ArgoCD will overwrite that change on the next sync. Git wins. Always.

2. The GitHub Actions CI pipeline

The full pipeline from source code to a signed, scanned image in GHCR.

Trigger conditions

Event Branch / path condition What runs
Push to branch Branch is main AND path matches services/** Full build: image build, push, sign, SBOM, Trivy.
Pull request PR targets main AND path matches services/** Build and scan only — image is NOT pushed to GHCR. Trivy results posted as PR comment.
Push to branch Branch is NOT main No CI runs automatically. Use local Docker build (20 Local dev).
Push touching non-service paths Path does NOT match services/** No image build. Other workflows (linting, docs checks) run if configured.

Pipeline steps — in order

  • 1
    Checkout code uses: actions/checkout@v4 Fetches the full repository including submodules. Uses fetch-depth: 0 so cosign can access full git history for provenance.
  • 2
    Set up Docker Buildx (multi-platform builder) uses: docker/setup-buildx-action@v3 Buildx is required for multi-platform builds (linux/amd64, linux/arm64). Also enables build cache export to GitHub Actions cache, which reduces build time on cache hits by 60–80%.
  • 3
    Log in to GHCR (GitHub Container Registry) uses: docker/login-action@v3 with: registry: ghcr.io username: ${{ github.actor }} password: ${{ secrets.GITHUB_TOKEN }} GITHUB_TOKEN is automatically issued per workflow run by GitHub Actions. No static credentials stored. The token expires when the workflow run ends.
  • 4
    Build image with all default build args uses: docker/build-push-action@v5 with: context: services/<service-name> file: services/<service-name>/Dockerfile platforms: linux/amd64,linux/arm64 push: ${{ github.ref == 'refs/heads/main' }} tags: ghcr.io/yarova-ca/pipeline-studio/<service>:${{ github.sha }} build-args: | NODE_VERSION=22 ALPINE_VERSION=3.20 cache-from: type=gha cache-to: type=gha,mode=max Only pushes on main. PR builds run to completion for scan results but the image is not stored. Cache mode max caches all layers including intermediate stages.
  • 5
    Push to GHCR Handled by the push: true flag in step 4. The image is pushed as ghcr.io/yarova-ca/pipeline-studio/<service>:<git-sha>. No :latest tag is pushed.
  • 6
    Sign image with cosign (supply chain security) uses: sigstore/cosign-installer@v3 # Then: cosign sign --yes \ ghcr.io/yarova-ca/pipeline-studio/<service>@${{ steps.build.outputs.digest }} cosign (a tool for signing container images using OIDC credentials) creates a verifiable signature stored alongside the image in GHCR. The signature proves the image was built by this exact GitHub Actions workflow. Any unsigned image is rejected at the cluster admission webhook.
  • 7
    Generate SBOM (Software Bill of Materials) attestation uses: anchore/sbom-action@v0 with: image: ghcr.io/yarova-ca/pipeline-studio/<service>@${{ steps.build.outputs.digest }} format: spdx-json output-file: sbom.spdx.json # Attach to image: cosign attest --yes \ --predicate sbom.spdx.json \ --type spdxjson \ ghcr.io/yarova-ca/pipeline-studio/<service>@${{ steps.build.outputs.digest }} An SBOM (a machine-readable list of every library and dependency in the image) is required for supply chain compliance. Stored as a cosign attestation alongside the image so it travels with every promotion.
  • 8
    Run Trivy security scan on the pushed image uses: aquasecurity/trivy-action@master with: image-ref: ghcr.io/yarova-ca/pipeline-studio/<service>@${{ steps.build.outputs.digest }} format: sarif output: trivy-results.sarif severity: CRITICAL,HIGH exit-code: '1' Trivy (a vulnerability scanner for container images) scans the final image for known CVEs. exit-code: 1 means the pipeline fails on CRITICAL or HIGH findings. Results are uploaded to GitHub Security tab as SARIF for triage. The pipeline does not proceed to ArgoCD promotion if Trivy fails.

Image tag format

ghcr.io/yarova-ca/pipeline-studio/<service-name>:<git-sha>

# Example:
ghcr.io/yarova-ca/pipeline-studio/api-gateway:a3f8d2c1b4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9
Decision: Git SHA as image tag — not :latest
Rejected: :latest tag
Why rejected: :latest is mutable. Rolling back with :latest is impossible — the tag always points at whatever was last pushed. Two pods can be running different images while both report :latest.
Tradeoff accepted: The values.yaml file must be edited with the exact SHA for each deployment. This is two extra lines of git work per deployment. The auditability and rollback precision gained are worth it.

Where to find the SHA after a build

Three places:

  • GitHub Actions run page — the workflow summary shows the digest and SHA.
  • git log --oneline -1 on your local branch after push.
  • GHCR package page at github.com/orgs/yarova-ca/packages — lists all tags by push time.

3. Promoting an image to a cluster environment

ArgoCD watches Git. Change Git, the cluster follows. Every environment is a separate Git path.

How ArgoCD watches for changes

ArgoCD (GitOps tool — syncs Kubernetes state from Git) is configured to watch two paths:

ArgoCD watches this Git path What it controls Sync interval
infra/argocd/ ArgoCD Application definitions — what apps exist and which cluster/namespace they deploy to. 3 minutes (default ArgoCD polling interval)
services/*/helm/values.yaml Per-service Helm values — including image.tag which controls which image version is running. 3 minutes (same polling interval)
Why separate overlays for each environment (dev / staging / prod): Dev, staging, and prod get different replica counts, resource limits (CPU/memory requests and limits), and environment variables. Sharing a single values file would force all three environments to use the same configuration. Kustomize overlays (per-environment patches applied on top of the base Helm values) let each environment diverge on exactly the fields it needs to diverge on, while inheriting everything else.

To deploy a new image to dev

  • 1
    Edit the dev values file services/<name>/helm/values.yaml Set image.tag to the new git SHA. Example: image.tag: a3f8d2c1b4e5f6a7
  • 2
    Commit and push to main git add services/<name>/helm/values.yaml git commit -m "promote <service> to <sha> in dev" git push origin main ArgoCD watches main. The commit must reach main before ArgoCD will pick it up.
  • 3
    ArgoCD detects the change within 3 minutes ArgoCD polls Git every 3 minutes by default. To force an immediate sync: argocd app sync <app-name>
  • 4
    ArgoCD applies the Helm chart with the new tag ArgoCD runs helm template with the updated values file, diffs the output against what is running in the cluster, and applies only the changed resources.
  • 5
    Kubernetes performs a rolling update Kubernetes replaces old pods with new pods one at a time (by default). The deployment does not go fully unavailable. New pods must pass readiness probes before old pods are terminated.

To promote from dev to staging

  • 1
    Edit the staging kustomize overlay kustomize/overlays/staging/kustomization.yaml Find the images: block. Update newTag to the SHA you validated in dev.
  • 2
    What the kustomization.yaml change looks like images: - name: ghcr.io/yarova-ca/pipeline-studio/<service> newTag: a3f8d2c1b4e5f6a7 # <-- update this line Kustomize overlays (per-environment config patches layered on top of the base Helm values) allow staging to get a different image tag than dev without touching the dev config.
  • 3
    Commit and push to main git add kustomize/overlays/staging/kustomization.yaml git commit -m "promote <service> to <sha> in staging" git push origin main
  • 4
    ArgoCD syncs the staging Application ArgoCD has separate Application resources for dev and staging. Updating the staging overlay only affects the staging Application. Dev is unchanged.

To promote from staging to prod

  • 1
    Edit the prod kustomize overlay kustomize/overlays/prod/kustomization.yaml Use the exact same SHA that passed in staging. Never promote a SHA that was not tested in staging.
  • 2
    Commit and push to main git add kustomize/overlays/prod/kustomization.yaml git commit -m "promote <service> to <sha> in prod" git push origin main
  • 3
    Verify the prod rollout immediately kubectl rollout status deployment/<service-name> -n <prod-namespace> Do not leave a prod promotion unmonitored. Watch the rollout to completion before closing the terminal. See Section 5 for full verification procedure.

Why separate overlays per environment — full detail

Config field Dev Staging Prod Why they differ
replicaCount 1 2 3+ Dev uses 1 replica to save cluster cost. Prod uses 3+ for high availability.
resources.requests.cpu 50m 100m 200m Dev is under-provisioned deliberately. Prod is right-sized from load testing.
resources.limits.memory 256Mi 512Mi 1Gi Prod handles real traffic volume. Dev handles one developer at a time.
image.tag latest dev SHA validated SHA release SHA Each environment points to a different commit, enabling staged promotion.
Environment variables DEV URLs, mock secrets Staging URLs, real but non-prod secrets Prod URLs, real secrets from Vault Services connect to different backing services per environment.

4. Manual deployment (for debugging — not normal operation)

Use only when ArgoCD is unavailable or you need to test a Helm chart change locally before committing.

Warning: Running helm upgrade manually bypasses ArgoCD. ArgoCD will detect the drift on its next sync and overwrite your manual change. Manual deployments are for debugging only. Any production change must go through Git.

Full helm upgrade command

helm upgrade --install <service-name> services/<name>/helm/ \
  --namespace <ns> \
  --values services/<name>/helm/values.yaml \
  --values services/<name>/helm/values.<env>.yaml \
  --set image.tag=<git-sha>

Concrete example for api-gateway in dev

helm upgrade --install api-gateway services/api-gateway/helm/ \
  --namespace pipeline-studio-dev \
  --values services/api-gateway/helm/values.yaml \
  --values services/api-gateway/helm/values.dev.yaml \
  --set image.tag=a3f8d2c1b4e5f6a7

Flags explained

Flag What it does When to change it
--install Creates the release if it does not exist. Without this flag, helm upgrade fails on a missing release. Never remove this. Always include it.
--namespace The Kubernetes namespace to deploy into. Must already exist. Change per environment: -dev, -staging, -prod suffix.
--values values.yaml Base values shared across all environments. Applied first. Never skip. This file contains all required defaults.
--values values.<env>.yaml Environment-specific overrides. Applied second — overrides base values on conflict. Swap <env> for dev, staging, or prod.
--set image.tag=<sha> Overrides the image tag at deploy time. Takes precedence over all values files. Use the exact git SHA from the CI build. Never use latest.

After a manual deploy — re-sync ArgoCD

After testing, update the values file in Git and push so ArgoCD re-owns the state.

# Re-sync ArgoCD to bring it back in line with Git:
argocd app sync <app-name>

# Or: trigger a sync via the ArgoCD web UI at:
# https://argocd.<your-domain>

5. Verifying a deployment

Four verification steps. Run them in order after any promotion.

Step 1 — Watch the rollout

kubectl rollout status deployment/<service-name> -n <namespace>

Expected output when healthy:

deployment "api-gateway" successfully rolled out

If the command hangs for more than 5 minutes, the rollout is stuck.

Press Ctrl+C and move to Step 2 to diagnose.

Step 2 — Check pods are running

kubectl get pods -n <namespace> -l app=<service-name>

Expected output when healthy:

NAME                           READY   STATUS    RESTARTS   AGE
api-gateway-7d8f9b6c4-xk2p9   1/1     Running   0          45s
api-gateway-7d8f9b6c4-m3n4p   1/1     Running   0          42s
Pod STATUS value What it means What to do
Running Pod is healthy and processing requests. Nothing. This is expected.
ContainerCreating Pod is starting. Image is being pulled or volumes are being mounted. Wait up to 2 minutes. If stuck, see Section 7 — ImagePullBackOff row.
ImagePullBackOff Kubernetes cannot pull the image from GHCR. Tag does not exist or auth failed. See Section 7 — Image pull fails row.
CrashLoopBackOff Container starts and immediately crashes. Health check or startup error. See Section 7 — Health check fails row.
Pending Pod is waiting to be scheduled. Could be resource pressure or a policy block. See Section 7 — Kyverno policy blocks pod row.
Terminating Old pods from previous rollout being cleaned up. Normal during a rolling update. Wait for it to complete.

Step 3 — Check health endpoint

# Forward the service port to localhost:
kubectl port-forward svc/<service-name> 3000:3000 -n <namespace>

# In a second terminal, hit the health endpoint:
curl http://localhost:3000/health/live

Expected response from a healthy service:

{"status":"ok","timestamp":"2026-06-05T10:00:00.000Z"}

When health check returns a non-200 status: the service is up but unhealthy.

Check the pod logs: kubectl logs <pod-name> -n <namespace> --tail=100

Step 4 — Check ArgoCD sync status

kubectl get application <service-name> -n argocd

Expected output when healthy:

NAME          SYNC STATUS   HEALTH STATUS
api-gateway   Synced        Healthy
SYNC STATUS value HEALTH STATUS value What it means What to do
Synced Healthy Cluster state matches Git. All pods running and passing health checks. Nothing. Deployment is complete.
OutOfSync Healthy Git has a change ArgoCD has not applied yet. Usually means ArgoCD is still in its polling interval. Wait 3 minutes, or run argocd app sync <name> to force immediately.
Synced Degraded ArgoCD applied the chart but the pods are not healthy. Likely a startup failure or bad config. Check pod events and logs. See Section 7.
OutOfSync Degraded Both a Git diff and an unhealthy pod. Likely a bad values.yaml was pushed. Revert the values.yaml change immediately. Push the revert. ArgoCD will re-sync.

Full verification checklist

  • 1
    Rollout status reports success kubectl rollout status deployment/<service-name> -n <namespace> Pass: "successfully rolled out". Fail: command hangs or reports error.
  • 2
    All pods in Running state, READY column shows 1/1 kubectl get pods -n <namespace> -l app=<service-name> Pass: every row shows STATUS=Running and READY=1/1. Fail: any row shows a different status.
  • 3
    Health endpoint returns 200 OK with status:ok kubectl port-forward svc/<service-name> 3000:3000 -n <namespace> curl -s http://localhost:3000/health/live | jq .status Pass: output is "ok". Fail: non-200 response or connection refused.
  • 4
    ArgoCD reports Synced / Healthy kubectl get application <service-name> -n argocd Pass: SYNC STATUS=Synced and HEALTH STATUS=Healthy. Fail: any other combination.
  • 5
    Image tag in running pod matches the SHA you promoted kubectl get pod <pod-name> -n <namespace> \ -o jsonpath='{.spec.containers[0].image}' Pass: output ends in the exact git SHA you set in values.yaml. Fail: old SHA still showing (rollout not complete or ArgoCD did not sync).

6. Rollback procedure

When a deployment fails, roll back in under 2 minutes. Three options. Pick Option B (Git revert) unless the cluster is on fire and you need Option A now.

Before rolling back: Capture the pod logs from the failing deployment first. Once you roll back, the failing pods will be terminated and logs may be lost. Run: kubectl logs <pod-name> -n <namespace> --tail=200 > /tmp/failure-logs.txt

Option A — Helm rollback (fastest, bypasses Git)

helm rollback <service-name> -n <namespace>

Helm rollback (reverts the cluster to the previous Helm release revision) takes effect in under 60 seconds.

This bypasses Git. ArgoCD will detect drift and overwrite this rollback on its next sync.

After using Option A: immediately do Option C to update Git before ArgoCD re-syncs.

# See available Helm revisions before rolling back:
helm history <service-name> -n <namespace>

# Roll back to a specific revision:
helm rollback <service-name> 3 -n <namespace>

Option B — Git revert (preferred — keeps Git as source of truth)

  • 1
    Revert the commit that changed the image tag git revert HEAD git push origin main This creates a new commit that undoes the image tag change. Git history is preserved. ArgoCD syncs the reverted values within 3 minutes. No manual cluster interaction required.
  • 2
    ArgoCD syncs the reverted values ArgoCD detects the new commit, re-renders the Helm chart with the old image tag, and applies the diff to the cluster. The rolling update replaces the failing pods with pods running the previous image.
  • 3
    Verify the rollback succeeded kubectl rollout status deployment/<service-name> -n <namespace> Confirm the pod is running again before declaring the incident resolved.
Why Option B (Git revert) is preferred over Option A (Helm rollback):
Option A bypasses Git. The cluster temporarily has a state that does not match Git. ArgoCD will overwrite the rollback when it next syncs, unless you also update Git. This creates a race condition.
Option B keeps Git as the single source of truth from the first second. No race condition. The revert commit is audit-logged. The cause and fix are both visible in git log.
Tradeoff accepted: Option B is 3 minutes slower than Option A (ArgoCD sync interval). If the cluster is on fire and 3 minutes matters, use Option A and immediately follow up with Option C.

Option C — Pin to previous tag immediately

Use when you know the exact previous SHA and want an explicit commit record (not a git revert).

  • 1
    Find the previous SHA git log --oneline services/<name>/helm/values.yaml Shows the last two commits that touched values.yaml. The second entry is the previous known-good SHA.
  • 2
    Edit values.yaml to pin to the previous SHA services/<name>/helm/values.yaml Change image.tag back to the previous SHA. Save the file.
  • 3
    Commit and push the rollback git add services/<name>/helm/values.yaml git commit -m "rollback <service> to <previous-sha>" git push origin main This gives the rollback an explicit commit with a clear message in git log, making it easier to trace the incident timeline.

Rollback options comparison

Option Speed Git is source of truth? Audit trail? Use when
A — Helm rollback <60 seconds No — cluster drifts from Git until ArgoCD syncs Helm history only Cluster on fire. Need immediate recovery. Follow up with C immediately.
B — Git revert ~3 minutes (ArgoCD sync) Yes — from first second Yes — full git log with revert commit Normal rollback. Preferred in all non-emergency cases.
C — Pin previous tag ~3 minutes (ArgoCD sync) Yes — from first second Yes — explicit rollback commit message When you want a clear commit message naming the SHA rather than a generic revert commit.

7. Failure modes during deployment

Six failure modes. Every one documented with trigger, symptom, diagnostic command, and fix.

What breaks What you see Diagnostic command Fix
Image pull fails Pod stuck in ImagePullBackOff or ErrImagePull kubectl describe pod <pod> -n <ns>

Look for: Events: section. The error will say either "not found" (tag does not exist) or "unauthorized" (GHCR credentials missing).		 1. Verify the image exists: open github.com/orgs/yarova-ca/packages and search for ghcr.io/yarova-ca/pipeline-studio/<service>:<sha>.
2. If tag missing: the CI build failed or was not pushed. Re-run the CI workflow.
3. If unauthorized: the imagePullSecret in the namespace is expired or missing. See 19 Cluster setup — GHCR pull secret section.
Helm chart invalid ArgoCD shows Degraded sync status. ArgoCD UI shows a red sync error. kubectl logs -n argocd -l app.kubernetes.io/name=argocd-server --tail=100

Also: argocd app get <app-name> shows the exact Helm rendering error.		 Read the Helm error message. Common causes: a required value is missing from values.yaml, a template syntax error, or an invalid resource definition. Fix the values file or template and push to main. ArgoCD will retry.
Health check fails / startup crash Pod in CrashLoopBackOff or OOMKilled kubectl logs <pod> -n <ns> --tail=100

Also: kubectl describe pod <pod> -n <ns> to see the last exit code and OOM indicator.		 CrashLoopBackOff: read the last 100 lines of logs for the startup error. OOMKilled: the pod hit its memory limit. Increase resources.limits.memory in the values overlay for that environment. Rollback first, then fix the config.
Kyverno policy blocks pod Pod stuck in Pending. kubectl describe pod shows admission webhook error. kubectl get events -n <ns> --sort-by=.lastTimestamp

The event will name the ClusterPolicy that denied admission and which rule matched.		 Read the policy name and rule in the event. Common violations: missing required labels, image not signed with cosign, container running as root when not permitted. Fix the Helm chart or values to comply with the policy. If the policy is wrong, update it in infra/kyverno/ and push.
PodDisruptionBudget blocks node drain Node stuck draining. kubectl drain hangs or times out. kubectl get pdb -n <ns>

Look for a PDB with minAvailable equal to the current replica count. This means zero pods can be evicted, blocking the drain forever.		 The PDB is misconfigured. It should use maxUnavailable instead of minAvailable. Fix: edit the Helm chart for that service to set podDisruptionBudget.maxUnavailable: 1 and remove the minAvailable field. Push to main. ArgoCD will update the PDB.
ArgoCD out of sync ArgoCD UI shows OutOfSync. The cluster state does not match Git. argocd app diff <app-name>

Shows exactly what ArgoCD wants to apply vs what is running. Also visible in the ArgoCD web UI diff view.		 If the diff shows your intended change: ArgoCD just has not synced yet. Wait 3 minutes or run argocd app sync <app-name>. If the diff shows unexpected drift (someone ran kubectl manually): push a no-op Git commit to trigger a clean sync, or run argocd app sync <app-name> --force.

Detailed failure blocks for the three most common failures

If image pull fails
Trigger: image.tag in values.yaml points to a SHA that does not exist in GHCR, or the imagePullSecret in the namespace is missing or expired.
System: Kubernetes retries the image pull with exponential backoff. Pod stays in ImagePullBackOff. The deployment stalls. No traffic is lost if old pods are still running (rolling update did not complete).
User sees: No visible impact if the rolling update did not complete — old pods continue serving traffic. The deployment simply does not finish.
To diagnose: kubectl describe pod <pod> -n <ns> — Events section.
To fix:
  1. If tag missing: check CI pipeline. Re-run if needed.
  2. If unauthorized: recreate the GHCR imagePullSecret in the namespace.
  3. Rollback to previous known-good SHA via Option B (Section 6).
If pod crashes on startup (CrashLoopBackOff)
Trigger: The container process exits immediately after starting. Could be: missing environment variable, unreachable database at startup, startup script error, OOM killed.
System: Kubernetes restarts the container with exponential backoff (1s, 2s, 4s, 8s, up to 5 minutes). Pod stays in CrashLoopBackOff. Rolling update stalls. Old pods continue serving.
User sees: No visible impact while old pods still running. If this is a fresh deploy (no old pods), users see service unavailable.
To diagnose: 
kubectl logs <pod> -n <ns> --previous --tail=200
kubectl describe pod <pod> -n <ns>
The --previous flag shows logs from the last crash, not the current (blank) container.
To fix: Read the crash log. Identify missing config or startup error. Fix in code or values file. Rollback first if users are impacted, then fix forward.
If Kyverno blocks the pod
Trigger: The pod spec violates a Kyverno ClusterPolicy. Common violations: image not signed with cosign, container runs as root UID, required label missing, image from non-approved registry.
System: Kubernetes admission webhook (Kyverno) rejects the pod creation request before the pod is scheduled. Pod never reaches Pending. It stays blocked at the admission layer.
User sees: Rolling update never starts. Old pods continue serving (no traffic loss).
To diagnose: 
kubectl get events -n <ns> --sort-by=.lastTimestamp | head -20
Events will name the policy and the violated rule.
To fix:
  1. If cosign signature missing: re-run the CI pipeline. The pipeline signs the image automatically. A manual push bypasses signing.
  2. If running as root: add securityContext.runAsNonRoot: true to the Helm chart.
  3. If label missing: add the required label to the Helm chart template.

8. Image signing verification

Every image pushed by CI is signed with cosign. Verify before manually pulling any image.

What cosign signing means

cosign (a tool for signing and verifying container images using short-lived OIDC credentials) attaches a cryptographic signature to each image in GHCR.

The signature proves: this exact image was built by the yarova-ca/pipeline-studio GitHub Actions workflow on GitHub’s infrastructure.

An unsigned image cannot have come from the legitimate CI pipeline. Reject it.

Verify an image before pulling

cosign verify ghcr.io/yarova-ca/pipeline-studio/<service>:<sha> \
  --certificate-identity-regexp="https://github.com/yarova-ca/pipeline-studio" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com"

Concrete example for api-gateway

cosign verify \
  ghcr.io/yarova-ca/pipeline-studio/api-gateway:a3f8d2c1b4e5f6a7 \
  --certificate-identity-regexp="https://github.com/yarova-ca/pipeline-studio" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com"

What the output means

cosign output What it means What to do
Outputs JSON with issuer, subject, and workflow fields. Exit code 0. Signature is valid. Image was built by the legitimate GitHub Actions workflow. Safe to deploy.
Error: no matching signatures. Exit code non-zero. Image was never signed. It was either built outside CI, tampered with, or the wrong image entirely. Do NOT deploy this image. Investigate where it came from. Alert the security contact for the org.
Error: certificate not found. Exit code non-zero. Image exists but the signing certificate was not stored correctly. The CI step may have failed partway through. Re-run the CI pipeline for this commit. The re-run will produce a new image with a valid signature.
Error: subject does not match. Exit code non-zero. Signature exists but was issued to a different repository or workflow path. Could indicate a supply chain attack. Do NOT deploy. Treat as a security incident. The image may be from a fork or a different repository.

How Kyverno enforces signing at the cluster level

Kyverno (the cluster admission policy engine) has a ClusterPolicy that runs cosign verification on every pod admission.

You do not need to run cosign manually before every deployment to the cluster — Kyverno does it automatically.

Manual verification is only needed when pulling an image to a local machine for inspection or debugging.

# Kyverno ClusterPolicy that enforces signing (reference — do not modify manually):
# infra/kyverno/policies/verify-image-signatures.yaml

# To see the current policy status:
kubectl get clusterpolicy verify-image-signatures

SBOM verification

Every image also has an SBOM (Software Bill of Materials — a machine-readable list of every library in the image) attached as a cosign attestation.

To view the SBOM for any image:

cosign verify-attestation \
  ghcr.io/yarova-ca/pipeline-studio/<service>:<sha> \
  --type spdxjson \
  --certificate-identity-regexp="https://github.com/yarova-ca/pipeline-studio" \
  --certificate-oidc-issuer="https://token.actions.githubusercontent.com" \
  | jq -r '.payload' | base64 -d | jq '.subject[].name'

This outputs every package name included in the image at build time.

Useful for: incident response (was a vulnerable library shipped?), compliance audits, supply chain review.

Image tag format — full reference

Tag type Format Example When used
Git SHA (primary) ghcr.io/yarova-ca/pipeline-studio/<service>:<40-char-sha> ghcr.io/yarova-ca/pipeline-studio/api-gateway:a3f8d2c1b4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9 All deployments. The only tag type deployed to clusters.
Short SHA (reference only) <service>:<7-char-sha> api-gateway:a3f8d2c Human reference only. The full 40-char SHA is used in all config files.
:latest Never used — Never. See decision block in Section 2.
Digest (immutable) <service>@sha256:<digest> api-gateway@sha256:abc123... Used by cosign for signing. ArgoCD can be configured to use digest pinning — see 19 Cluster setup.

Quick reference — most-used commands

Copy-paste commands for the five most common deployment tasks.

Deploy to dev

# Edit:
services/<name>/helm/values.yaml   # set image.tag: <sha>

# Push:
git add services/<name>/helm/values.yaml
git commit -m "promote <service> to <sha> in dev"
git push origin main

# Monitor:
kubectl rollout status deployment/<name> -n pipeline-studio-dev

Promote to staging

# Edit:
kustomize/overlays/staging/kustomization.yaml   # set newTag: <sha>

# Push:
git add kustomize/overlays/staging/kustomization.yaml
git commit -m "promote <service> to <sha> in staging"
git push origin main

# Monitor:
kubectl rollout status deployment/<name> -n pipeline-studio-staging

Promote to prod

# Edit:
kustomize/overlays/prod/kustomization.yaml   # set newTag: <sha>

# Push:
git add kustomize/overlays/prod/kustomization.yaml
git commit -m "promote <service> to <sha> in prod"
git push origin main

# Monitor (stay on this until it completes):
kubectl rollout status deployment/<name> -n pipeline-studio-prod

Rollback (preferred — Option B)

# Capture logs from the failing pod first:
kubectl logs <pod> -n <ns> --previous --tail=200 > /tmp/failure-logs.txt

# Revert the values change:
git revert HEAD
git push origin main

# Watch the rollback:
kubectl rollout status deployment/<name> -n <ns>

Force ArgoCD sync (when you can’t wait 3 minutes)

argocd app sync <app-name>

# To sync all apps at once:
argocd app sync --selector app.kubernetes.io/part-of=pipeline-studio

Check deployment health (all four checks at once)

# 1. Rollout status
kubectl rollout status deployment/<name> -n <ns>

# 2. Pod status
kubectl get pods -n <ns> -l app=<name>

# 3. ArgoCD status
kubectl get application <name> -n argocd

# 4. Image tag in running pod
kubectl get pod \
  $(kubectl get pods -n <ns> -l app=<name> -o name | head -1) \
  -n <ns> \
  -o jsonpath='{.spec.containers[0].image}'
