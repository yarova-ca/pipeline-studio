1. What “a service” means in pipeline-studio

A service is not just an application. It is a complete, self-describing, deployable unit.

Every service must have all of the following components. Missing any one component blocks deployment, compliance, or catalog registration.

Required file structure — every service must have all of these
Path What it is Required Why it is required
services/<group>-<name>/ Root directory for the service REQUIRED Naming convention enforced by CI. Wrong group number breaks catalog indexing.
Dockerfile Multi-stage build with all 8 build ARGs REQUIRED CI builds every service from this file. Missing ARGs break axis switching.
docker-compose.yml Local development compose definition REQUIRED Local dev workflow depends on docker compose up from this file.
.dockerignore Files excluded from Docker build context REQUIRED Without this, node_modules or .env files may be copied into the image.
.env.example All required environment variables with placeholder values REQUIRED Developer onboarding depends on this file. Missing it causes silent runtime failures.
helm/ Helm chart for Kubernetes deployment REQUIRED All cluster deployments go through Helm. No chart means no cluster deployment path.
kustomize/ Kustomize overlays for dev, staging, prod REQUIRED GitOps pipeline uses kustomize overlays to promote between environments.
catalog-info.yaml Backstage service catalog registration REQUIRED Without this, the service does not appear in the Backstage catalog. No discoverability.
compliance/ Pipeline compliance descriptors REQUIRED CI reads these descriptors to enforce the correct compliance pipeline for the service.
src/ Application source code with axis variant folders REQUIRED Axis switching (compliance, observability, auth, db) depends on the variant folder structure.
The axis variant folder structure inside src/

Every service must implement all four build axes. Each axis has variant subfolders. The Dockerfile copies the selected variant into the active/ slot at build time.

compliance/

  • standard/
  • active/
  • hipaa/
  • pci/
  • pipeda/
  • fips/
  • soc2/
  • cmmc/
  • nerc/

observability/

  • none/
  • active/
  • otel/
  • prometheus/
  • datadog/

auth/

  • none/
  • active/
  • jwt/
  • oauth2/
  • apikey/
  • all/

db/

  • none/
  • active/
  • prisma/
  • typeorm/
  • drizzle/
Why the active/ folder: The Dockerfile always imports from src/<axis>/active/. At build time, the selected variant is copied into active/ via a COPY instruction. The application code never references the variant name directly. This keeps axis switching transparent to the application.

2. Naming convention

Service directory names follow a strict format. Deviating from this format breaks catalog indexing and CI group detection.

Group Category Example services
01 SSR (Server-side rendering) 01-nextjs, 01-nuxt, 01-sveltekit, 01-remix, 01-astro
02 CSR (Client-side rendering) 02-react, 02-vue, 02-angular, 02-svelte
03 Static site generators 03-gatsby, 03-hugo, 03-eleventy
04 Mobile (React Native, Expo) 04-expo, 04-react-native
05 Desktop (Electron, Tauri) 05-electron, 05-tauri
06 CLI tools 06-oclif, 06-commander
07 Serverless / Edge functions 07-cloudflare-workers, 07-aws-lambda
08 gRPC services 08-grpc-node, 08-grpc-go
09 GraphQL servers 09-apollo, 09-pothos
10 Message queue workers 10-bullmq, 10-kafka-consumer
11 Data / ML pipelines 11-spark, 11-ray
12 Databases / Stateful services 12-postgres-sidecar
13 Infrastructure tools 13-nginx, 13-envoy
14 Node.js HTTP backends 14-express, 14-fastify, 14-hono, 14-koa, 14-nest
15 Python HTTP backends 15-fastapi, 15-flask, 15-django, 15-litestar
16 Go HTTP backends 16-gin, 16-echo, 16-fiber, 16-chi

18 Java / JVM backends 18-spring, 18-quarkus, 18-micronaut
19 .NET backends 19-aspnet, 19-minimal-api
20 Ruby backends 20-rails-api, 20-sinatra
Name format rules
  • R1
    Format: <group>-<framework-name> All lowercase. Hyphens only. No underscores. No spaces. No dots.
  • R2
    Group number: must match the taxonomy table above. CI uses the group number to select the correct base image family and pipeline variant.
  • R3
    Framework name: use the official framework name, lowercased. 14-fastify not 14-fast or 14-fastifyjs. The name must be globally unique within the group.
  • R4
    New group: add the group to the taxonomy table in this doc and in 01-framework-catalog.html before adding the service. A service in an undocumented group will pass CI but fail catalog import.
Name clash: If two services have the same group-name pair, CI will fail on the catalog import step with a duplicate entity key error. Check 01-framework-catalog.html before creating the directory.

3. Step-by-step: adding 14-fastify as an example

This walkthrough uses 14-fastify as the concrete example. Substitute your service name and group wherever 14-fastify or 14-express appears.

Note: All commands run from the repository root: /path/to/pipeline-studio. Never run these commands from inside a service directory.

Step 2: Create the Dockerfile

Copy the existing 14-express Dockerfile as the starting template. The Dockerfile structure is identical across all Node.js group 14 services.

cp services/14-express/Dockerfile services/14-fastify/Dockerfile

Open services/14-fastify/Dockerfile and make the following changes:

What to change From (Express) To (Fastify) Why
Image label LABEL service=14-express LABEL service=14-fastify CI uses this label to correlate image to catalog entry.
npm start script CMD ["node", "dist/index.js"] Same, unless Fastify's package.json defines a different entry point. Only change if Fastify entry point differs.
Port exposure EXPOSE 3000 EXPOSE 3001 (if port differs) Fastify default port is 3000 but change if your service uses a different port.
Do not change these: All 8 ARG declarations must remain unchanged. All COPY src/<axis>/${AXIS}/ lines must remain unchanged. These are the axis mechanism. Modifying them breaks all axis switching for this service.

The 8 required ARG declarations (verify all 8 are present):

ARG NODE_VERSION=20
ARG ALPINE_VERSION=3.19
ARG COMPLIANCE=standard
ARG OBSERVABILITY=none
ARG AUTH=none
ARG DB=none
ARG RUNTIME=default
ARG BUILD_ENV=production

The required COPY axis lines (verify all 4 are present):

COPY src/compliance/${COMPLIANCE}/ ./src/compliance/active/
COPY src/observability/${OBSERVABILITY}/ ./src/observability/active/
COPY src/auth/${AUTH}/ ./src/auth/active/
COPY src/db/${DB}/ ./src/db/active/
If an ARG is missing from the Dockerfile
Trigger: docker build --build-arg COMPLIANCE=hipaa . fails with warning: one or more build-args [COMPLIANCE] were not consumed and the wrong variant is copied.
System: The COPY line uses an empty string fallback, copying nothing into active/.
Fix: Add the missing ARG declaration before the first FROM that needs it.
Verify: grep "^ARG " services/14-fastify/Dockerfile | wc -l must return 8.

Step 3: Create the application source

Create the application entry point. Every service must implement three endpoints. These endpoints are required by the cluster health check system and the compliance pipeline.

Create the entry point:

touch services/14-fastify/src/index.ts

Minimum viable src/index.ts for a Fastify service:

import Fastify from 'fastify'
import './observability/active/index.js'
import { applyCompliance } from './compliance/active/index.js'

const fastify = Fastify({ logger: true })

// Apply compliance middleware (HIPAA headers, PCI masking, etc.)
applyCompliance(fastify)

// ── Health endpoints (REQUIRED on every service) ──────────────────────────────

// GET /health/live
// Returns 200 with {"status":"ok"} always.
// No auth required. No DB check. Used by Kubernetes liveness probe.
fastify.get('/health/live', async (_req, reply) => {
  return reply.status(200).send({ status: 'ok' })
})

// GET /health/ready
// Returns 200 {"status":"ready"} when the service can accept traffic.
// Returns 503 {"status":"unavailable"} when DB or dependencies are not ready.
// No auth required. Used by Kubernetes readiness probe.
fastify.get('/health/ready', async (_req, reply) => {
  try {
    // TODO: replace with real DB check when DB axis is active
    return reply.status(200).send({ status: 'ready' })
  } catch {
    return reply.status(503).send({ status: 'unavailable' })
  }
})

// GET /metrics
// Only active when OBSERVABILITY=prometheus.
// The observability/active/index.js file registers this route when prometheus variant is active.
// When observability=none, this route is not registered.

const port = Number(process.env.PORT ?? 3000)
const host = process.env.HOST ?? '0.0.0.0'

fastify.listen({ port, host }, (err) => {
  if (err) {
    fastify.log.error(err)
    process.exit(1)
  }
})
Required endpoint contract — all three must be implemented on every service
Endpoint Method Success response Failure response Auth required Used by
/health/live GET 200 {"status":"ok"} Never — this endpoint always returns 200 None Kubernetes liveness probe
/health/ready GET 200 {"status":"ready"} 503 {"status":"unavailable"} None Kubernetes readiness probe
/metrics GET Prometheus text format 404 when OBSERVABILITY≠prometheus None (internal network only) Prometheus scrape
Never return 404 or 500 from /health/ready. The readiness probe must return exactly 200 or 503. Any other status code causes Kubernetes to remove the pod from the load balancer rotation and then immediately restart it, causing a crash loop.

Step 4: Wire the compliance and observability axes

The axis wiring is two import lines at specific positions in src/index.ts. Both lines are required. Neither is optional.

Import line Position in index.ts What it does
import './observability/active/index.js' First line, before any other imports Initializes the observability variant (OTel tracer, Prometheus registry, or no-op). Must be first to capture all spans including Fastify initialization.
import { applyCompliance } from './compliance/active/index.js'
then applyCompliance(fastify)		 After Fastify is created, before any routes are registered Registers compliance hooks (HIPAA audit logging, PCI header masking, FIPS cipher enforcement). Must run before routes so hooks apply to all routes.

The observability/active/index.ts export contract:

// Every observability variant must export a default no-op or real init.
// The file may have side effects (registering tracer, Prometheus registry).
// It must not throw. If the variant is "none", the file is an empty module.

// observability/none/index.ts
export {}   // empty — no-op

// observability/otel/index.ts
import { NodeSDK } from '@opentelemetry/sdk-node'
// ... OTel SDK initialization as side effect

The compliance/active/index.ts export contract:

// Every compliance variant must export applyCompliance.
// applyCompliance receives the Fastify instance and registers hooks.
// It must not throw. If the variant is "standard", the function is a no-op.

// compliance/standard/index.ts
import type { FastifyInstance } from 'fastify'
export function applyCompliance(_app: FastifyInstance): void {
  // standard = no additional compliance controls
}

// compliance/hipaa/index.ts
import type { FastifyInstance } from 'fastify'
export function applyCompliance(app: FastifyInstance): void {
  app.addHook('onRequest', async (request, reply) => {
    // Add HIPAA audit header
    reply.header('X-Audit-Id', crypto.randomUUID())
    // Mask PHI fields in request logging
  })
}

Step 5: Copy compliance variant files

Copy all 9 compliance variant folders from the closest existing service in the same group. For group 14, copy from 14-express.

# Copy all 9 compliance variants from 14-express
for variant in standard hipaa pci pipeda fips soc2 cmmc nerc; do
  cp -r services/14-express/src/compliance/${variant}/ \
        services/14-fastify/src/compliance/${variant}/

After copying, update each variant file to use Fastify types instead of Express types. The only changes required are in the function signature and middleware registration.

Framework adaptation table — compliance variant changes by language
Framework Type import Middleware registration Request object Response object
Express import type { Express } from 'express' app.use((req, res, next) => { ... }) req: Request res: Response
Fastify import type { FastifyInstance } from 'fastify' app.addHook('onRequest', async (request, reply) => { ... }) request: FastifyRequest reply: FastifyReply
Hono import type { Hono } from 'hono' app.use('*', async (c, next) => { ... }) c.req c (context)
Koa import type { Application } from 'koa' app.use(async (ctx, next) => { ... }) ctx.request ctx.response
FastAPI (Python) from fastapi import FastAPI, Request, Response app.add_middleware(BaseHTTPMiddleware, dispatch=handler) request: Request response: Response
Flask (Python) from flask import Flask, request, g @app.before_request request (global) g (global context)
Gin (Go) import "github.com/gin-gonic/gin" r.Use(func(c *gin.Context) { ... }) c *gin.Context c *gin.Context
Echo (Go) import "github.com/labstack/echo/v4" e.Use(func(next echo.HandlerFunc) echo.HandlerFunc { ... }) c.Request() c.Response()
Do not change: all logging logic, all header values, all PHI/PII masking logic. These are compliance requirements. Only the function signature and middleware registration method change. The business logic inside the hook remains identical.

Example: HIPAA variant before (Express) and after (Fastify):

// BEFORE: compliance/hipaa/index.ts (Express)
import type { Express, Request, Response, NextFunction } from 'express'

export function applyCompliance(app: Express): void {
  app.use((req: Request, res: Response, next: NextFunction) => {
    const auditId = crypto.randomUUID()
    res.setHeader('X-Audit-Id', auditId)
    res.setHeader('X-PHI-Access-Log', 'true')
    // Log access for HIPAA audit trail
    console.log(JSON.stringify({
      auditId,
      method: req.method,
      path: req.path,
      ip: req.ip,
      timestamp: new Date().toISOString()
    }))
    next()
  })
}
// AFTER: compliance/hipaa/index.ts (Fastify)
import type { FastifyInstance, FastifyRequest, FastifyReply } from 'fastify'

export function applyCompliance(app: FastifyInstance): void {
  app.addHook('onRequest', async (request: FastifyRequest, reply: FastifyReply) => {
    const auditId = crypto.randomUUID()
    reply.header('X-Audit-Id', auditId)
    reply.header('X-PHI-Access-Log', 'true')
    // Log access for HIPAA audit trail — UNCHANGED from Express version
    console.log(JSON.stringify({
      auditId,
      method: request.method,
      path: request.url,
      ip: request.ip,
      timestamp: new Date().toISOString()
    }))
    // Note: Fastify onRequest hooks do not call next(). Just return.
  })
}

4. Compliance variant adaptation guide

When copying compliance variants from one framework to another, make the minimum required changes. The compliance logic must not change. Only the integration method changes.

The rule: change exactly one thing per variant
What changes What does NOT change
The function signature: Express → FastifyInstance → Gin → FastAPI All audit logging logic
The middleware registration method: app.use → addHook → r.Use → add_middleware All header names and header values
The request/response object types to match the target framework All PHI/PII masking logic
— All SOC2 control identifiers
— All CMMC practice references
— All NERC CIP requirement numbers
— All FIPS cipher enforcement logic
— All PCI PAN masking logic
Variant-by-variant adaptation checklist

hipaa/index.ts — HIPAA audit trail

Adds: request audit ID header, PHI access log, patient data masking in error responses.

// hipaa/index.ts — Fastify
import type { FastifyInstance } from 'fastify'
export function applyCompliance(app: FastifyInstance): void {
  app.addHook('onRequest', async (request, reply) => {
    const auditId = crypto.randomUUID()
    reply.header('X-Audit-Id', auditId)
    reply.header('X-PHI-Access-Log', 'true')
    request.log.info({
      auditId, method: request.method, url: request.url, ip: request.ip
    }, 'HIPAA access log')
  })
  // Mask PHI fields in error responses
  app.addHook('onError', async (request, reply, error) => {
    // Remove any patient identifiers from error message
    error.message = error.message.replace(/\b\d{3}-\d{2}-\d{4}\b/g, '[SSN-REDACTED]')
  })
}

pci/index.ts — PCI DSS card data masking

Adds: PAN masking in logs, security headers required by PCI DSS, TLS enforcement header.

// pci/index.ts — Fastify
import type { FastifyInstance } from 'fastify'
export function applyCompliance(app: FastifyInstance): void {
  app.addHook('onRequest', async (_request, reply) => {
    reply.header('Strict-Transport-Security', 'max-age=31536000; includeSubDomains')
    reply.header('X-Frame-Options', 'DENY')
    reply.header('X-Content-Type-Options', 'nosniff')
    reply.header('X-PCI-Zone', 'cardholder-data-environment')
  })
  // Mask PAN in any logged request body
  app.addHook('preHandler', async (request) => {
    if (request.body && typeof request.body === 'object') {
      const body = request.body as Record<string, unknown>
      for (const key of Object.keys(body)) {
        if (key.match(/card|pan|cvv|expiry/i) && typeof body[key] === 'string') {
          const val = body[key] as string
          body[key] = val.replace(/\d(?=\d{4})/g, '*')
        }
      }
    }
  })
}

5. Verification checklist

Run all 10 verify commands before opening a PR. All 10 must pass. A single failure blocks the PR merge.

Run from the service directory: cd services/14-fastify before running the Docker commands below. The helm and kubectl commands run from the service directory too.
  • V1
    Default Docker build must complete with no errors docker build -t test-14-fastify:default . Verifies the Dockerfile, all 8 ARGs present, and standard axis variant files exist.
  • V2
    docker compose up must start and respond on health endpoint docker compose up -d && sleep 5 && curl -sf http://localhost:3000/health/live Verifies the service starts, binds to the correct port, and the health endpoint is reachable.
  • V3
    /health/live must return exactly {"status":"ok"} curl -s http://localhost:3000/health/live | jq -e '.status == "ok"' Kubernetes liveness probe depends on this exact response. jq -e returns exit 1 if condition is false.
  • V4
    /health/ready must return 200 or 503 (not 404, not 500) STATUS=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/health/ready); [ "$STATUS" = "200" ] || [ "$STATUS" = "503" ] && echo PASS || echo "FAIL: got $STATUS" Any status other than 200 or 503 indicates the readiness endpoint is missing or broken.
  • V5
    COMPLIANCE=hipaa build must complete with no errors docker build --build-arg COMPLIANCE=hipaa -t test-14-fastify:hipaa . Verifies the hipaa compliance variant files exist and have no TypeScript compile errors.
  • V6
    OBSERVABILITY=prometheus build must complete with no errors docker build --build-arg OBSERVABILITY=prometheus -t test-14-fastify:prometheus . Verifies the prometheus observability variant files exist and compile correctly.
  • V7
    FIPS multi-arg build must complete with no errors docker build --build-arg COMPLIANCE=fips --build-arg RUNTIME=fips -t test-14-fastify:fips . Verifies that multiple build ARGs can be combined without conflict, and the FIPS runtime base image resolves correctly.
  • V8
    catalog-info.yaml must validate with 0 errors npx @backstage/cli catalog-info validate catalog-info.yaml Backstage will reject the catalog entry silently if it has validation errors. Run this before pushing.
  • V9
    Helm template must produce valid YAML with no errors helm template helm/ --debug 2>&1 | grep -E "^(Error|FAIL)" || echo PASS Verifies the Helm chart renders correctly. Any template error is a deployment blocker.
  • V10
    Kustomize dev overlay must pass dry-run kubectl --dry-run=client apply -k kustomize/overlays/dev/ Verifies the Kustomize overlay produces valid Kubernetes manifests and all resource names are correct.

Go example: compliance hook pattern (Gin)
// compliance/hipaa/middleware.go — Gin
package compliance

import (
  "crypto/rand"
  "encoding/json"
  "fmt"
  "log"
  "time"

  "github.com/gin-gonic/gin"
)

// ApplyCompliance registers HIPAA compliance hooks on the Gin router.
// Call this after creating the router and before registering any routes.
func ApplyCompliance(r *gin.Engine) {
  r.Use(hipaaAuditMiddleware())
}

func hipaaAuditMiddleware() gin.HandlerFunc {
  return func(c *gin.Context) {
    auditID := newUUID()
    c.Header("X-Audit-Id", auditID)
    c.Header("X-PHI-Access-Log", "true")

    // Log access for HIPAA audit trail
    entry, _ := json.Marshal(map[string]string{
      "auditId":   auditID,
      "method":    c.Request.Method,
      "path":      c.Request.URL.Path,
      "ip":        c.ClientIP(),
      "timestamp": time.Now().UTC().Format(time.RFC3339),
    })
    log.Println(string(entry))

    c.Next()
  }
}

func newUUID() string {
  b := make([]byte, 16)
  rand.Read(b)
  return fmt.Sprintf("%x-%x-%x-%x-%x", b[0:4], b[4:6], b[6:8], b[8:10], b[10:])
}
Python example: compliance hook pattern (FastAPI)
# compliance/hipaa/__init__.py — FastAPI
import uuid
import time
import json
import logging
from fastapi import FastAPI, Request, Response
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.types import ASGIApp

logger = logging.getLogger(__name__)

class HIPAAAuditMiddleware(BaseHTTPMiddleware):
    """HIPAA audit middleware.

    Adds X-Audit-Id header to every response.
    Logs all access with patient-data masking.
    """

    async def dispatch(self, request: Request, call_next) -> Response:
        audit_id = str(uuid.uuid4())

        # Log access for HIPAA audit trail
        logger.info(json.dumps({
            "auditId": audit_id,
            "method": request.method,
            "path": request.url.path,
            "ip": request.client.host if request.client else "unknown",
            "timestamp": time.strftime("%Y-%m-%dT%H:%M:%SZ", time.gmtime()),
        }))

        response = await call_next(request)
        response.headers["X-Audit-Id"] = audit_id
        response.headers["X-PHI-Access-Log"] = "true"
        return response


def apply_compliance(app: FastAPI) -> None:
    """Register HIPAA compliance middleware on the FastAPI app.

    Call this after creating the app and before adding any routes.
    """
    app.add_middleware(HIPAAAuditMiddleware)

8. Common mistakes and how to avoid them

Mistake What happens How to detect Prevention
Forgetting to create an active/ directory for an axis Docker build fails with COPY error at the axis COPY line find src/ -name "active" -type d | wc -l — must be 4 Use the mkdir -p command from Step 1 verbatim. Do not shorten it.
Leaving import type { Express } in a copied compliance file TypeScript compile error during Docker build grep -r "import type { Express }" src/compliance/ After copying each compliance file, grep for Express imports immediately.
Missing ARG declaration in Dockerfile Build ARG is ignored, wrong variant is copied grep "^ARG " Dockerfile | wc -l — must be 8 Copy the full ARG block from an existing service. Never write ARGs by hand.
Setting spec.lifecycle: production before cluster testing The service appears in production catalogs before it is ready grep "lifecycle" catalog-info.yaml Default to experimental. Promote to production only after all 10 verify steps pass on a real cluster.
Not updating all 4 docs pages after adding the service The catalog shows the wrong count. The status dashboard misses the service. CI link-check fails on catalog-info.yaml references Add doc updates as required items in the PR checklist.
Forgetting to add the service to backstage/catalog/all.yaml The service never appears in the Backstage catalog despite having a valid catalog-info.yaml grep "14-fastify" backstage/catalog/all.yaml Step 10 is always the last code change. Never skip it.
Copying Helm _helpers.tpl without renaming the service helm template fails with undefined function or wrong labels grep "14-express" helm/templates/_helpers.tpl After cp -r, immediately run sed -i 's/14-express/14-fastify/g' helm/templates/_helpers.tpl
Using app.use() in a Fastify compliance variant Runtime error: app.use is not a function grep -r "app\.use" src/compliance/ Fastify uses app.addHook('onRequest', ...), not app.use().
Returning next() in a Fastify addHook handler Runtime error: next is not a function Code review — look for next() calls in Fastify hook bodies Fastify hooks are async. No next parameter. Just return or throw.
/health/ready returns 500 instead of 503 Kubernetes sees a crash, restarts the pod, causes crash loop V4 verify step catches this Wrap the ready handler in try/catch. Return 503 in the catch block, never let exceptions propagate.

#!/bin/bash
# Add a new pipeline-studio service
# Usage: set SERVICE and SOURCE below, then run the script.

SERVICE="14-fastify"   # the new service directory name
SOURCE="14-express"    # the existing service to copy from
PORT="3000"            # the port this service listens on

echo "=== Step 1: Create directory structure ==="
mkdir -p services/${SERVICE}/src/compliance/{standard,active,hipaa,pci,pipeda,fips,soc2,cmmc,nerc}
mkdir -p services/${SERVICE}/src/observability/{none,active,otel,prometheus,datadog}
mkdir -p services/${SERVICE}/src/auth/{none,active,jwt,oauth2,apikey,all}
mkdir -p services/${SERVICE}/src/db/{none,active,prisma,typeorm,drizzle}
mkdir -p services/${SERVICE}/helm/templates
mkdir -p services/${SERVICE}/kustomize/{base,overlays/{dev,staging,prod}}
mkdir -p services/${SERVICE}/compliance
echo "PASS: directory structure created"

echo "=== Step 2: Copy Dockerfile and update service name ==="
cp services/${SOURCE}/Dockerfile services/${SERVICE}/Dockerfile
sed -i "s/LABEL service=${SOURCE}/LABEL service=${SERVICE}/g" services/${SERVICE}/Dockerfile
echo "PASS: Dockerfile copied. Review and update npm scripts if needed."

echo "=== Step 3: Copy compose and env files ==="
cp services/${SOURCE}/docker-compose.yml services/${SERVICE}/docker-compose.yml
cp services/${SOURCE}/.dockerignore services/${SERVICE}/.dockerignore
cp services/${SOURCE}/.env.example services/${SERVICE}/.env.example
sed -i "s/${SOURCE}/${SERVICE}/g" services/${SERVICE}/docker-compose.yml
echo "PASS: compose and env files copied"

echo "=== Step 4: Copy compliance variants ==="
for variant in standard hipaa pci pipeda fips soc2 cmmc nerc; do
  cp -r services/${SOURCE}/src/compliance/${variant}/ \
        services/${SERVICE}/src/compliance/${variant}/
done

        services/${SERVICE}/src/observability/${variant}/
done
for variant in none jwt oauth2 apikey all; do
  cp -r services/${SOURCE}/src/auth/${variant}/ \
        services/${SERVICE}/src/auth/${variant}/
done
for variant in none prisma typeorm drizzle; do

        services/${SERVICE}/src/db/${variant}/
done

find services/${SERVICE}/kustomize -type f -name "*.yaml" | \
  xargs sed -i "s/${SOURCE}/${SERVICE}/g"
echo "PASS: Kustomize overlays copied and updated"

cp services/${SOURCE}/catalog-info.yaml services/${SERVICE}/catalog-info.yaml
