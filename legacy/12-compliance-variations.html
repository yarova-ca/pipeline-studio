Compliance Variations
What this covers: How 8 compliance values change runtime middleware behaviour. Each value is a named variant shipped in src/compliance/active/index.ts. Covers standard, hipaa, pci, pipeda, fips, soc2, cmmc, nerc.
When to read: Choosing a compliance value for a service. Debugging middleware behaviour. Read after doc 11 (Dockerfile catalog) and doc 05 (Invariants).
Pillar 5 — Compliance & Security  ·  Related: 11 Dockerfiles  ·  10 Distro compliance
Decision: runtime middleware variants — one module per compliance value, loaded at startup
Rejected: Dockerfile-only compliance. Dockerfile controls the image layer. It cannot enforce request-level rules (e.g., block HTTP, mask PHI in logs, check consent headers). Runtime middleware is required for those guarantees.
Accepted tradeoff: middleware adds ~0.2ms per request for audit-log variants. Measured acceptable. The guarantee — that no PHI or PCI data escapes in logs or responses — is not achievable at the Dockerfile layer.

How the middleware model works

The compliance value is set at deploy time. The service loads one module. That module installs all required middleware.

The module contract

1
Every service imports src/compliance/active/index.ts. That file exports one function: applyCompliance(app). The function installs middleware appropriate for the active compliance value.
2
The active compliance value is selected at build time. The active/ symlink (or conditional import) points to the correct variant. One service image = one compliance value. No branching at runtime.
3
standard is the default. applyCompliance is a no-op. No middleware installed. No overhead. Every new service starts here.
4
fips is the only startup-check-only variant. It runs one check at startup and throws if FIPS mode is not active. It installs no request middleware. The Dockerfile RUNTIME=fips build guard enforces the correct base image.
5
All other values install request middleware. The middleware runs on every incoming request. Audit log, PII masking, header enforcement, and network isolation are all expressed as middleware.

The 8 compliance values — overview

What each value does and when to use it.

standard — no-op

The default compliance value. No middleware installed. No overhead.

standard
File: src/compliance/active/index.ts (default export)
applyCompliance is exported and is a no-op. No middleware is installed. No request overhead. This is the starting point for every new service.
// src/compliance/active/index.ts — standard variant export function applyCompliance(app: Express | FastAPI | GinRouter): void { // no-op — no compliance middleware required for standard tier }
All services import this function. Standard services get a no-op. Switching to a regulated variant replaces only this module — no service code changes required.

hipaa — PHI protection

Middleware runs on every request. PHI fields are masked in all logs. Verbose errors are disabled.

hipaa
File: src/compliance/hipaa/index.ts
HIPAA requires an audit log for every access to PHI and requires that PHI does not appear in system logs. The logger patch ensures no PHI field escapes — even if application code logs a full object.
Audit log — every request
// Fields logged on every request { event: "hipaa_access", actor: req.user?.id ?? "anonymous", action: req.method, resource: req.path, requestId: req.headers["x-request-id"], timestamp: new Date().toISOString() }
PHI masking — logger patch
// Fields masked in every log call — value replaced with ***REDACTED*** const PHI_FIELDS = [ "ssn", "dob", "mrn", "diagnosis", "patientName", "address", "phone" ] // Node: logger.info is patched at startup // Python: BaseHTTPMiddleware patches the logging handler // Go: slog handler wraps each log call
Verbose errors — disabled
// Node (Express) app.set("env", "production") // disables stack traces in error responses // Python (FastAPI / Starlette) app.debug = False // Go (Gin) gin.SetMode(gin.ReleaseMode)
The audit log and PHI masking are both active on every request — not just requests that touch PHI fields. This is intentional: HIPAA auditors require that every access attempt is logged, not just successful ones.

pci — cardholder data environment

Startup check blocks weak secrets. Request middleware enforces TLS and blocks debug endpoints.

pci
File: src/compliance/pci/index.ts
PCI DSS requires TLS on all cardholder data traffic, prohibits debug/diagnostic endpoints in production, and requires that secrets meet minimum length requirements.
Startup check — weak secrets
// Runs once at process startup — before any requests are served for (const [key, value] of Object.entries(process.env)) { if (/SECRET|KEY|TOKEN|PASS/i.test(key) && (value?.length ?? 0) < 16) { // fatal — process exits, service does not start process.exit(1) } }
Request middleware
// Rule 1: Block plain HTTP if (req.headers["x-forwarded-proto"] === "http") { res.status(403).json({ error: "TLS required" }) return } // Rule 2: Block debug and metrics endpoints if (/^\/debug\/|^\/metrics/.test(req.path)) { res.status(404).end() return } // Rule 3: Add required response headers res.setHeader("X-PCI-DSS", "enforced") res.setHeader("Strict-Transport-Security", "max-age=31536000; includeSubDomains")
The startup check is fatal. When a secret is shorter than 16 characters, the service refuses to start. This is intentional — a service that starts with a weak secret in a PCI environment is itself a compliance violation.

pipeda — Canadian personal information

Consent gate on all write methods. Canadian data-residency header on all responses. PII fields masked.

pipeda
File: src/compliance/pipeda/index.ts
PIPEDA requires informed consent before collecting personal information. The consent gate enforces this at the API layer. The data-residency header signals to downstream systems that data must not leave Canada.
Consent gate — POST, PUT, PATCH
// Applies to POST, PUT, PATCH only — read-only methods are not gated if (["POST", "PUT", "PATCH"].includes(req.method)) { if (req.headers["x-consent"] !== "granted") { // 451 Unavailable For Legal Reasons — IETF RFC 7725 res.status(451).json({ error: "Consent required (PIPEDA)" }) return } }
Data-residency header — all responses
res.setHeader("X-Data-Residency", "CA")
PII masking — logger patch
// Fields masked in every log call — value replaced with *** const PII_FIELDS = [ "email", "name", "phone", "address", "sin" ]
The X-Data-Residency header is advisory — it signals intent to downstream services and gateways. Actual data residency is enforced at the infrastructure layer (ca-central-1 AWS or Canada Central Azure). The header alone is not sufficient for PIPEDA compliance.

fips — FIPS 140 cryptographic validation

Startup check only. No request middleware. Fails hard when OpenSSL FIPS mode is not active.

fips
File: src/compliance/fips/index.ts · Requires: RUNTIME=fips Dockerfile build guard
FIPS 140 requires that all cryptographic operations use validated modules. If the OS is not in FIPS mode, all crypto — including TLS — is non-compliant. The startup check catches this before any traffic is served.
Startup check — Node
// Node — checks OpenSSL FIPS flag import crypto from "node:crypto" if (!crypto.getFips()) { throw new Error("FIPS mode is not active. Service cannot start.") }
Startup check — Python
# Python — checks OpenSSL FIPS flag via ssl module import ssl if not ssl.OPENSSL_VERSION.startswith("OpenSSL") or \ "fips" not in ssl.OPENSSL_VERSION.lower(): raise RuntimeError("FIPS mode is not active. Service cannot start.")
Startup check — Go
// Go — checks FIPS build tag and OpenSSL binding if !fipsEnabled() { log.Fatal("FIPS mode is not active. Service cannot start.") }
Dockerfile build guard
# The Dockerfile enforces RUNTIME=fips at build time. # Services built without --build-arg RUNTIME=fips will not have the UBI9 FIPS base. # The startup check catches the case where the wrong image is deployed. ARG RUNTIME=standard # When RUNTIME=fips: base image is ubi9/nodejs (FIPS-validated OpenSSL) # When RUNTIME=standard: base image is node:22-alpine (non-FIPS)
fips installs no request middleware. The compliance guarantee is at the crypto layer, not the HTTP layer. Every TLS connection, every hash, every signature uses the validated OpenSSL FIPS module automatically — no per-request code required.

soc2 — SOC 2 audit trail

Immutable audit log on every request. Verbose errors disabled.

soc2
File: src/compliance/soc2/index.ts
SOC 2 Type II requires evidence of continuous monitoring and an immutable audit trail. The soc2_audit event provides that trail. Verbose errors are disabled to prevent sensitive system information from appearing in API responses.
Audit log — every request
// Fields logged on every request { event: "soc2_audit", actor: req.user?.id ?? "anonymous", action: req.method, resource: req.path, timestamp: new Date().toISOString() }
Verbose errors — disabled
// Same pattern as hipaa — production mode in all three runtimes app.set("env", "production") // Node app.debug = False // Python gin.SetMode(gin.ReleaseMode) // Go
The soc2_audit log must be shipped to an immutable log store (e.g., CloudWatch Logs with object lock, S3 with WORM). The middleware emits the event — shipping is configured at the infrastructure layer via the service's log driver.

cmmc — CUI network isolation

Audit log on every request. Debug endpoints blocked. Network egress marker set.

cmmc
File: src/compliance/cmmc/index.ts
CMMC requires an audit log for every access to CUI, prohibition of debug interfaces in production, and network isolation to prevent CUI exfiltration. NO_EGRESS=true signals the network policy to block all outbound connections.
Audit log — every request
{ event: "cmmc_audit", actor: req.user?.id ?? "anonymous", action: req.method, resource: req.path, timestamp: new Date().toISOString() }
Request rules
// Block debug endpoints if (/^\/debug\//.test(req.path)) { res.status(404).end() return } // Set network isolation marker process.env.NO_EGRESS = "true" // Disable verbose errors app.set("env", "production") // Node app.debug = False // Python gin.SetMode(gin.ReleaseMode) // Go
NO_EGRESS=true is a process-level marker. The actual network policy (blocking outbound connections) is enforced by the Kubernetes NetworkPolicy or firewall. The marker tells the application layer to skip any outbound calls it would otherwise make (webhooks, telemetry).

nerc — OT/IT boundary enforcement

OT/IT boundary headers on all responses. Network egress marker set.

nerc
File: src/compliance/nerc/index.ts
NERC CIP requires strict separation between OT (operational technology) and IT networks. The response headers signal to gateways and proxies that traffic has crossed the OT/IT boundary and must be handled accordingly.
Response headers — all responses
res.setHeader("X-OT-IT-Boundary", "enforced") res.setHeader("X-Network-Isolation", "on")
Network isolation marker
// Set on startup — same marker as cmmc process.env.NO_EGRESS = "true"
NERC CIP compliance also requires an air-gapped image registry (no public pulls) and a 35-day patch cycle. Both are enforced at the Dockerfile and CI/CD layer — see doc 11. The middleware enforces only the runtime-observable requirements.
