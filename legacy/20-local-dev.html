20 — Local Development
What this covers: How to run any of the 106 services locally. Prerequisites, one-command setup, verify steps, and failure modes for every tool.
When to read: Before writing any code. When setting up a new machine. When a service fails to start locally.
Guides · Related: 21 Build axes · 22 Add a service · 24 Debugging

How local development works

Every service in services/ runs in two modes: Docker Compose (fastest, no native tooling required) and natively (faster iteration once tools are installed).

Docker Compose is the default path. Use native mode when you need hot-reload or are actively changing source files.

All services expose /health/live and /health/ready. Live returns 200 when the process is running. Ready returns 200 only when the database is reachable.

1. Prerequisites

Install all six tools before starting. Each row includes the minimum version, install command for both platforms, and the verify command.

⚠ Version matters. Services use language features and CLI flags tied to these minimum versions. Running an older version produces confusing errors, not a clear version mismatch message.
Tool Minimum version Install (macOS) Install (Linux) Verify
Docker 25.0+ brew install --cask docker apt install docker.io docker info
Node.js 22.x brew install node@22 nvm install 22 node --version
Python 3.12+ brew install python@3.12 apt install python3.12 python3 --version
Go 1.23+ brew install go apt install golang-1.23 go version
kubectl 1.28+ brew install kubectl snap install kubectl kubectl version --client
Helm 3.14+ brew install helm snap install helm helm version
Note: Node and Python are only required if you plan to run those services natively. Docker is sufficient for all services via Docker Compose.

2. Clone and navigate

One-time setup. Run these two commands before anything else.

  • 1 Clone the repository. git clone https://github.com/yarova-ca/pipeline-studio.git Downloads the full monorepo including all 106 service directories.
  • 2 Enter the repo root. cd pipeline-studio All service paths are relative to this root. Run every subsequent command from here unless a section specifies otherwise.
Repo structure: Each service lives at services/<N>-<name>/. Every service directory contains a docker-compose.yml, a .env.example, and source code. The number prefix matches the service index in the service status page.

3. Run any service with Docker Compose

Fastest path. No native language tooling required. Every service has a docker-compose.yml at services/<N>-<name>/docker-compose.yml.

Example — 14-express

  • 1 Enter the service directory. cd services/14-express
  • 2 Start the service. docker compose up Builds the image if not cached, starts the container, and streams logs to the terminal.
  • 3 Verify the service is running. curl http://localhost:3000/health/live Expected response: {"status":"ok"} with HTTP 200.

Health check response reference

Health endpoint HTTP status Meaning What to do
/health/live 200 Service process is running. Nothing — service is up.
/health/ready 200 Service is running and database is reachable. Nothing — fully ready.
/health/ready 503 Database is not reachable. Start PostgreSQL — see section 7.
/health/live 404 Wrong URL or port. Check the PORT env var in docker-compose.yml.
Connection refused — Container is not running. Run docker compose up or check docker compose ps.
⚠ Port conflicts: Each service uses a default port defined in its docker-compose.yml. When running multiple services, each must use a unique port. Check docker compose ps to see which ports are bound.

4. Run with build axes

Build axes are compile-time switches that activate compliance and auth variants. Pass them as environment variables or Docker build args.

Via Docker Compose — environment override

COMPLIANCE=hipaa docker compose up

Pass any axis as a prefix environment variable. Docker Compose reads it and injects it as a build arg.

Via docker build + run

docker build \
  --build-arg COMPLIANCE=hipaa \
  --build-arg AUTH=jwt \
  -t myapp .

docker run -p 3000:3000 myapp

Available build axes

Axis Values Default What it activates
COMPLIANCE none, hipaa, pci, fips none Audit logging, encryption, FIPS runtime swap.
AUTH none, jwt, oidc none Authentication middleware and token validation.
RUNTIME standard, fips standard FIPS-validated OpenSSL module in place of default.
Note: Full axis documentation including combination rules and what each value changes at the code level: 21 Build axes.

5. Run natively (without Docker)

Use when you need hot-reload or are actively editing source. Each framework has its own setup sequence.

14-express — Node / TypeScript

  • 1 Enter the service directory. cd services/14-express
  • 2 Install dependencies. npm install
  • 3 Copy the env file. cp .env.example .env The service will not start without a .env file. Edit the values before running if you need a real database.
  • 4 Start the dev server. npm run dev Starts tsx in watch mode. Source changes restart the server automatically.
  • 5 Verify. curl http://localhost:3000/health/live Expected: {"status":"ok"} with HTTP 200.

15-fastapi — Python

  • 1 Enter the service directory. cd services/15-fastapi
  • 2 Create and activate a virtual environment. python3 -m venv .venv && source .venv/bin/activate Isolates dependencies from the system Python. Always activate before installing or running.
  • 3 Install dependencies. pip install -r requirements.txt
  • 4 Copy the env file. cp .env.example .env
  • 5 Start the dev server. uvicorn src.main:app --reload --port 3001 --reload restarts on source changes. Port 3001 avoids conflict with 14-express on 3000.
  • 6 Verify. curl http://localhost:3001/health/live Expected: {"status":"ok"} with HTTP 200.

16-gin — Go

  • 1 Enter the service directory. cd services/16-gin
  • 2 Download Go module dependencies. go mod download
  • 3 Copy the env file. cp .env.example .env
  • 4 Run the service. go run main.go Compiles and runs in one step. No hot-reload — restart manually after source changes, or use air for watch mode.
  • 5 Verify. curl http://localhost:3002/health/live Expected: {"status":"ok"} with HTTP 200.
⚠ Go hot-reload: go run does not watch for file changes. Install air (go install github.com/cosmtrek/air@latest) and run air instead of go run main.go for automatic restarts.

6. Required environment variables

Every service reads from .env in its own directory. Copy from .env.example first. The three variables below are required in every service.

⚠ Copy first, then edit. Running without .env causes an immediate crash with "required env var" errors. Always cp .env.example .env before the first run.
Variable Required Example value When missing: what happens When invalid: what happens
JWT_SECRET required dev-secret-change-in-prod-32ch Service exits: JWT_SECRET required When < 32 chars: service exits: JWT_SECRET must be at least 32 chars
DATABASE_URL required postgresql://postgres:localdev@localhost:5432/pipeline_dev Service starts but /health/ready returns 503 Service starts but all DB queries fail with connection error
PORT optional 3000 Defaults to 3000 when not set Non-numeric value causes crash: invalid port

Service-specific variables

Each .env.example lists every variable that service reads. Variables not in the table above are service-specific. The example file includes a comment for each one explaining what it controls.

Note: JWT_SECRET must be identical across services that share a JWT-authenticated boundary. A mismatch causes 401 errors on every cross-service request.

7. Local database setup

Three services require PostgreSQL: 14-express (Prisma ORM), 15-fastapi (SQLAlchemy), 16-gin (GORM). All three share the same local PostgreSQL instance.

Start PostgreSQL via Docker

docker run -d \
  -p 5432:5432 \
  -e POSTGRES_PASSWORD=localdev \
  --name pg \
  postgres:16-alpine

This starts a named container called pg on port 5432. The container persists across reboots. Use docker start pg to restart it without creating a new one.

Set DATABASE_URL in .env

DATABASE_URL=postgresql://postgres:localdev@localhost:5432/pipeline_dev

Copy this exact value into the .env of each service that needs a database. The database name pipeline_dev is created automatically by each service's ORM on first run.

Verify database connection

Command Expected output What it means
curl http://localhost:3000/health/ready {"status":"ready"} HTTP 200 PostgreSQL is running and reachable.
curl http://localhost:3000/health/ready HTTP 503 PostgreSQL is not running. Run docker start pg.
If PostgreSQL is not running
Trigger: /health/ready returns 503. Service logs show Connection refused :5432.
System: Service starts and handles HTTP. All routes that touch the database return 500.
User sees: 503 on /health/ready. 500 on any data endpoint.
To restore: Run docker start pg. If the container does not exist, run the full docker run command above. Verify with /health/ready.
One database, three services: All three services can connect to the same pipeline_dev database simultaneously. Each ORM manages its own schema in separate tables. There are no naming conflicts.

8. Failure modes

Every common failure you will encounter, what the error looks like, and exactly what to do. All commands are copy-pasteable.

What breaks What you see What to do
Port in use Error: listen EADDRINUSE :3000 Find and kill the process:
lsof -i :3000
Then: kill <PID>

Linux: sudo systemctl start docker
npm install fails ERESOLVE peer dep errors Delete lockfile and modules, then reinstall:
rm -rf node_modules package-lock.json
Then: npm install
Database not running Connection refused :5432 in logs Restart existing container: docker start pg
Container missing: run the full docker run command from section 7.
.env missing Service crashes: "required env var" cp .env.example .env
Edit the values if you need real database credentials.
FIPS build fails FIPS module not active Add both axes together:
--build-arg COMPLIANCE=fips --build-arg RUNTIME=fips
FIPS compliance requires the FIPS runtime — neither alone is sufficient.
Go build fails cannot find package internal/compliance/active The COPY for the compliance variant in the Dockerfile must run before go build.
Check the Dockerfile layer order: compliance COPY must precede the RUN go build step.

Detailed failure block — FIPS

If FIPS build fails
Trigger: docker build --build-arg COMPLIANCE=fips without --build-arg RUNTIME=fips.
System: Build fails at the runtime validation step with FIPS module not active.
User sees: Docker build exits non-zero. No image is produced.
To restore: Add both build args: --build-arg COMPLIANCE=fips --build-arg RUNTIME=fips. Both are required together. See 21 Build axes for the full FIPS setup.

Detailed failure block — Go compliance variant

If Go build fails with missing compliance package
Trigger: go build runs before the compliance variant files are copied into the build layer.
System: Build fails: cannot find package internal/compliance/active.
User sees: Docker build exits non-zero. No image is produced.
To restore: Open the service Dockerfile. Ensure the COPY for the compliance variant appears before RUN go build. The correct order: COPY internal/compliance/${COMPLIANCE}/. internal/compliance/active/ then RUN go build ./...
