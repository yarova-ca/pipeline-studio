When to update vs when to skip

Binary rules — apply them in order. First match wins.
Base image CVE patches:
When a base image receives a CVE patch rated HIGH severity: update within 7 days.
When a base image receives a CVE patch rated MEDIUM severity: update within 30 days.
When a base image has no new CVE patches: skip. Review at 30-day cadence regardless.
Framework major version releases:
When a framework releases a major version containing a security fix: update immediately.
When a framework releases a major version with no security fix: update within 90 days.
When a framework releases a minor or patch version: update at next scheduled 30-day review.
If version registry is not updated within cadence:
Trigger: Base image has 30+ days since last review in 15-version-registry.html.
System: Trivy scan finds CVE in base image. PR gate blocks all open pull requests using that image.
User sees: CI failure — "HIGH CVE in node:22-alpine (CVE-2024-XXXX) — update base image before merge."
User can: Run python3 scripts/generate_full_bundles.py --only env to regenerate .env.example. Update the version pin in 15-version-registry.html. Re-push to unblock the PR gate.

Adding a new service — step by step

Follow in order. Each step has a defined output. Do not skip steps.
  1. 1
    Add entry to SERVICES_META in scripts/generate_full_bundles.py. Input: service name, port, tier, stack. Output: script knows the service exists.
  2. 2
    Create services/XX-name/ with Dockerfile, app code, and tests. Input: service directory. Output: buildable image. CI runs Docker build on push.
  3. 3
    Run: python3 scripts/generate_full_bundles.py --service XX-name Output: .env.example, docker-compose snippet, and bundle manifest for this service.
  4. 4
    Add workflow template to workflow-templates/XX-name/ if this service introduces a new pipeline pattern. Skip this step if the service reuses an existing pattern exactly.
  5. 5
    Add row to 01-framework-catalog.html framework table. Output: service is discoverable from the hub. Missing row = service is invisible to users of the catalog.
  6. 6
    Add row to 17-feature-matrix.html with tier and status. Output: feature matrix is accurate. Status = Draft / Beta / Stable. Tier = 1 / 2 / 3.
  7. 7
    Commit and push. Docker build CI validates the new service immediately on push. CI result = binary pass/fail. Fail = service directory or Dockerfile has an error. Fix before merging.

Adding a new doc (doc N)

Use this checklist every time a new HTML doc is added, whether doc 17, 20, or any number.

Do this in order. Steps 1–3 must complete before step 4. Steps 4–6 can run in any order after step 3.

Step 1 — Create the HTML file

  1. 1 Create the file at: /mnt/c/Users/RohithY/yarova/pipeline-studio/NN-doc-name.html Why: all docs live in the root of pipeline-studio — no subdirectories. touch NN-doc-name.html
  2. 2 Use an existing doc as a CSS/nav template base. Copy from: 08-canada-market.html (cleanest nav pattern). Why: all docs share the same CSS variables, font stack, sidebar width (220px), and nav structure. Diverging breaks dark mode and print styles.
  3. 3 Add all 16 existing doc links to the new doc's nav sidebar. Why: every doc must link to every other doc — this is the only navigation mechanism. grep -A2 'nav-top' 08-canada-market.html # copy the nav block pattern
  4. 4 Mark the new doc's own nav link with aria-current="page". Why: this is how the sidebar highlights the active doc — missing it leaves no visual indicator of which doc you're reading.

Step 2 — Add the new doc's nav link to all existing docs

  1. 5 In every existing HTML file, add the new link before the ↑ Back to top line. Why: the nav link must appear in all existing docs — readers of any doc must be able to reach the new one. python3 - <<'EOF' NEW = '<a href="NN-doc-name.html">NN Doc label</a>' BACK = '<a href="#main-content" class="nav-top">↑ Back to top</a>' import glob for f in glob.glob('*.html'): with open(f) as fh: html = fh.read() if NEW in html or f == 'NN-doc-name.html': continue if BACK in html: html = html.replace(BACK, NEW + '\n' + BACK) with open(f, 'w') as fh: fh.write(html) print('updated:', f) EOF
  2. 6 Verify: every HTML file now contains the new link. grep -rL 'NN-doc-name.html' *.html Why: any file missing the link breaks navigation from that doc to the new one. Output must be empty (no files missing the link).

Step 3 — Update index.html

  1. 7 Update the page-meta count chip: N design docs → N+1 design docs. Why: the page meta is the first thing readers see — a stale count implies the docs are not maintained.
  2. 8 Add a doc card to the "All N+1 docs" grid. Copy the card template below. Why: the index is the entry point — a missing card means the doc is invisible to new readers.
  3. 9 Update the "All N docs" h2 heading to All N+1 docs.
  4. 10 Add an entry to the Recommended reading order section. Why: reading order tells readers when to use the new doc relative to existing ones — without it, readers encounter it out of context.
  5. 11 Add an entry to the Cross-doc dependencies section. Why: the dependency map shows which docs the new one depends on — missing it means the relationship is undocumented.

Step 4 — Update doc 15 (Version Registry)

  1. 12 Add any new versioned items from the new doc to 15-version-registry.html. Why: doc 15 is the single maintenance index — a versioned item not listed there will never be updated. To add: open scripts/generate_15.py → add entries to the ITEMS list → re-run the script. python3 scripts/generate_15.py
  2. 13 Update doc 15's nav to include the new doc link (it will be added automatically in step 5 above if you ran the nav script). Verify: grep 'NN-doc-name.html' 15-version-registry.html

Step 5 — Update this runbook (doc 16)

  1. 14 Update the File map section below to include the new doc. Why: the file map is the authoritative list of all docs — a missing entry creates confusion about what exists. To update: open scripts/generate_16.py → add row to DOCS list and file map table → re-run. python3 scripts/generate_16.py
  2. 15 Update the counts bar at the top of this page: 16 Docs → 17 Docs.

Updating a version

Use when any tool, framework, base image, or distro version changes. Check doc 15 for review cadence.

  1. 1 Open doc 15 — Version Registry. Find the item. Click its source link. Why: doc 15 lists every versioned item with its source URL — no need to search individual docs.
  2. 2 Note the new version. Note which docs it appears in (the "Used in" badges column).
  3. 3 Update the version string in every listed doc. Why: the same item appears in multiple docs — updating only one creates inconsistency that the next audit will flag. grep -n "OldVersion" doc-NN.html # find exact location first
  4. 4 For base images: also update the FROM tag in the actual Dockerfile shown in doc 11. Why: doc 11 contains literal Dockerfile content that readers copy — a stale FROM tag ships a vulnerable image.
  5. 5 Update the "Last verified" date in doc 15 for the changed item. To do: open scripts/generate_15.py → update the VERIFIED constant at the top to today's date → re-run. python3 scripts/generate_15.py
  6. 6 Commit with exact format: git commit -m "Update [item name] [old version] → [new version] (verified YYYY-MM-DD)"
Base image version change requires extra check: when node:22-alpine bumps to node:24-alpine, the Node.js major version inside the container changes. Verify the framework in doc 01 still supports that Node.js version before shipping. Check: doc 01 framework runtime column.

Updating market or industry data

Use when re-verifying Canadian market data (docs 07, 08, 14) or US industry data (doc 06). Annual cadence.

  1. 1 Identify what changed: demand signal, wage band, tier, employer name, compliance framework, or entry notes.
  2. 2 Update the relevant doc: Why: data is in three separate docs — 07 (schema/compliance), 08 (market signals), 14 (deep-dive). All three must stay consistent.
  3. 3 Cross-check: if demand changes in doc 08 for an industry, verify the same industry's entry notes in doc 14 still match. Why: a "Flat" demand industry in doc 08 with "Hot — roles open immediately" entry notes in doc 14 is a direct contradiction.
  4. 4 Update the data verified date in the page-meta line of each updated doc. grep -n "data verified" doc-NN.html # find the meta line
  5. 5 Update the market data row in doc 15 (last verified date).

Nav link pattern — exact format required

Every nav sidebar in every doc must follow this exact structure. Deviation breaks navigation.

Standard nav link (all docs except the active one)

<a href="NN-doc-name.html">NN Label text</a>

Active doc nav link (the doc you are currently in)

<a href="NN-doc-name.html" aria-current="page">NN Label text</a>

Back to top link — always last in nav

Two variants exist in the repo. Both are correct — do not change them:

<!-- Most docs use: -->
<a href="#main-content" class="nav-top">↑ Back to top</a>

<!-- Doc 14 uses HTML entity (both render identically): -->
<a href="#main-content" class="nav-top">&#8593; Back to top</a>
When adding a nav link: always insert it immediately before the Back to top line — never after. The Back to top link must always be last.

Platform Health — Weekly Checklist

Run every week. Each item has a pass/fail command. Fix any fail before the next review cycle.

GitHub Actions all green
gh run list --repo yarova-ca/pipeline-studio --limit 10 | grep -v success
Pass: no output (all runs successful)
Fail: fix the failing workflow before proceeding to any other checklist item
No HIGH/CRITICAL CVEs in base images
gh run list --repo yarova-ca/pipeline-studio --workflow security-scan.yml --limit 1 --json conclusion
Pass: conclusion = success
Fail: open the security-scan run → find failing Trivy job → update base image tag in doc 11 → update verified date in doc 15
Version registry reviewed in last 30 days
grep -c "2026-" 15-version-registry.html
Pass: all items show a date within 30 days of today
Fail: open doc 15 → find items with stale dates → re-verify each one → update date
All 17 HTML docs updated if code changed this week
git log --since="7 days ago" --oneline -- '*.html' | wc -l
Pass: HTML commit count matches code commit count for the week
Fail: identify which code change lacks a matching doc update → update the relevant HTML page before this week's review
Nav links consistent across all docs
for f in *.html; do count=$(grep -c 'nav-link' "$f" 2>/dev/null); echo "$f: $count"; done | sort -t: -k2 -n | head -3
Pass: all docs show the same nav-link count (lowest 3 match the others)
Fail: the lowest-count docs are missing nav links — run the nav propagation script in Step 2 of Adding a new doc
Generation scripts still produce clean output
python3 scripts/generate_15.py 2>&1 | tail -3
Pass: script exits 0, last line confirms file written
Fail: script has a Python error → fix the script before running it in production → do not commit broken script
No open PRs older than 7 days
gh pr list --repo yarova-ca/pipeline-studio --state open --json number,title,createdAt
Pass: all PRs created within the last 7 days
Fail: review stale PRs — either merge, close, or add a comment explaining the block

Commit style — required format

All commits to this repo must follow this format. No exceptions.

Rules

  1. R1 Never commit directly to main. Always: new branch → commit → PR → merge. Why: direct commits to main bypass review and break the audit trail.
  2. R2 One topic per PR. A version update PR contains only version changes. A new doc PR contains only the new doc and nav updates. Why: mixed PRs make rollback impossible without also reverting unrelated changes.
  3. R3 Commit message: plain English, one line, why not what. No feat:, fix:, chore: prefixes.
