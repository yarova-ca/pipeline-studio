Start here — concept in 90 seconds

If you have never shipped a signed container, read this first. If you have, skip to Phase 2.

What this is

A complete CI/CD pipeline that takes source code and produces a cryptographically signed container image in your registry.

Boundary: source → registry. Deployment (K8s manifests, Helm) is out of scope — your CD tool consumes the signed image.

DevOps vs DevSecOps

DevOps = build + deploy fast.
DevSecOps = build + deploy fast + prove the artifact is safe.

This pipeline adds the proof: SBOM, signature, provenance, scan results — all verifiable at deploy time.

Who is this for

Any team shipping a containerized web app.
Junior dev: copy the files, follow Phase 0.
Staff/Principal: scan invariants, validate compliance map.

Why exactly 4 phases?

Each phase has a single job. Skip one — you ship broken or unverifiable code. Runtime security (Kyverno admission, Falco, Trivy Operator) is out of scope here — that's platform/SRE territory, installed once per cluster.

PhaseSingle jobSkip it — what breaks
1 — DeveloperCatch obvious issues on the laptop before CI spends a minute.Developer time wasted; CI budget burned on lint errors.

Why exactly 20 invariants?

Each invariant is a rule that must hold for every release. Click any invariant in the sidebar — it scrolls to the stage that enforces it.

The 20 are not arbitrary: each maps to a real incident class that has shipped malware or destroyed production.

    Phase 4 promotions deploy successfully only if the target cluster already has these set up. If your platform team hasn't, deploys will reach the cluster but admission may not be enforced (signed-image checks become opt-in instead of mandatory). Ask your SRE/platform team — these are one-time, per-cluster installs:

    • Admission controller (Kyverno or OPA Gatekeeper) with a verifyImages policy that matches your signing identity
    • Allowed-registries policy restricting workloads to your registry hostname(s)
    • ArgoCD or Flux controller running, with permission to read your config / app repo
    • Runtime detection (Falco, Trivy Operator) — optional but recommended; cluster-wide, owned by SRE/SOC

    These are platform concerns, not pipeline concerns. The studio emits the per-app config that uses these — it doesn't try to install them. Why: cluster bootstrap is a one-time setup owned by a different team; pipeline runs every commit, owned by app teams.

    📋 Sample admission policy — copy this and send to your platform team

    If your cluster doesn't already enforce signed images, paste this Kyverno ClusterPolicy. It refuses any pod whose image isn't signed by your GitHub Actions workflow. Replace YOUR_ORG with your GitHub org name.

    apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-signed-images
spec:
  validationFailureAction: Enforce      # I-20: fail-closed
  background: false
  rules:
    - name: verify-image-signature
      match:
        any:
          - resources:
              kinds: [Pod]
              namespaceSelector:
                matchLabels:
                  env: production       # only enforced in prod namespaces
      verifyImages:
        - imageReferences:
            - "REGISTRY/*"              # restrict to YOUR registry (gates allowed-registries too)
          attestors:
            - entries:
                - keyless:
                    subject: "https://github.com/YOUR_ORG/*/.github/workflows/*"
                    issuer: "https://token.actions.githubusercontent.com"

    Install Kyverno first: helm install kyverno kyverno/kyverno -n kyverno --create-namespace. Then apply the policy: kubectl apply -f admission.yaml. Label your prod namespaces: kubectl label ns myapp-prod env=production.

    Phase 0 — Repository Bootstrap ONE-TIME SETUP

    Complete before running any pipeline stage. No pipeline stage can succeed until these are done.

    Phase 1 — Developer Local — shift-left security before CI runs

    Shift-left security. Catches issues before they hit CI. Zero pipeline minutes spent.

    Phase 2 — PR Gate — blocks insecure or broken code from merging

    Blocks broken, insecure, or malformed code from merging. Every PR must pass all stages.

    .github/workflows/pr-checks.yml

    Phase 3 — Main Build — builds, signs, and attests the production image

    Runs on every merge to main. Builds the signed, attested production image.

    .github/workflows/main-pipeline.yml

    Phase 4 — Environment Promotions — moves the signed image dev → staging → prod

    Moves the signed image through environments. CD tooling (ArgoCD/Flux) handles the actual deploy. This pipeline verifies the image before each promotion.

    Image Verification at Each Environment Gate

    Before any environment receives the image, the CD pipeline verifies the cosign signature. Kyverno policy enforces this at the cluster admission layer.

    # Verify before every environment deploy (ArgoCD PreSync hook)
cosign verify \
  --certificate-identity-regexp "https://github.com/YOUR_ORG/YOUR_REPO/.github/workflows/" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  REGISTRY/IMAGE@sha256:DIGEST
