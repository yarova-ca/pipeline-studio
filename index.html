Recommended reading order

02 — Pipeline Schema
Start here. Understand the shape — 6 phases, 44 stages, 4 environments, what runs in what order.
03 — Stage Types
Classify each stage. Know who owns a failure — engineering or security.
04 — Tool Catalog
Pick tools for each stage. Understand mandatory vs conditional vs optional.
05 — Invariants
Confirm no ordering rule is violated. Check before changing stage order or adding stages.
01 — Framework Catalog (orthogonal)
Select a framework. Orthogonal to docs 02–05 — any framework runs the same pipeline.
06 — US Industry Schema or 07 — Canada Industry Schema
Add industry-specific controls. Select your industry → see which stages get extra controls.
08 — Canada Market Navigation (Canada only)
Navigate Canadian market entry — hubs, demand, wages, immigration pathway.
09 — Linux Distro Catalog (orthogonal)
Select a base image or CI runner OS. Understand libc, FIPS status, EOL, and tradeoffs before writing any FROM line.
10 — Linux Compliance Matrices
Cross-reference the chosen distro against compliance frameworks, industry verticals, and framework categories. Read after 09.
11 — Dockerfile Catalog
Copy the Dockerfile for your chosen framework. Select --target runtime or --target runtime-fips. Read after 09 and 10.
12 — Compliance Variations
Apply compliance layers on top of doc 11 Dockerfiles. 11 standards, 10 industry profiles, 5 concrete Dockerfile additions. Read after 11.
13 — Pipeline Build Catalog
Full pipeline for all 105 frameworks — every stage, every tool, every command. Compliance deltas and registry config included. Ticket-ready for Notion/Jira. Read after 02 and 11.
14 — Canada Industry Catalog
Deep-dive for all 73 Canadian industries — systems, project types, employers by province, tech stack, compliance, and entry notes. Read after 08 to combine market-level data with industry-level intelligence.
15 — Version Registry
Maintenance reference only — not part of the reading flow. Open it when verifying or updating any version across the docs. Review base images every 30 days, everything else every 90 days.
16 — Maintainability Runbook
Maintenance reference only. Read before adding a new doc, running a version update batch, or re-verifying data. Contains the exact checklist for every maintenance task.

Glossary — terms used across all docs

TermPlain-English definitionUsed in
SCASoftware Composition Analysis — scans third-party dependencies for known CVEs (Common Vulnerabilities and Exposures).02, 03, 04, 05
SASTStatic Application Security Testing — scans source code for insecure patterns without running the code.02, 03, 04, 05
DASTDynamic Application Security Testing — runs the application and probes it for vulnerabilities over HTTP.02, 03, 04, 05
IaCInfrastructure as Code — Terraform, Helm, Kubernetes manifests, CloudFormation, etc. Checked by checkov and hadolint.02, 03, 04, 05
SBOMSoftware Bill of Materials — machine-readable list of every component in the build artifact. Required for supply chain compliance.02, 03, 04, 05, 10
SLSASupply-chain Levels for Software Artifacts — provenance attestation proving the artifact was built by a trusted pipeline from a known source.02, 03, 04, 05, 10
OIDCOpenID Connect — used for keyless CI authentication. Each job gets a short-lived token; no stored credentials in CI.02, 03, 04, 05
PIPEDAPersonal Information Protection and Electronic Documents Act — Canada's federal private-sector privacy law.07, 08
OSFIOffice of the Superintendent of Financial Institutions — Canadian federal financial regulator.07, 08
FINTRACFinancial Transactions and Reports Analysis Centre of Canada — federal AML (anti-money laundering) regulator.07
GC PBMMGovernment of Canada Protected B, Medium Integrity, Medium Availability — cloud architecture compliance framework for federal workloads.07
OCAP®Ownership, Control, Access and Possession — First Nations data sovereignty principles. Registered trademark of the First Nations Information Governance Centre.07
NOCNational Occupational Classification — Canada's system for classifying occupations. Used in immigration applications (Express Entry, PNP).08
TEERTraining, Education, Experience and Responsibilities — NOC 2021 tier system. TEER 1 = university degree required for immigration classification.08
CRSComprehensive Ranking System — points-based score used in Express Entry to rank candidates. Higher score = invited to apply sooner.08
PNPProvincial Nominee Program — each province nominates candidates for permanent residency. Adds ~600 CRS points when issued.08
DevOpsDevelopment Operations — workflow and infrastructure stages (build, deploy, tag, test, notify). Failure = engineering team owns the fix.02, 03, 04
DevSecOpsDevelopment Security Operations — security enforcement stages (scan, sign, attest, verify, policy). Failure = artifact does not ship.02, 03, 04, 05
libcC standard library — the lowest-level runtime every Linux process links against. Determines binary compatibility across distros.09, 10
glibcGNU C Library — the standard libc used by Debian, Ubuntu, RHEL, Fedora, and most general-purpose distros. Broad binary compatibility.09, 10
muslAlternative C library used by Alpine Linux. Smaller and stricter than glibc. ~15% of precompiled glibc binaries fail on musl without recompilation.09, 10
OCIOpen Container Initiative — industry standard for container image formats and runtimes. An "OCI image" is a standard Docker-compatible container image.09, 10
UBIUniversal Base Image — Red Hat's free-to-redistribute RHEL subset. Includes FIPS crypto modules. Three tiers: UBI-minimal, UBI-micro, and standard UBI.09, 10
FIPS 140-2 / 140-3US federal cryptographic module validation standards (NIST). Required for US government contracts. FIPS 140-2 closing new submissions September 2026; 140-3 is the current standard.09, 10
DISA STIGDefense Information Systems Agency Security Technical Implementation Guide — hardening checklist for DoD and FedRAMP High workloads. Per-distro STIGs published at public.cyber.mil.09, 10
CIS BenchmarkCenter for Internet Security hardening guide — scored configuration checks for OS, container, and cloud targets. Widely used in SOC 2 and ISO 27001 audits.09, 10
