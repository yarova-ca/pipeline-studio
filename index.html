What this platform is

Pipeline Studio is a complete platform engineering catalog. It covers every framework from hello world through production-grade deployment — with Dockerfiles for all runtime axes, GitHub Actions for all 6 CI/CD phases, Helm charts, Kustomize overlays, compliance configs, and observability wiring.

The HTML pages are the knowledge layer. They document everything — all 106 frameworks, 22 Linux distros, 15 compliance standards, 74 US industries, 73 Canadian industries — with full WHY explanations and implementation details.

The services/ directory is the implementation layer. Each folder is a complete, runnable service with health endpoints, tests, and a Dockerfile that works today.

End goal: Every service bundle is a Backstage scaffolder template. A team picks framework + industry + compliance → gets a complete GitHub repo ready to push.

Pillar 4 Pipeline & Tools The 6-phase CI/CD pipeline — what runs, when, why, and which tool to use for each stage.
Doc 02
Pipeline Schema
The full pipeline shape. 6 phases (Bootstrap → Local Dev → PR Gate → Main Build → Registry → Promotions), 44 stages, 4 environments. Phase trigger conditions, stage outputs, parallel stage groups. Why 6 phases and what each phase boundary protects.
When to read: First doc to read. Understanding what runs when and in what order. Adding a new stage.
Doc 03
Stage Types
44 stages classified as DevOps (16) or DevSecOps (28). DevOps failure = workflow problem, engineering team owns it, pipeline may retry. DevSecOps failure = security event, artifact does not ship, security team is notified.
When to read: Deciding who owns a failed stage. Understanding the difference between a workflow error and a security gate.
Doc 04
Tool Catalog
The selected tool for every pipeline stage. Per tool: version, license, what it detects, false positive rate, why it was chosen over the nearest alternative, exact CLI invocation, CI integration, mandatory vs conditional vs optional, and primary tradeoff.
When to read: Choosing a scanner, signer, or runner. Swapping one tool for another.
Doc 13
Pipeline Build Catalog
Full CI/CD pipeline for all 106 frameworks. Every stage across all 6 phases with language-specific commands. Compliance deltas for all 11 standards. Push format and OIDC auth for all 6 registries. Links to service code in services/. Ticket-ready format for Backstage/Jira.
When to read: Implementing the full CI/CD pipeline for any framework. Read after Docs 02, 09, 11.

Use-case reading paths

Every path below is self-contained. You do not need to read all 17 docs to act on your use case.

New service — pick framework, write Dockerfile, set up pipeline
01 Frameworks→ 09 Distros→ 11 Dockerfiles→ 13 Pipeline→ services/XX/
01 narrows framework choices by language. 09 picks the base image. 11 gives the copy-paste Dockerfile. 13 gives every stage command. Then clone the matching service starter from services/ and build on top.
Canadian regulated industry — fintech, healthcare, or government
08 CA Market→ 14 CA Industries→ 07 CA Schema→ 10 Compliance→ 12 Compliance layers→ 13 Pipeline
Start with 08 to pick hub and province. Then 14 to identify which systems get built and which regulations apply. 12 adds the Dockerfile compliance deltas (PIPEDA, OSFI B-10, FINTRAC, PHIPA) on top of the base Dockerfile from 11.
FIPS 140-2 or FedRAMP compliance — any framework
10 Distro compliance→ 11 runtime-fips target→ 12 FIPS delta→ 05 Invariants
10 shows which distros have FIPS 140-2 validated modules. 11 shows the runtime-fips named target — uses UBI9 (Red Hat Universal Base Image v9) instead of alpine. 12 shows the full Dockerfile delta. Every Dockerfile in services/ already has RUNTIME=fips wired.
Protocol service — gRPC, GraphQL, or WebSocket
01 #c28–30→ 15 #protocols→ 11 #cat-grpc→ 13 #grpc→ services/28–30/
All 3 protocol groups (28 gRPC · 29 GraphQL · 30 WebSocket) have anchor-linked sections in 01, 11, 13, and 15. Use the anchor links to jump directly. Each language variant has its own service starter in services/.
Federal government Canada — ITSG-33, Protected B, GCKey
14 #gov→ 07 CA Industries→ 10 Compliance→ 12 ITSG-33 delta→ 05 Invariants→ 13 Pipeline
14 #gov covers GCKey, Protected B, WCAG 2.1 AA, Official Languages Act. Reliability Status clearance required for all federal roles. Secret clearance required for ~40% of federal IT positions.
Understanding the pipeline before building anything
02 Schema→ 03 Stage types→ 04 Tools→ 05 Invariants
This is the foundational reading path. 02 gives the shape. 03 tells you who owns each failure. 04 tells you which tool to use at each stage. 05 tells you the rules you cannot violate. Every other doc assumes you have read these four.
