Frontend Web — Cols 9 + 10 read together
Perf (col 9): TTFB or TTI in ms — how fast the server responds and the browser becomes ready.
Bundle size (col 10): gzipped JS delivered on first load — controls how long the browser takes to become interactive after the first byte arrives.
Fast TTFB + large bundle = page appears, then freezes 3–5s while JS loads. Fast TTFB + small bundle = fast all the way through. Every 100 kb extra ≈ +1s TTI on 4G mid-range.
Column definitions — Rendering modes, Hydration, Concurrency, GC

Rendering modes (col 20)

SSR = server renders full HTML per request. TTFB: 50–200ms. TTI: fast after TTFB. Good for SEO.

CSR = server sends empty HTML + JS bundle. Browser renders everything. TTFB: fast. TTI: slow until JS loads and executes.

SSG = all pages built at compile time. Served as static files from CDN. TTFB: <10ms. Cannot personalize per user.

ISR = SSG pages regenerated in background at set intervals. Stale content possible within the revalidation window.

Edge = SSR executed at CDN edge nodes worldwide. TTFB: 20–60ms global instead of 80–200ms origin. Constraint: no Node.js APIs — V8 Isolate only.

Streaming SSR = HTML sent in chunks as generated. Browser renders above-fold before full page is ready.

Hydration types (col 21)

Full = browser re-executes all component JS to attach event listeners. Cost: proportional to app size. Large apps: 3–8s hydration on mobile 4G.

Partial (Islands — Astro, Fresh) = only interactive "island" components hydrate. Static HTML areas stay inert. Cost: proportional to number of interactive components only.

Resumable (Qwik) = server serializes component state into HTML. Browser resumes from that state — does not re-execute component logic. Cost: ~0ms regardless of app size.

Concurrency (col 7)

Event loop = one thread handles all requests. Non-blocking I/O. CPU-bound work blocks all requests.

Use when: I/O-heavy workloads (DB calls, API calls). Do NOT use for: CPU-heavy processing on the same process.

Memory / GC (col 8)

V8 GC = stop-the-world pauses 1–10ms per collection. Acceptable for most web apps. Problem: real-time systems where any pause causes jank.

Mobile — Cols 9 + 10 read together
Perf (col 9): sustained frame rate in fps — below 60fps users see dropped frames (jank). 120fps = ProMotion smooth on supported hardware.
Bundle size (col 10): compressed app binary — controls download time from app store and on-device storage used.
High fps + large bundle = smooth UX, slow install. Low fps + small bundle = fast install, poor UX. PWA (cat 13) uses TTFB from Service Worker cache, not fps.
Column definitions — Concurrency, GC, Rendering, State init

Concurrency (col 7)

JS thread + native UI thread (React Native): JS runs on Hermes engine, UI renders on native thread. JSI (new arch) enables near-synchronous calls between them.

Dart isolates (Flutter): main isolate handles UI. Background isolates for CPU tasks. No shared memory between isolates.

GCD (Swift): Grand Central Dispatch — work queues distributed across device thread pool. Async/await (Swift 5.5+) structured on top.

Coroutines (Kotlin): lightweight concurrent tasks. Main dispatcher for UI, IO/Default dispatchers for async/CPU work.

WebView event loop (Ionic): same as browser — single JS thread. Separate native UI render thread. JS-to-native bridge for Capacitor plugins.

Memory / GC (col 8)

Hermes GC (React Native): Meta's JS engine optimized for mobile. Pre-compiled bytecode — faster startup than V8.

Dart VM GC (Flutter): generational GC per isolate. Each isolate has own heap — no global pause across isolates.

ARC (Swift / iOS): Automatic Reference Counting — deterministic. Memory freed immediately when last reference drops. No pause.

JVM ART (Android): Ahead-of-time compiled GC. Android Runtime — faster than Dalvik. Generational GC with concurrent collection.

CLR Mobile GC (.NET MAUI): Xamarin-derived mobile GC. Conservative collection on mobile constraint.

Rendering (col 20)

Native UI components (React Native, .NET MAUI): uses platform's actual button, text, list controls. Looks and behaves exactly like the platform expects.

Custom GPU renderer (Flutter Impeller): Flutter draws every pixel directly on GPU canvas. Pixel-perfect consistency across platforms — looks the same on iOS and Android.

WebView (Ionic): renders HTML/CSS inside a native shell. Looks web-like, not native-like. Full web attack surface.

Native canvas (Kotlin Compose): Skia canvas via HWUI — native Android rendering with custom draw calls.

State init (col 21) — cats 9–12

How app state is initialized at launch before the first frame renders. Unlike web hydration, there is no HTML to reattach JS to — state is initialized from scratch on every app launch.

Backend — Cols 9 + 10 read together
Perf (col 9): p99 latency in ms — the worst 1-in-100 user's wait time at sustained load. This is what users feel under real load, not the average.
Throughput (col 10): peak req/s under sustained load. Source: TechEmpower R22 JSON serialization test.
High throughput + high latency = handles volume but users wait. Low throughput + low latency = fast but cannot scale to traffic. Neither number alone tells the full story.
Column definitions — Concurrency models, GC models

Concurrency models (col 7)

Event loop (Node.js, Deno, Bun): one thread handles all requests. Non-blocking I/O. CPU-bound work blocks all requests. Use when: I/O-heavy workloads. Do NOT use for: CPU-heavy processing on the same process.

Goroutines (Go): lightweight threads — millions fit in memory. Runtime scheduler uses all CPU cores. Use when: mixed I/O + CPU workloads, high concurrency.

Thread-per-request (Java Spring MVC, Ruby, PHP FPM): one OS thread per request. Blocked on I/O = thread idle but still allocated. Use when: team familiar with thread-based model, existing codebase.

Actor model (Elixir / Phoenix BEAM): one isolated process per connection. No shared memory. Process crash does not affect others. Use when: massive concurrent connections (chat, IoT, real-time).

Async/await thread pool (ASP.NET Kestrel, Swift NIO, Tokio): thread pool — async I/O does not block threads. Multi-threaded. Use when: .NET, Swift, or Rust team; high throughput; CPU + I/O mixed.

Memory / GC models (col 8)

V8 GC (Node.js, Deno, Bun): stop-the-world pauses 1–10ms per collection. Acceptable for most web apps. Problem: real-time systems where any pause causes jank.

Go tri-color concurrent GC: runs concurrently with app. Pause: <1ms. Predictable at scale. Problem: very high allocation rate increases GC pressure.

JVM G1GC (Spring, Quarkus JVM, Scala): tunable heap regions. Pause: 5–50ms (configurable). Standard enterprise choice.

JVM ZGC (Spring WebFlux, high-throughput JVM): sub-1ms pause. Higher CPU cost than G1GC. Use when: latency is critical, CPU budget is available.

ARC (Swift): Automatic Reference Counting — deterministic. No pause — memory freed immediately when last reference drops. Problem: retain cycles must be broken manually.

BEAM per-process GC (Elixir / Phoenix): each process has isolated heap. No global GC pause. Best GC model for long-running concurrency. Process termination = instant memory release.

Rust borrow checker (Axum, Actix-web): no GC at all. Compiler proves memory safety at compile time. Zero runtime overhead. Zero pause. Cannot compile memory-unsafe code.
