Compliance standard definitions

FIPS 140-2 / 140-3

Federal Information Processing Standard. US gov requirement for cryptographic modules. Required for FedRAMP Moderate+, DoD IL2+, ITAR. FIPS 140-2 still valid; new certs issued under 140-3.

FedRAMP

Federal Risk and Authorization Management Program. US gov cloud security framework. Moderate = standard agencies. High = sensitive data (DoD, Intelligence, Healthcare). Distro must supply a FIPS-validated crypto module at FedRAMP Moderate+.

HIPAA

Health Insurance Portability and Accountability Act. US healthcare data privacy. No specific OS requirement — distro must support encryption-at-rest and encryption-in-transit, and have audit logging capability.

PCI-DSS

Payment Card Industry Data Security Standard. Required for any system that stores, processes, or transmits cardholder data. Requires CIS or DISA STIG hardening, patch management, and audit logging.

SOC 2 Type II

Service Organization Control 2. Audit of security, availability, and confidentiality controls. No specific OS requirement — auditor evaluates controls on whatever OS is used. CIS Benchmark hardening is commonly cited evidence.

ISO 27001

International standard for information security management systems. Applies globally. No specific OS requirement — requires documented access controls, patch management, and incident response.

SBOM and SLSA — supply chain compliance

SBOM

Software Bill of Materials — a machine-readable list of all components in a software artifact. Required by US Executive Order 14028 (2021) for software sold to the US government. Wolfi/Chainguard produces an SBOM for every package natively. Other distros require third-party tools (Syft, Trivy) to generate an SBOM post-build.

SLSA

Supply-chain Levels for Software Artifacts — a framework for measuring build pipeline integrity. Level 3 requires a hermetic build, reproducible artifacts, and provenance attestation. Wolfi/Chainguard images are built at SLSA Level 3 natively. All other distros require additional build pipeline tooling to reach SLSA Level 3.

Build-time vs runtime — the critical distinction

Build-time (Matrix 3): the OS where your CI pipeline runs — it compiles code, installs dependencies, and runs tests. This is the CI runner.

Runtime (Matrix 4): the OS inside the final container that runs in production. This is the Dockerfile FROM image.

Why they are often different distros

Multi-stage Dockerfiles use a heavier build image for the compile step, then copy only the output artifact into a minimal runtime image. Example: build a Next.js app on Ubuntu 24.04 (needs Node.js, npm, TypeScript compiler), then copy the built output into an Alpine + Node.js runtime image — 80MB final vs 400MB if Ubuntu was the runtime.

Go and Rust — extreme case

A Go binary compiled with CGO_ENABLED=0 is statically linked — no libc dependency. The runtime image can be scratch (zero bytes, no OS at all). Build anywhere, run on nothing. Rust with the x86_64-unknown-linux-musl target achieves the same.

Implication for this doc

Use doc 09 twice: once to select your CI runner distro (Matrix 3 tells you if it can build your framework), once to select your runtime container base (Matrix 4 tells you if it can host your output). They can and often should be different distros.

Mobile build — Linux limitations (Matrix 3)

Linux distros can build some mobile targets — but not all.

Native iOS — Linux cannot build it

iOS apps require macOS and Xcode. This is enforced by Apple's toolchain (codesigning, XCBuild). No Linux distro can build or sign a native iOS app. All cells in the Native iOS column in Matrix 3 are N/A for this reason.

Native Android — Linux can build it

Android apps require a JDK and the Android SDK/Gradle. Both are available on Linux. In Matrix 3, the Android column has the same value as the JVM column — if the distro can run a JVM build, it can build Android.

React Native (Cross-platform JS)

The JavaScript bundle can be built on any Linux distro that supports Node.js. iOS signing and device builds still require macOS. Android APK/AAB production builds can be done on Linux CI.

Flutter (Cross-platform non-JS)

Flutter uses Dart. Dart SDK is available on glibc distros (official dart.dev packages for Debian, Ubuntu, RHEL). Alpine (musl) is not officially supported — Dart doesn't ship a musl build. Flutter Android builds work on glibc Linux CI. iOS still requires macOS.

Framework compat — musl vs glibc warning (Matrix 3 and 4)

The "Workaround" rating in Matrix 3 (build runner) means: the framework's build tools run on this distro, but native modules or compiled extensions must be recompiled for musl. The default pip wheel or npm package may fail at build time.

The "Workaround" rating in Matrix 4 (runtime) means: the runtime works, but native extensions in the deployed artifact must have been compiled against musl — glibc-compiled binaries crash at runtime.

Alpine Linux uses musl. All other distros use glibc. "Workaround" ratings apply only to Alpine in both matrices.

Fix: in the Dockerfile, use a multi-stage build — build stage on Debian (glibc), copy the compiled binary to Alpine for the final stage. Or: rebuild all native extensions inside Alpine using apk add gcc musl-dev.

5 rules that explain 90% of Matrix 3 (build-runner)

  • Alpine = Workaround for Node / Python / Ruby / PHP / JVM — musl libc breaks native C extensions. Exceptions: Go, Rust, and .NET support musl natively and show Native.
  • scratch / Distroless / UBI-micro / Google COS / Flatcar / Talos / Bottlerocket = — everywhere — these have no shell. A CI build pipeline cannot run on them.
  • Native iOS column = — on every distro — iOS builds require macOS + Xcode. No Linux distro can build or sign iOS apps.
  • Swift = Native only on Ubuntu 22.04 and 24.04. Other distros require manual download from swift.org and show Install or Workaround.
  • Elixir on RHEL family = Install (not Native). Elixir is not in standard RHEL / CentOS repos. Use EPEL or the Erlang Solutions package.

5 rules that explain 90% of Matrix 4 (runtime)

  • scratch = Go static binary and Rust static binary only. No libc, no shell, no packages. Everything else is Not supported.
  • Distroless: use the language-specific variant. gcr.io/distroless/nodejs22-debian12, /python3-debian12, /java21-debian12. The wrong variant = Not supported.
  • Alpine = Compat for everything except ML inference. NVIDIA CUDA container images require glibc — Alpine (musl) is not supported. ML inference on Alpine = Workaround requiring a custom CUDA build.
  • Ubuntu 22.04 and 24.04 = Optimal for ML inference. All official NVIDIA CUDA container images are Ubuntu-based. Pick Ubuntu for any GPU / ML workload.
  • Wolfi = Optimal for most runtime types. glibc (no musl breakage) + SBOM natively + Alpine-sized images + rolling CVE patches.
