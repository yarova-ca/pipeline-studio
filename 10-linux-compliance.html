Compliance standard definitions

FIPS 140-2 / 140-3

Federal Information Processing Standard. US gov requirement for cryptographic modules. Required for FedRAMP Moderate+, DoD IL2+, ITAR. FIPS 140-2 still valid; new certs issued under 140-3.

FedRAMP

Federal Risk and Authorization Management Program. US gov cloud security framework. Moderate = standard agencies. High = sensitive data (DoD, Intelligence, Healthcare). Distro must supply a FIPS-validated crypto module at FedRAMP Moderate+.

HIPAA

Health Insurance Portability and Accountability Act. US healthcare data privacy. No specific OS requirement — distro must support encryption-at-rest and encryption-in-transit, and have audit logging capability.

PCI-DSS

Payment Card Industry Data Security Standard. Required for any system that stores, processes, or transmits cardholder data. Requires CIS or DISA STIG hardening, patch management, and audit logging.

SOC 2 Type II

Service Organization Control 2. Audit of security, availability, and confidentiality controls. No specific OS requirement — auditor evaluates controls on whatever OS is used. CIS Benchmark hardening is commonly cited evidence.

ISO 27001

International standard for information security management systems. Applies globally. No specific OS requirement — requires documented access controls, patch management, and incident response.

SBOM and SLSA — supply chain compliance

SBOM

Software Bill of Materials — a machine-readable list of all components in a software artifact. Required by US Executive Order 14028 (2021) for software sold to the US government. Wolfi/Chainguard produces an SBOM for every package natively. Other distros require third-party tools (Syft, Trivy) to generate an SBOM post-build.

SLSA

Supply-chain Levels for Software Artifacts — a framework for measuring build pipeline integrity. Level 3 requires a hermetic build, reproducible artifacts, and provenance attestation. Wolfi/Chainguard images are built at SLSA Level 3 natively. All other distros require additional build pipeline tooling to reach SLSA Level 3.

Framework compat — musl vs glibc warning

The "Requires workaround" rating in Matrix 3 means: the framework works on this distro, but native modules or compiled extensions must be recompiled for musl. The default pip wheel or npm package may fail.

Alpine Linux uses musl. All other distros in this table use glibc. A "Requires workaround" rating applies only to Alpine.

Workaround: in the Dockerfile, install build tools (apk add gcc musl-dev), pip install from source instead of wheel, or use a multi-stage build where the build stage is Debian and the final stage is Alpine (copy the built binary, not the source).
