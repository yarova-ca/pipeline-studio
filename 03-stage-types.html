Stage Types
What this covers: All 44 pipeline stages classified as DevOps or DevSecOps — who owns a failure, what the system does, what the team must do next.
When to read: A stage failed and you need to know who owns it. Understanding the difference between a workflow error and a security gate.
Pillar 4 — Pipeline & Tools  ·  Related: 02 Pipeline schema  ·  04 Tool catalog
Why the DevOps vs DevSecOps classification matters
DevOps failure: Engineering owns it. Can be retried, skipped with justification, or bypassed by a senior engineer.
The artifact is not blocked from shipping — the pipeline can proceed after the issue is resolved or documented.
DevSecOps failure: Security owns it. Cannot be bypassed without security team sign-off.
The artifact does not ship. Period. The pipeline stops and stays stopped until the finding is resolved.
Decision: two-class model (DevOps / DevSecOps) — not a finer-grained taxonomy
Rejected: 4-class model (DevOps / Security / Compliance / Process) — teams argue about classification instead of fixing the failure. The question "who owns this" must have a binary answer at 2am.
Accepted tradeoff: some stages are borderline (e.g. license scanning — legal risk but not a CVE). These are classified DevSecOps conservatively — the cost of a false-block is lower than the cost of a false-pass.

What this doc covers
Every pipeline stage classified as DevOps or DevSecOps.
DevOps — infrastructure and workflow. Build, deploy, tag, test, notify.
DevSecOps — security enforcement. Scan, sign, attest, verify, policy.
Pipeline flow and ordering: doc 02. Tool selection per stage: doc 04. Ordering constraints: doc 05.
Acronym definitions — terms used in this doc

SCA — Software Composition Analysis. Scans third-party dependencies for known CVEs.

SAST — Static Application Security Testing. Scans source code for insecure patterns without running the code.

DAST — Dynamic Application Security Testing. Runs the application and probes it for vulnerabilities over HTTP.

IaC — Infrastructure as Code. Terraform, Helm, Kubernetes manifests, CloudFormation, etc.

SBOM — Software Bill of Materials. Machine-readable list of every component in the build artifact.

SLSA — Supply-chain Levels for Software Artifacts. Provenance attestation proving the artifact was built by a trusted pipeline.

OIDC — OpenID Connect. Used for keyless CI authentication — each job gets a short-lived token, no stored credentials.

Why the split matters — gate ownership and failure response

DevOps failure = workflow problem. Engineering team owns the fix. No security escalation required.

DevSecOps failure = security event. Engineering and security both respond. The artifact does not ship until the stage passes.

Phase 0 has equal DevOps (3) and DevSecOps (3). The security baseline must exist before any code is written — not added later.

Phase 3 has 11 DevSecOps vs 5 DevOps. Every artifact that enters the registry must be scanned, signed, and attested. The ratio reflects that requirement.

Promotions have 1 DevSecOps stage (cosign verify). Signing happened in Phase 3. Promotions re-verify — they do not re-sign.
